Redis八股快速入门

作者: 老江湖
日期: 五月 12, 2026  16:52:12
分类: 八股 Redis合集

Hi，我是 meisijiya ，也是 老江湖 ，感谢你的阅读，我们一起进步。

Redis是后端面试的一块基石，考查的深度与广度兼具，且常与数据库和分布式系统结合提问。

下面我将高频考点梳理成了一套“企业级面试体系”，并以与MySQL对比、串联的方式，帮你构建一个更立体的知识网络。

一、核心定位：高性能缓存与轻量级存储

在面试中，首先要能清晰地阐述Redis的核心定位。简单来说，Redis（Remote Dictionary Server）是一款开源的、基于内存的键值对NoSQL数据库，由C语言开发。它的核心特点是快、数据结构丰富、且支持持久化。

当面试官问“为什么用Redis而不用MySQL做缓存”，核心差距如下：

对比维度 Redis MySQL
数据类型 基于内存的键值存储，支持多种数据结构 (String, Hash, List, Set, ZSet等) 关系型数据库，数据以表格形式存储
性能核心 纯内存操作，读写速度在纳秒级 (100纳秒级) 基于磁盘，操作速度在毫秒级，依赖索引优化
持久化 支持RDB快照和AOF日志，但本质是内存数据库 强制持久化到磁盘，保证数据安全
数据模型 键值对，也支持发布/订阅、Stream等扩展 严格的结构化数据，支持SQL查询和事务(ACID)
核心场景 缓存、分布式锁、计数器、消息队列 核心业务数据的持久化存储

面试官追问：既然Redis快，为什么不用它完全替代MySQL？
可以从三个角度回应：

  • 数据模型：Redis是键值模型，无法替代SQL复杂查询。
  • 事务可靠性：Redis事务简单，MySQL有成熟的ACID保障。
  • 成本：内存成本远高于磁盘，不太适合存储所有业务数据。

一句话总结就是 “Redis是加速的” ，让高频读写的热点数据从慢速磁盘走向快速内存；“MySQL是放心的” ，为全量数据提供可靠的持久化记录。

二、缓存与DB联动核心：双写一致性与缓存问题

这部分是面试的重中之重，考察候选人对Redis和MySQL协同工作中问题的理解深度。

1. MySQL与Redis的数据一致性如何保证？

当系统需要同时更新数据库和缓存时，如何保证数据的一致性是一道经典难题。这里有几种常见的策略，可以根据对“一致性”的要求逐级递升：

  • 策略路径1：先删缓存，再更新数据库（延迟双删）：这是最常用的方案，通过延迟删缓存来避免并发读请求将旧数据回种到缓存中。此策略适用于能接受短期数据不一致的场景，是典型的最终一致性方案。
  • 策略路径2：先更新数据库，再删除缓存：能保证下次读请求能重新从数据库加载最新数据，可能出现瞬间不一致。
  • 策略路径3：订阅数据库Binlog，异步更新缓存：通过监听MySQL的Binlog（二进制日志），使用MQ异步更新Redis，实现最终一致性，解耦且可靠性高。
  • 策略路径4：分布式读写锁：在更新数据时加排他锁，保证读请求被阻塞，实现强一致性，但会牺牲并发性能。

你可以这样总结给面试官：“在项目中，我们优先考虑最终一致性。对于一般业务，使用‘延迟双删’或‘先更新DB再删缓存’的方案。对于对数据实时性要求极高的场景，我们会引入分布式锁实现强一致性读写。”

2. 缓存“三大坑”：穿透、击穿、雪崩

当Redis和MySQL结合时，如果缓存处理不当，会引发经典的三大问题，面试必考。

问题 定义 解决方案 (附代码/设计思路)
缓存穿透 查询一个数据库和缓存中都不存在的数据，导致请求直接打到了数据库 1. 布隆过滤器：在缓存层前加一个过滤器，先判断key是否存在，不存在则直接拦截。
2. 缓存空值：对于查不到数据的key，也缓存一个空值，并设置一个较短的过期时间。
缓存击穿 一个热点key在缓存中过期的瞬间，大量并发请求同时涌入数据库 1. 互斥锁：在缓存失效后，只有一个线程能获得锁去加载DB并重建缓存，其他线程等待。
2. 逻辑过期：不给热点key设置物理过期时间，在value里放一个逻辑过期时间，发现过期后异步去加载新数据。
缓存雪崩 大量缓存key在同一时间失效，或Redis服务宕机，导致海量请求直接打到数据库 1. 过期时间加随机值：给每个key的过期时间加上一个随机因子，避免集中失效。
2. 高可用架构：使用Redis Cluster或哨兵模式，避免单点故障。
3. 多级缓存：结合本地缓存（如Caffeine），在Redis不可用时提供兜底。

三、分布式锁：从MySQL到Redis的跨越

与MySQL的对比：MySQL可通过SELECT ... FOR UPDATE等行锁实现分布式锁，但性能差、易死锁。Redis更轻量、高性能，更受欢迎。

  • 基础实现：使用SET key value NX PX 30000命令，该命令是原子操作，代表 **”仅在key不存在时（NX）设置，并附加30000毫秒的过期时间（PX）”**。
  • 防误删：释放锁时不能简单地DEL key，必须通过Lua脚本原子性地检查锁的value（通常是UUID）是否属于当前线程，是才删除；避免线程A误删了线程B的锁。
  • 自动续期：如果一个业务执行时间可能超过锁的过期时间，需要一个Watch Dog机制来自动延长锁的持有时间，避免锁提前释放导致并发问题。
  • Redisson：在实际项目中，推荐使用Redisson来实现分布式锁，它封装了上述所有细节，可靠性更高。

四、内部落地手册

1. 数据结构选择（对比MySQL，选择更灵活）：缓存用户信息用Hash（对应MySQL的一行记录，key为ID，field为列名），避免反复序列化整个对象。实现评论列表用List（类似MySQL的ORDER BY create_time）。实现实时排行榜用Sorted Set（比MySQL用ORDER BY score DESC LIMIT N更高效）。实现用户签到用Bitmap，极其节省内存。

2. 高可用与集群部署：回答生产环境部署时，可以这样说：使用Redis Cluster架构实现水平扩展和自动故障转移。共部署5主5从，使用32G内存、8核CPU的机器，但每个Redis实例的内存限制在10G以内，避免单实例内存过大导致RDB快照和主从同步时出现卡顿。总缓存空间为50G，常驻约200万条商品数据，占用不到50%内存，留有充足余量。

3. 性能优化：避免在线生产环境使用KEYS *等慢命令，应使用SCAN命令进行渐进式遍历。使用Pipeline批量发送命令，减少网络往返时间（RTT）。在高并发写入时，推荐使用混合持久化（aof-use-rdb-preamble yes），结合RDB的快速恢复和AOF的数据安全性。
