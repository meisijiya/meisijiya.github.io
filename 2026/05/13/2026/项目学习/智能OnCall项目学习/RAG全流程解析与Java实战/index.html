3. RAG的Java代码实战

3.1 文档分片服务（DocumentChunkService）

关键设计点：

  • ✅ 优先按标题分割，保持语义完整性
  • ✅ 支持Markdown多级标题
  • ✅ 相邻chunk有overlap，避免信息丢失

3.2 向量化服务（VectorEmbeddingService）

关键设计点：

  • ✅ 使用DashScope官方SDK
  • ✅ 区分document和query两种向量化模式
  • ✅ 支持批量向量化

3.3 向量检索服务（VectorSearchService）

关键设计点：

  • ✅ L2欧氏距离相似度算法
  • ✅ Top-K返回最相似的K个结果
  • ✅ 返回内容+元数据+相似度分数

3.4 RAG核心服务（RagService）

关键设计点：

  • ✅ 检索 → 上下文构建 → LLM生成的完整流程
  • ✅ 支持多轮对话历史
  • ✅ 流式输出提升用户体验
  • ✅ 系统提示词约束回答基于文档

3.5 文档索引服务（VectorIndexService）

关键设计点：

  • ✅ 完整的文档索引管道
  • ✅ 批量向量化提高效率
  • ✅ 元数据序列化存储

4. RAG召回实战

🎯 召回流程详解

📊 相似度算法对比

算法 公式 特点 适用场景
L2 (欧氏距离) $d = \sqrt{\sum(x_i - y_i)^2}$ 考虑向量方向和大小 本项目使用
Cosine (余弦相似度) $\cos(\theta) = \frac{A \cdot B}{|A| |B|}$ 只考虑向量方向 文本相似度
IP (内积) $IP = A \cdot B$ 考虑向量方向和大小 推荐系统

🔧 召回优化策略

策略1：Top-K调优

策略2：相似度阈值过滤

策略3：文档分片优化

分片策略建议：

  • ✅ 按标题分割，保持语义完整性
  • ✅ chunk大小：200-1000 tokens
  • ✅ overlap：50-100 tokens
  • ✅ 避免过小chunk（语义不完整）

6. 面试话术与总结

🎯 面试话术

面试官：”介绍一下你的RAG系统是怎么设计的？”

你：

“我的RAG系统分为离线和在线两个阶段：

离线阶段：文档上传后，通过DocumentChunkService进行智能分片，优先按Markdown标题切分，保持语义完整性。然后通过DashScope Embedding API将文本向量化为1536维向量，最后存储到Milvus向量数据库。

在线阶段：用户提问时，先将问题向量化，然后在Milvus中进行L2欧氏距离相似度检索，返回Top-3最相关的文档片段。最后将检索结果作为上下文，调用大模型生成回答。

亮点是：我设计了InternalDocsTools这个Agent工具，让对话Agent和运维Agent都能通过Function Calling机制调用RAG能力，实现了一次开发，多场景复用。”
