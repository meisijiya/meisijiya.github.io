1. 项目诞生原因：运维痛点

😫 传统运维的痛点

💡 解决方案：智能OnCall Agent

智能OnCall Agent 能够像一位资深工程师一样工作：

  • 自动接收告警
  • 按照标准化流程进行排查
  • 分析根因并给出处理建议
  • 甚至还能自动执行修复操作

彻底将我们从重复的劳动中解放出来！

2. 智能OnCall Agent是什么？

🎯 一句话定义

OnCall Agent = 基于AI的运维自动化助手

核心目标：取代人工，完成那些高频、标准化的告警处理工作

📌 四大核心维度

维度 说明 效果
自动响应 实时接收监控系统告警通知 告别人工值守
智能排查 自动调用日志、监控工具，聚合多方数据定位问题 排查效率提升10x
根因分析 AI分析日志、指标、文档，定位问题根因 减少误判
闭环沉淀 处理过程自动沉淀为知识文档 经验可传承

🆚 对比传统运维

3. 核心功能拆解

🗺️ 功能全景图

📚 3.1 知识库Agent

功能：将散落的技术文档转化为可检索的知识资产

核心价值：

  • ✅ 一次开发，多场景复用
  • ✅ 知识资产统一管理
  • ✅ 新人快速上手

💬 3.2 对话Agent

功能：智能问答，基于知识库回答运维问题

核心价值：

  • ✅ 7x24小时在线问答
  • ✅ 减少重复咨询
  • ✅ 回答质量一致

🚨 3.3 运维Agent（核心亮点）

功能：自动化告警排查，生成诊断报告

核心价值：

  • ✅ 跨系统联动（日志+监控+文档）
  • ✅ 标准化排查流程
  • ✅ 自动生成报告

6. 面试亮点：怎么讲这个项目？

🎯 架构设计亮点

亮点1：知识库Agent = 统一数据资产

面试话术：

“我设计的知识库Agent不是简单的文档上传功能，而是一个统一知识中台。它将散落在各处的技术文档、告警手册、历史工单转化为可检索的向量数据资产，一次开发，对话Agent和运维Agent都能复用。”

技术深度：

  • 向量化存储（Milvus）
  • 语义检索（Embedding相似度）
  • 多场景复用（API设计）

亮点2：对话Agent = 中台知识引擎

面试话术：

“对话Agent不是简单的问答机器人，我把它抽象成了一个中台知识引擎。它把’重复咨询→文档检索→问题匹配’的逻辑封装起来，支持多场景无缝复用。”

技术深度：

  • RAG检索增强生成
  • 多轮对话上下文管理
  • 流式输出优化体验

亮点3：运维Agent = 跨系统联动架构

面试话术：

“运维Agent实现了跨系统联动，打破了日志、监控、告警群、文档之间的信息孤岛。通过Planner-Executor-Replanner架构，实现了标准化的告警排查流程。”

技术深度：

  • 多Agent协作（SupervisorAgent调度）
  • ReAct推理框架
  • 结构化任务执行

💡 技术实现亮点

Agent 亮点 面试话术
知识库Agent 一次开发，多场景受益 “设计了统一的向量检索API，对话Agent和运维Agent都能直接调用”
对话Agent 动态任务分解，减少幻觉 “通过RAG检索真实文档作为上下文，让AI基于事实回答，减少幻觉”
运维Agent 结构化任务执行 “采用Planner-Executor-Replanner闭环，确保排查流程标准化、可追溯”

🗣️ 面试回答模板

面试官：”介绍一下你的智能OnCall Agent项目”

你：

“这是一个基于AI的运维自动化助手，核心解决运维团队的三个痛点：

  1. 告警风暴：自动接收和分类告警
  2. 排查低效：自动调用日志、监控工具定位问题
  3. 知识碎片化：统一管理运维知识资产

系统有三个核心Agent：

  • 知识库Agent：将文档转化为可检索的向量数据
  • 对话Agent：基于RAG的智能问答
  • 运维Agent：多Agent协作的告警诊断

技术栈是Spring Boot + Spring AI + Milvus + DashScope，亮点是统一知识中台设计和Planner-Executor-Replanner架构…”
