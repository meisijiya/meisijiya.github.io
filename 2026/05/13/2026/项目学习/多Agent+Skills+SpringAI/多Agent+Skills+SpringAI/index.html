本文完整整合《多Agent+Skills+SpringAI》系列学习项目全章节内容，参考资料时间范围为2025.10~2026.1，官网参考内容截止2026.4.30。AI技术迭代极快，请注意鉴别过时内容。
学完你将掌握：Spring AI Alibaba全版本特性、MCP/A2A协议落地、ReAct/PlanAct思考框架、AgentScope分布式框架、Skills技能体系、多Agent协同设计、旅游规划场景全流程落地、工程化部署与可观测能力。

一、项目背景与场景拆解：我们要做什么？

本系列项目以旅游规划为落地场景，逐步搭建从单Agent到分布式多Agent协同的完整系统，帮你理解多Agent设计、工具集成、工程化落地的全流程。

1.1 旅游规划核心需求拆解

旅游规划并非单一需求，而是由四类强关联的核心需求构成，需先拆解再匹配专属能力：

需求类型 核心动作 说明
景点 ① 景点推荐（结合用户偏好+天气） ② 景点路线规划 需同时满足用户喜好与实际天气（如恶劣天气不推荐户外项目）
交通 路线规划 包含景点间、景点与住宿间的路线串联
费用 费用计算+折扣监控 固定费用（交通、门票无折扣）+浮动费用（住宿、餐饮可监控折扣）
住宿餐饮 选址+推荐 结合预算、位置、用户偏好输出方案

对应需要搭建5类智能体组成完整团队：

智能体类型 属性 核心职责
规划智能体（主管） 通用 识别用户意图、拆解子需求、根据能力分配任务
路线制定专员Agent 专属 全场景路线规划、经纬度转换、地图可视化
行程规划经理Agent 专属 景点/餐饮推荐、行程安排、天气适配、本地文件读取
费用统筹管家Agent 专属 全链路费用统计、折扣监控、表格制作
汇总智能体 通用 整合所有智能体输出，生成含文字/图片/表格的结构化文档

还可集成百度地图MCP提供的5类工具提升能力：天气查询、地理编码、地点搜索、路线规划、地图可视化。

1.2 多Agent协同设计理论

（1）两种核心设计模式

设计模式 核心特点 代表框架 适用场景
工作流编排模式 基于Graph引擎预定义节点、边、协作关系，流程可视化可控 Spring AI Alibaba Graph引擎 流程固定、节点关系明确、需要可视化编排
Agent自主代理模式 智能体自主决定执行步骤、工具使用，无需预定义流程 AgentScope、Spring AI Alibaba Flow Agent 需要智能体动态分配任务、调用工具

（2）思考框架：PlanAct vs ReAct

对比维度 Plan-and-Execute（PlanAct） Reason-and-Action（ReAct）
任务复杂度 多层级嵌套复杂任务 线性单一简单任务
核心逻辑 两阶段：规划阶段拆解子任务+排序，执行阶段监控调整 两步骤：思考阶段明确步骤与工具，行动阶段执行+校验调整
资源分配 需要多智能体协作 通常单智能体完成
调整频率 动态调整频率高，侧重全局计划优化 侧重执行过程优化

（3）行业参考架构：Manus与JManus

  • Manus：首创「自主决策+隔离容器+工具调用」的多Agent闭环架构，全自主流程无需人工介入，每个Agent在独立容器运行避免数据混乱，支持MCP协议调用第三方工具，是生产力落地的标志性框架。
  • OpenManus：Python版Manus复刻，核心特点：智能体模块化即插即用、工具箱支持MCP动态扩展、规划+执行双主干处理复杂任务。
  • JManus：Java版OpenManus，基于Spring Boot + Spring AI Alibaba开发，核心为计划协调器，管理计划全生命周期：
    • 简单计划用PlanExecutor：拆分步骤→分配Agent→顺序执行
    • 复杂计划用MapReducePlanExecutor：拆分分支→子任务并行/串行执行→汇总结果

（4）多Agent框架核心组成

双核心流程+五大模块：

  1. 双核心流程：
    • 多Agent协同流程：用户需求→意图识别→任务拆解→分配Agent→执行→结果整合→返回
    • 单Agent执行流程：步骤规划→执行→监测→评估→调整→循环至完成
  2. 五大模块：全局模块（Env环境类、Action动作类）、任务切分模块、角色模块、通信及冲突协调模块（A2A协议、冲突调解、WSLL共识算法）、SOP标准流程模块。

2.3 MCP协议：模型上下文交互

MCP（Model Context Protocol）是连接大模型与外部数据/工具的通用标准，解决大模型无法获取实时数据、私人数据的问题。

（1）核心架构

三节点：MCP Host（大模型端）、MCP Client（客户端，基于Spring AI Alibaba实现）、MCP Server（服务端，提供工具/数据）。
分层设计：协议层统一工具返回格式，传输层支持多种方式，大模型无需关心调用细节。

（2）1.1+版本传输方式（共4种）

传输方式 适用场景 特点 推荐框架
stdio 同机器进程间通信 轻量，本地MCP Server/Client通信 无特殊要求
SSE 网页端与远程模型通信（默认） 单向实时流式，比WebSocket轻量 WebFlux（非阻塞，性能优于WebMVC）
Streamable HTTP SSE升级版 流式传输，支持更多交互场景 WebFlux
Stateless HTTP 微服务/分布式场景 无状态流式传输 WebFlux

⚠️ 所有流式传输均推荐使用Spring WebFlux，单线程可处理多请求，性能优于传统WebMVC。

2.4 A2A协议：分布式智能体通信

A2A（Agent-to-Agent）协议由Google提出，用于跨模型、跨节点智能体通信，Spring AI Alibaba 1.1+原生支持，核心依赖Nexus 3.1注册中心。

（1）核心组件

  1. 服务提供者（Provider）：提供远程智能体服务，需注册到Nexus
  2. 服务消费者（Consumer）：从Nexus发现并调用远程智能体
  3. 注册中心：阿里Nexus 3.1/Nacos 3.2，存储智能体卡片（AgentCard）

（2）AgentCard规范（服务发现核心依据）

元数据字段 说明 要求

description 智能体功能描述 必填，消费者靠此判断智能体能力
instruction 智能体执行指令 必填，缺失会导致调用报错
skills 关联的技能列表 可选，精细匹配能力

3.3 工作流编排：Graph引擎实战

Spring AI Alibaba 1.1支持两种编排方式，可按场景选择：

对比项 Graph引擎 Flow Agent基类
核心实现 StateGraph定义节点、边、状态 继承FlowAgent，组合子智能体
执行单元 函数/方法的节点（Node） 具备思考、工具使用能力的智能体
流程定义 通过边（Edge）预定义流程 智能体自主决策流程，支持并行/顺序/路由分发
适用场景 流程固定、节点关系明确、需要可视化编排 需要智能体自主决策、动态任务分配

✅ 建议：需要智能体自主决策优先选Flow Agent；流程固定的场景优先选Graph引擎。

（1）Graph核心三要素

  1. 状态（OverAllState）：贯穿工作流的共享内存，Key-Value结构，更新策略：
    • ReplaceStrategy：新值覆盖旧值，适合存最终结果
    • AppendStrategy：新值追加到旧值后，适合存多轮消息
    • RemoveByHash：根据hashCode删除指定值，适合清理历史记录
  2. 节点（Node）：执行单元，支持同步（NodeAction）、异步（AsyncNodeAction）两种类型，输入输出均基于OverAllState。
  3. 边（Edge）：定义节点流转关系，支持普通边（固定流转）、条件边（根据路由函数匹配节点）、特殊节点（START开始、END结束）。

（2）旅行规划Graph工作流案例

目标：搭建5智能体旅行规划工作流，支持费用超1000时退回行程规划重排

  1. 节点：主管（任务分发）、路线规划、行程规划、费用统筹、汇总
  2. 边：
    • START→主管节点
    • 主管条件分发：路线相关→路线规划，行程相关→行程规划
    • 路线/行程并行执行→费用统筹节点
    • 费用统筹条件判断：总费用<1000→汇总节点→END；≥1000→行程规划节点（重排）

（3）高级特性

  • 持久化与检查点：保存工作流执行状态，支持断点续跑、历史回溯，内置MemorySaver（测试），支持自定义RedisSaver（生产）。
  • 人机协作（HITL）：工作流执行到指定节点前中断，等待人工输入后恢复执行，依赖检查点机制保存/恢复状态。
  • 流式输出：原生支持Flux作为流式载体，与Spring WebFlux天然契合，可通过SSE协议返回前端实现打字机效果。

四、实战篇：分布式旅游规划落地（AgentScope框架）

AgentScope是面向工程实践的Java智能体框架，基于Spring AI Alibaba 1.1开发，比原生封装更优，课程后续替代原生Agent开发。核心特性：ReAct推理-行动范式、全程可控（安全中断、优雅取消、人机协作）、生产级能力（长期记忆、RAG、MCP/A2A支持）、响应式架构（基于Project Reactor）。

六、附录：关键避坑指南与最佳实践

通用最佳实践

✅ 依赖管理：优先使用BOM方式统一管理版本，避免版本冲突
✅ 注解规范：@Tool/@McpTool的description必填，否则大模型无法识别工具用途
✅ 通信选择：MCP传输优先选WebFlux框架，A2A通信优先用Nacos 3.2注册中心
✅ 配置规范：AgentCard的name、description、instruction必填，shareState设为true
✅ 记忆管理：短期记忆用checkpointer持久化，长期记忆用Store存储，结合使用提升交互能力
✅ Skills编写：description尽量丰富细节包含触发关键词，SKILL.md控制在500行以内，详细文档放到参考文件中

常见错误排查

❌ Agent注册报错：检查依赖是否仅放在需要注册的模块，避免无需注册的Agent（如主管Agent）引入注册依赖
❌ MCP连接失败：检查客户端与服务端传输协议是否一致，API Key是否正确
❌ 任务调用类型报错：统一使用AgentBase作为Agent类型参数，兼容本地和远程Agent
❌ 工具未加载：MCP客户端注册到Toolkit时需加block()阻塞，等待工具加载完成
❌ 模型不支持：结构化输出、思考模式需提前确认模型支持（如qwen3系列支持思考模式，DeepSeek R1免费版支持思考链）
