题目：239. 滑动窗口最大值

给你一个整数数组 nums，有一个大小为 k 的滑动窗口从数组的最左侧移动到数组的最右侧。你只可以看到在滑动窗口内的 k 个数字。滑动窗口每次只向右移动一位。

返回 滑动窗口中的最大值 。

一、思考

第一轮人机对话

我：既然是固定滑动窗口且每次只向右移动一位，那么，初始时比较初始窗口的各个元素将最大值值存放到变量A。接下来，每次窗口向右滑动，先比较划出窗口的元素是否等于A，等于则重新在新窗口中比较得到最大值，不等于的话只需要将滑动前的窗口末位置加一的元素和A比较，取最大存到A即可。

AI：目前算法实现：朴素滑动窗口 + 条件性重扫描

  • 时间复杂度：最坏情况 O(n×k)，如递减数组每次滑动都需重新遍历窗口
  • 空间复杂度：O(1) 额外空间

原算法在大数据集下会超时（题目约束 nums.length ≤ 10^5），O(n×k) 无法通过全部测试用例

请你进一步思考时间复杂度，有没有一种数据结构，能在最大值失效时，快速获取下一个候选最大值，而不需要重新遍历整个窗口？

采用单调队列方案，这是滑动窗口最大值问题的标准最优解

第三轮人机对话（含算法总结）

我：补充目前代码框架的思考。

  • 步骤1：应当剩余数组的元素个数大于k时，才需要队首索引移除；
  • 步骤2：移除当前元素小的，因为加入的是大的话，小的又用于会先于后加入的出队，所以不可能再成为最大值；
  • 步骤4：因为数组从0开始的，所以要k-1。最大值在队列的队首。

AI：你的理解修正与补充：

步骤 你的理解 补充说明
步骤1 “剩余元素个数大于k” 更准确：队首索引是否小于窗口左边界 i-k+1，即 peekFirst() <= i-k
步骤2 ✓ 正确 大元素”覆盖”小元素，因为大元素更大且存活更久
步骤4 ✓ 正确 索引 k-1 时窗口首次包含k个元素 [0, k-1]

一、题目核心
要素 描述
输入 整数数组 nums + 窗口大小 k
输出 每个滑动窗口的最大值组成的数组
约束 nums.length ≤ 10^5，要求高效解法
难点 窗口滑动时，最大值可能失效，需快速找到新最大值
二、两种解法对比
维度 朴素解法 单调队列（最优）
核心思想 最大值滑出时重新扫描窗口 双端队列维护候选最大值
时间复杂度 O(n×k) 最坏 O(n)
空间复杂度 O(1) O(k)
关键操作 条件性遍历 队首过期检查 + 队尾单调维护
能否通过 ❌ 大数据集超时 ✅ 全部通过
三、单调队列核心技巧

为什么每个元素只入队出队一次？

  • 入队：遍历到它时入队一次
  • 出队：要么被更大元素淘汰（队尾），要么过期（队首）
  • → 均摊 O(1)，整体 O(n)

考点1：单调队列思想

核心问题： 面试官会问 “为什么想到用单调队列？”

回答思路：

面试金句：

“单调队列本质是剪枝——提前淘汰那些’既小又早死’的候选者，只保留有可能成为最大值的元素。”

类似场景：

  • 求滑动窗口最小值 → 单调递增队列
  • 求窗口内最大-最小≤limit → 双单调队列（同时维护最大和最小）

三、面试模拟问答

面试官问题 期望回答要点
“这道题能用堆吗？” 可以，但堆删除任意元素是O(k)，整体O(nlogk)，不如单调队列O(n)
“如果窗口大小可变呢？” 单调队列仍适用，只需调整过期判断条件
“并行化能加速吗？” 分块处理后合并，但单调队列本身是顺序依赖的，并行收益有限
“空间能优化到O(1)吗？” 不能，最坏情况（递增序列）队列需要存储k个索引
