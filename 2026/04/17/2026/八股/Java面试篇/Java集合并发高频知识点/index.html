Java并发--面试

作者: 老江湖
日期: 四月 17, 2026  13:58:02
分类: Java面试篇

Hi，我是 meisijiya ，也是 老江湖 ，感谢你的阅读，我们一起进步。

一、整体定位

Java并发是中高级后端面试必考模块（像我这种嘞，就……学！），覆盖底层JMM、锁机制、线程池、并发场景；

核心高频考点：synchronized/ReentrantLock、volatile/CAS、ThreadLocal、线程池、死锁。

二、多线程基础

1. JMM – Java内存模型

  • 核心目标：作为一套规则，用于解决多线程可见性、原子性、有序性问题
    • 可见性：线程修改值其他线程立即可见
    • 原子性：操作不可中断
    • 有序性：禁止不合理指令重排。
  • 内存结构：主内存（共享变量）+ 线程工作内存（私有缓存）
  • 保障机制：volatile（可见性+禁指令重排）、synchronized（同时保证三大特性）

2. 线程核心特性

  • 线程模型：
    • 传统Java线程（平台线程）：OS 内核线程，1:1 模型，调度由 OS 负责，开销大；
    • Java21虚拟线程：JVM 用户态调度，M:N 模型，轻量级，映射少量 OS 线程，接近 Go 协程。
  • 线程6种状态：NEW→RUNNABLE→BLOCKED/WAITING/TIMED_WAITING→TERMINATED
  • BLOCKED vs WAITING：前者被动等锁（线程尝试获取synchronized内置排他锁时竞争失败，被 JVM 放入锁池阻塞，仅当锁释放且抢锁成功时才能恢复），后者主动等待唤醒（wait/join/park）：
    • wait()必须在synchronized同步块内，调用锁对象本身的wait()，唤醒也必须调用同一个对象的notify()/notifyAll()
    • 当前线程调用目标线程对象的join()方法，当前线程进入WAITING状态，直到目标线程执行终。
    • park()是LockSupport的静态方法，直接调用LockSupport.park()就会让当前线程进入WAITING状态；唤醒时调用LockSupport.unpark(目标线程)即可。
  • sleep vs wait：
    • ①所属类：sleep是 Thread 静态方法，wait是 Object 实例方法；
    • ②锁释放：sleep虽然不释放锁但会主动让出CPU时间片，wait释放锁；
    • ③使用前提：sleep任意场景，wait需在同步块内；
    • ④唤醒：sleep超时自动醒，wait需notify（JDK：随机唤醒一个等待线程；HotSpot：先进先出顺序唤醒）/notifyAll（唤醒所有等待线程，所有线程重新竞争锁）唤醒。

3. 线程操作

  • 创建方式：

    • 继承Thread重写其run()方法，创建该类的实例后，通过调用start()方法启动线程。。
    • 实现Runnable/Callable重写run()方法，然后将此Runnable对象作为参数传递给Thread类的构造器，创建Thread对象后调用其start()方法启动线程。使用Thread.currentThread()方法访问当前线程。
    • 实现Callable接口，特别的，Callable的call()方法有返回值，执行任务前，需要将它包装进一个FutureTask（实现了Runnable接口），因为Thread类的构造器只接受Runnable参数。

  • 启动：必须调用start()，而非直接执行run()

  • 安全停止：

    • interrupt()中断标志：thread.interrupt() 不会立刻停止线程，只是给线程打一个中断标记 = true，线程在合适时机检查这个标记，自己决定是否退出。
      如果线程正在 sleep()、wait()、join()，别的线程调用 interrupt() 会：

      • 解除阻塞并抛出 InterruptedException
      • 同时清除中断标记（设为 false）

      这里应当要捕获异常并重新设置中断。

    • volatile标志位：定义一个 volatile boolean 变量，线程循环判断该变量，外部线程修改为 false，该线程退出。

    • 线程池 Future.cancel(boolean mayInterruptIfRunning)：线程池场景下停止任务的标准方式，底层也是基于 interrupt() 实现。任务内部必须配合 interrupt() 机制（检查中断标志、处理 InterruptedException），否则 cancel(true) 也无法停止任务。

    • 禁用 stop()、suspend() /resume ()

  • 线程通信：

    • 基础核心通信方式 wait/notify/notifyAll：最原生最基础的等待唤醒机制，Object类的本地方法，基于对象的内置锁synchronized实现（必须在synchronized同步块 / 方法内调用，否则直接抛出IllegalMonitorStateException），通过Java对象自带的核心队列1️⃣锁池（竞争锁的阻塞队列） + 2️⃣等待集（wait 的等待队列），实现线程的「等待 - 唤醒」协同。

    • 进阶通信方式 JUC 工具集(java.util.concurrent)：

      AQS（AbstractQueuedSynchronizer，抽象队列同步器）:使用一个整型的 volatile变量（state） 来表示同步状态（如锁的重入次数、信号量许可数），并通过一个内置的 FIFO双向队列（MCS：CLH队列变体。每个线程自旋查看自己阶段的locked状态 ，由前一节点释放锁后通知） 来管理那些竞争资源失败的线程

      • Lock + Condition（精准唤醒版 wait/notify）：Condition是 JUC 提供的等待唤醒接口，基于Lock显式锁实现，一个Lock可以绑定多个Condition独立等待队列
      • volatile 共享内存（最轻量的状态通信）：基于Java 内存模型（JMM）的可见性保证实现通信，修改后会立刻强制刷新回主内存，保证多线程之间的状态可见性。
      • CountDownLatch（一次性倒计时门闩，多对一协同）：基于 AQS（抽象队列同步器）实现，一个线程等待多个线程完成全部任务后，再继续执行，一次性使用。
      • CyclicBarrier（循环栅栏，多对多协同）：基于ReentrantLock+Condition实现，一组线程互相等待，所有线程都到达栅栏位置后，再一起继续执行，支持循环重置使用。
      • Semaphore（信号量，流量控制通信）：基于 AQS 实现，通过控制许可数量，限制同时访问共享资源的线程数量。
      • BlockingQueue（阻塞队列，生产者消费者终极解决方案）：JUC 提供的线程安全阻塞队列接口，底层已经封装好了Lock+Condition的等待唤醒机制 1️⃣队列为空时，消费者线程调用take()会自动阻塞，直到队列有数据。2️⃣队列满时，生产者线程调用put()会自动阻塞，直到队列有空闲位置。

4. Java线程 vs Go协程

  • ①调度：Java 平台线程是 OS 内核调度，Go 协程是用户态 GMP 调度；
  • ②资源：Java 线程栈 1MB，Go 协程初始 2KB 可伸缩；
  • ③模型：Java 传统 1:1，Go M:N；
  • ④通信：Java 靠共享内存 + 锁，Go 靠 Channel；
  • ⑤Java21 虚拟线程对齐 Go 协程能力。

5.数据一致性保证

  • 数据库事务（ACID）：原子性保证操作不中断，一致性保证数据合法，隔离性保证并发不乱，持久性保证结果不丢。

    • Atomicity（原子性）：事务是最小执行单元，要么全做，要么全不做。实现：数据库回滚日志（undo log）

    • Consistency（一致性）：事务执行前后，数据的完整性约束不变，数据库始终合法。

    • Isolation（隔离性）：多个事务同时运行时，互相不干扰，避免脏数据。

    数据库隔离级别（从低到高）：

    读未提交 → 读已提交（常用）→ 可重复读（MySQL 默认）→ 串行化

    隔离性解决以下并发问题：

    1. 脏读：读到其他事务未提交的数据
    2. 不可重复读：同一事务内，两次查询结果不一样
    3. 幻读：范围查询时，突然多出 / 少了数据
    • Durability（持久性）：事务一旦提交，永久生效，断电、宕机都不会丢失。实现：数据库重做日志（redo log）

  • Java 锁机制（synchronized/ReentrantLock）：解决多线程并发修改共享数据导致的线程安全问题（比如超卖、数据错乱）。

    • synchronized关键字（内置锁）：1️⃣JVM 层面实现，自动加锁、自动释放锁；2️⃣可重入锁（同一个线程可以重复获取锁）；3️⃣非公平锁（线程抢锁随机，不按排队顺序）

  • 乐观锁版本控制。

三、并发安全（核心模块）

1. 线程安全三大特性保障

  • 原子性：synchronized、Lock、Atomic原子类
  • 可见性：volatile、synchronized、Lock
  • 有序性：volatile、synchronized、happens-before规则

2. 锁体系

(1) synchronized

  • 内置监视器锁，自动加锁/解锁
  • 锁升级（JDK1.6+）：无锁→偏向锁→轻量级锁→重量级锁（JDK15+默认禁用偏向锁）
  • 锁对象：普通方法锁当前实例，静态方法锁Class对象

(2) ReentrantLock

  • 基于AQS实现，手动加解锁
  • 支持：公平/非公平锁、可中断、多Condition条件队列

(3) 其他锁

  • 读写锁：读共享、写独占，适配读多写少场景
  • 可重入锁：同一线程可重复获取锁（synchronized/ReentrantLock均支持）
  • 公平锁：按申请顺序获取锁；非公平锁：插队竞争，吞吐量更高

3. 悲观锁 vs 乐观锁

  • 悲观锁：默认存在竞争，加锁执行（synchronized、ReentrantLock）
  • 乐观锁：无锁设计，用CAS+版本号实现
  • CAS缺陷：ABA问题（AtomicStampedReference解决）、自旋耗CPU、仅支持单个变量原子操作

4. volatile

  • 作用：保证可见性、禁止指令重排
  • 局限：不保证原子性，无法完全保证线程安全

5. AQS 抽象队列同步器

  • 核心：volatile状态state + FIFO等待队列，通过CAS修改状态
  • 应用：ReentrantLock、CountDownLatch、Semaphore等均基于AQS实现

6. ThreadLocal

  • 作用：提供线程私有变量，避免共享竞争
  • 原理：Thread持有ThreadLocalMap，key为ThreadLocal（弱引用），value为业务数据
  • 问题：内存泄漏（key被回收后value强引用滞留）
  • 解决：使用后手动调用remove()清理

7. JUC常用工具

  • CountDownLatch：主线程等待一组子线程执行完毕
  • CyclicBarrier：一组线程相互等待至屏障点，可复用
  • Semaphore：控制同时访问资源的线程数
  • 并发集合：ConcurrentHashMap、CopyOnWriteArrayList

8. 死锁

  • 四大必要条件：互斥、请求保持、不可剥夺、循环等待
  • 解决方式：破坏任一条件、避免循环等待、使用定时锁

四、线程池

  1. 核心价值：复用线程，降低创建/销毁开销，控制并发线程数
  2. 核心参数：核心线程数、最大线程数、工作队列、拒绝策略、线程空闲时间
  3. 拒绝策略：抛异常、丢弃任务、丢弃最老任务、调用者执行
  4. 关闭方式：shutdown()（温和关闭）、shutdownNow()（强制中断）

五、典型并发场景题要点

  1. 线程顺序控制：wait/notify、Lock、CountDownLatch
  2. 单例双重校验：volatile + synchronized，防止指令重排
  3. 多线程并发累加：原子类、加锁保证原子性
  4. 等待多线程完成：使用CountDownLatch
