一、基础篇

1.1 MySQL整体执行流程

MySQL采用Server层+存储引擎层的解耦架构：

  • Server层（所有存储引擎共用）：负责连接管理、SQL解析、优化、执行，包含：

    • 连接器：管理客户端连接，验证用户名密码，读取用户权限（连接建立后权限修改不影响已建立的连接），默认长连接，空闲超时由wait_timeout控制（默认8小时）。

      命令行连接：mysql -h$ip -u$user -p

      查看客户端连接：show processlist

      手动断开 空闲/长 连接（避免超出连接数上限/释放内存）：kill connection + id

      客户端主动重置连接（释放内存）：代码调用mysql_reset_connection

      MySQL 基于 TCP 协议进行传输

    • 查询缓存：MySQL 8.0已删除该模块，8.0之前会缓存查询语句和结果，表有更新就会清空缓存，命中率极低。

      这里说的查询缓存是 server 层的，也就是 MySQL 8.0 版本移除的是 server 层的查询缓存，并不是 Innodb 存储引擎中的 buffer pool。

    • 解析器：做词法分析（识别关键字、表名、字段名）、语法分析（判断SQL是否符合MySQL语法），构建语法树；表/字段是否存在不在该阶段校验。

    • 预处理器（prepare）：校验表/字段是否存在，将select *扩展为表所有列。

    • 优化器（optimize）：基于查询成本选择最优执行计划，决定使用哪个索引。

    • 执行器（execute）：调用存储引擎接口执行SQL，返回结果。

  • 存储引擎层：负责数据存储和提取，插件式架构，默认InnoDB，支持MyISAM、Memory等。

    InnoDB 支持索引类型是 B+树 ，且是默认使用，也就是说在数据表中创建的主键索引和二级索引默认使用的是 B+ 树索引。

1.2 一行记录的存储结构

InnoDB默认行格式为Dynamic（MySQL 5.7+），兼容Compact格式，一行记录分为两部分：

  • 记录的额外信息：

    • 变长字段长度列表：存储变长字段（如varchar）的实际占用字节数，仅当数据表有变长字段时才存在，逆序存放。
    • NULL值列表：用1字节存储NULL值的列（当大于八个字段则2字节存储，以此类推），若所有字段都定义NOT NULL则无该部分，逆序存放。
    • 记录头信息：存储记录的元数据（如是否删除、下一条记录位置、当前记录的类型等）。

  • 记录的真实数据：

    • 隐藏字段：row_id（无主键/唯一索引时自动生成，6字节）、trx_id（事务ID，6字节）、roll_pointer（回滚指针，7字节，指向undo log版本链）。

  • 用户定义的字段数据。

1.3 表空间与数据文件

表空间按 **行 ⊂ 页 ⊂ 区 ⊂ 段 ⊂ 表 **空间层级组织：

  • 行（row）是数据库表记录的存储单位，表中数据按行存放；每行记录的存储结构由InnoDB行格式（如Compact、Dynamic(默认)、Compressed等）定义，最终存放在页中。

  • 页（page）是InnoDB存储引擎磁盘管理的最小单元，也是IO读写的基本单位：

    • 默认大小为16KB，每次磁盘读写都以页为单位整体操作（单次最少读写16KB内容）；

    • 类型包括存储行记录的数据页、undo日志页、溢出页等；

    • 64个连续的页可组成1个区；

    • 读取记录时，会将记录所在的整个页读入内存，而非仅读取单行。

      InnoDB 数据页 7 个核心部分：

      • 文件头（38字节）：存储页基础信息，用指针串联上下页，形成双向链表
      • 页头（56字节）：记录页内记录数量、空间状态等元数据
      • 最大 / 最小虚拟记录（26字节）：划定数据边界，维系行记录单向链表
      • 用户记录（不确定大小）：存放真实业务行数据，按主键有序排列
      • 空闲空间（不确定大小）：页内未分配的剩余存储空间
      • 页目录（不确定大小）：记录分组槽位，通过二分快速定位页内记录
      • 文件尾（8字节）：数据校验，保障数据页完整性

  • 区（extent）是固定大小为1MB的存储单位，由64个连续的16KB页组成：

  • 仅当表数据量较大时，InnoDB为索引分配空间会改用区为单位（不再按页分配），目的是保证B+树中相邻页的物理位置连续，避免随机IO，提升范围查询时的顺序IO性能。

  • 段（segment）是表空间的组成单位，由多个区构成，按用途分为三类：

    • 索引段：存放B+树非叶子节点对应的区集合；
    • 数据段：存放B+树叶子节点（即实际行记录）对应的区集合；
    • 回滚段：存放回滚数据的区集合，用于支撑MVCC机制实现多版本查询。

  • 文件结构：每个数据库对应/var/lib/mysql/库名目录，包含：

    • db.opt：存储数据库默认字符集和校验规则。
    • 表名.frm：存储表结构元数据（MySQL 8.0前）。
    • 表名.ibd：存储表数据和索引，innodb_file_per_table=1（MySQL 5.6.6+默认）时每张表对应独立.ibd文件，否则数据存在共享表空间ibdata1中。

1.4 基础概念补充

  • varchar(n)：n代表最多存储的字符数，不是字节数；一行记录除TEXT/BLOB外总字节数不能超过65535，还要减去变长字段长度列表、NULL值列表的空间，例如ascii字符集、允许NULL的单varchar字段，n最大为65532。

    当采用ascii字符集一字符占用1字节，因此理论来说可以存65535，减去了3，是因为其中变长字段长度列表可覆盖65536占用2字节，字段个数在八个以内占用1字节

  • 行溢出：当一行记录超过16KB（页大小），InnoDB会把部分数据存到溢出页：Compact格式会保留部分数据+20字节溢出页指针，Dynamic/Compressed格式只保留20字节指针，全量数据存溢出页。

  • 存储引擎对比：InnoDB（默认，支持事务、行锁、外键、crash-safe）；MyISAM（不支持事务、表锁、外键）；Memory（内存存储，支持哈希索引，重启数据丢失）。

二、索引篇

InnoDB 以 16KB 的数据页作为磁盘读写的最小单位：页间通过双向链表逻辑串联，页内行记录按主键组成单向链表，并借助页目录 + 二分查找实现页内快速检索。为了高效管理海量数据页，InnoDB 采用矮胖的 B+ 树构建索引：B+ 树中每个节点都是数据页，非叶子节点仅存储索引目录项，叶子节点有序串联并存放真实数据。索引分为两类：聚簇索引（唯一，叶子节点存储完整行数据）和二级索引（叶子节点仅存储主键值）；使用二级索引查询非索引字段时需要回表访问聚簇索引，若仅查询主键则可实现索引覆盖。

MySQL 之所以选用 B+ 树作为索引结构，核心原因在于磁盘 IO 速度远慢于内存，索引设计必须尽量减少磁盘 IO 并兼顾范围查询效率。相比之下，二叉查找树容易退化为链表；平衡二叉树 / 红黑树属于二叉结构，树高较大，会显著增加 IO 次数；B 树虽为多叉树，可降低树高，但其节点同时存储索引与完整数据，单页 IO 开销大，且范围查询效率低。而 B+ 树的非叶子节点仅存索引，可容纳更多索引项，树结构更矮胖、磁盘 IO 更少；同时冗余索引提升了增删效率，叶子节点通过双向链表相连，极大优化了范围查询性能。结合 InnoDB 引擎特性——聚簇索引叶子节点存放完整数据、二级索引叶子节点存放主键——MySQL 最终采用 B+ 树作为索引结构。

2.1 索引基础

  • 定义：类似书籍目录，用于加速查询，减少数据扫描量，是存储引擎层的数据结构。

  • 分类：

    • 按数据结构：B+Tree索引（默认）、Hash索引、Full-text索引。

    • 按物理存储：聚簇索引（主键索引）、二级索引（辅助索引）。

    • 按字段特性：主键索引（PRIMARY KEY）、唯一索引（UNIQUE KEY）、普通索引（INDEX）、前缀索引（INDEX，对字符类型字段的前几个字符建立的索引）。

    • 按字段个数：单列索引、联合索引。

      单列索引：

      • 建表时：INDEX idx_a (a)
      • 建表后：CREATE INDEX idx_a ON table_name(a);

      前缀索引：

      • 建表时：INDEX idx_a (a(10))
      • 建表后：CREATE INDEX idx_a ON table_name(a(10));

      唯一索引：

      • 建表时：UNIQUE KEY uk_a (a)
      • 建表后：CREATE UNIQUE INDEX uk_a ON table_name(a);

      联合索引：

      • 建表时：INDEX idx_ab (a, b)
      • 建表后：CREATE INDEX idx_ab ON table_name(a, b);

      联合唯一索引：

      • 建表时：UNIQUE KEY uk_ab (a, b)
      • 建表后：CREATE UNIQUE INDEX uk_ab ON table_name(a, b);

      主键索引：

      • 建表时：PRIMARY KEY (id)
      • 建表后：ALTER TABLE table_name ADD PRIMARY KEY (id);

2.2 B+树索引原理

InnoDB默认使用B+树作为索引结构，对比其他结构优势：

  • 对比B树：B+树非叶子节点只存索引，不存数据，相同页大小下能存更多索引，树更矮胖，磁盘IO更少；叶子节点用双向链表连接，支持高效范围查询，B树范围查询需要中序遍历，效率低。
  • 对比二叉树：二叉树每个节点最多2个子节点，数据量大时树高过高，磁盘IO次数多；B+树节点子节点数可以很大（d>100），千万级数据树高仅3-4层，IO次数3-4次。
  • 对比Hash：Hash仅支持等值查询，O(1)复杂度，但不支持范围查询；B+树同时支持等值、范围查询，适用场景更广。

2.3 聚簇索引与二级索引

  • 聚簇索引（主键索引）：叶子节点存储完整行数据，一张表仅有一个，选索引优先级：有主键用主键→无主键用第一个非空唯一列→都没有则生成隐式row_id作为聚簇索引。
  • 二级索引（辅助索引）：叶子节点仅存储主键值，查询时需要先查二级索引拿到主键，再回表查聚簇索引拿完整数据；若查询字段全部在二级索引中，则无需回表，称为覆盖索引。

2.4 联合索引与最左匹配

  • 联合索引：多个字段组合创建的索引，例如(a,b,c)，按字段顺序排序，先按a排序，a相同再按b排序，b相同再按c排序。

  • 最左匹配原则：查询条件需要从索引最左列开始匹配，否则索引失效：

    • where a=1 and b=2 and c=3：完整走联合索引。
    • where a=1 and c=3：仅a走索引，c通过索引下推（MySQL 5.6+）在存储引擎层先过滤，减少回表。
    • where b=2：完全不走联合索引。

  • 联合索引范围查询：范围查询的字段可以用到联合索引，但是在范围查询字段的后面的字段无法用到联合索引；但是对于大于等于和小于等于的等于部分，例如select * from t_table where a >= 1 and b = 2中的a = 1 的二级索引记录的范围里，b 字段的值是「有序」的，因此只有 a=1 and b=2 走了联合索引，a>2andb=2的部分b只起到了索引过滤的作用。

    因此对于 >=、<=、BETWEEN、like 前缀匹配的范围查询，并不会完全停止最左匹配

  • 创建原则：区分度高的字段放前面（区分度=字段不同值个数/总记录数），低区分度字段（如性别）不适合建索引或放前面。

2.5 索引失效场景

共6种常见情况：

  1. 左/左右模糊匹配：like %xx或like %xx%无法走索引，like xx%可以走索引。
  2. 对索引列使用函数：例如length(name)=6，索引存的是原始值，无法走索引，MySQL 8.0后支持函数索引可解决。
  3. 对索引列做表达式计算：例如id+1=10，等价于对索引列用函数，无法走索引，改为id=10-1可走索引。
  4. 索引列隐式类型转换：字符串列用数字查询时，MySQL会把字符串转数字，等价于对索引列用CAST函数，例如phone=13000000001（phone是varchar），会触发隐式转换导致索引失效；数字列用字符串查询不会失效。
  5. 联合索引不遵循最左匹配原则。
  6. where中or连接：只要有一个条件列没有索引，整个查询就会全表扫描（MySQL 5.1+支持index_merge优化，可能合并多个索引，但不保证生效）。

2.6 索引优化

  • 前缀索引：对大字符串字段取前N个字符建索引，减小索引大小；又因为前缀索引缺完整 字符串/字段值，所以无法用于order by、覆盖索引。

  • 覆盖索引：查询字段全部包含在索引中，避免回表，提升性能。

  • 主键自增：避免页分裂，提升插入性能，UUID等无序值会导致频繁页分裂，索引树变高，性能下降。

    主键字段的长度不要太大，因为主键字段长度越小，意味着二级索引的叶子节点越小（二级索引的叶子节点存放的数据是主键值），这样二级索引占用的空间也就越小

  • 索引不是越多越好：索引占用磁盘空间，写入时需要维护B+树，影响写性能。

  • 索引字段设为 NOT NULL：既能避免 NULL 导致优化器判断复杂、索引统计异常，又能省去 InnoDB 存储 NULL 标识的额外空间开销，提升索引利用率。

2.7 单表2000W行问题

  • 2000W是经验值，不是硬限制：B+树3-4层可支撑千万级数据，2000W行对应B+树3层左右，超过后树高增加，IO次数变多，性能下降。
  • 实际最大值和行大小有关：行越大，单页能存的记录越少，单表能存的数据量越小；本质是要控制B+树高度在3-4层，减少磁盘IO。
  • MySQL 为了提高性能，会将表的索引装载到内存中，在 InnoDB buffer size 足够的情况下，其能完成全加载进内存，查询不会有问题。但是，当单表数据库到达某个量级的上限时，导致内存无法存储其索引，使得之后的 SQL 查询会产生磁盘 IO，从而导致性能下降

2.8 索引创建场景

  • 需要创建索引：
    • 字段有唯一性约束（如商品编码）
    • 频繁用于WHERE 查询的字段（多字段可建联合索引）
    • 经常用于GROUP BY / ORDER BY的字段（避免额外排序）
  • 不需要创建索引：
    • 不用于查询、分组、排序的字段
    • 字段重复值极多（如性别）
    • 表数据量很小
    • 字段频繁更新（如用户余额）

2.9 执行计划参数

  • possible_keys 字段表示可能用到的索引；
  • key 字段表示实际用的索引，如果这一项为 NULL，说明没有使用索引；
  • key_len 表示索引的长度；
  • rows 表示扫描的数据行数。
  • type 表示数据扫描类型，执行效率从低到高：
    • All（全表扫描）：最坏的情况
    • index（全索引扫描）：跟ALL差不多
    • range（索引范围扫描）：一般在 where 子句中使用 < 、>、in、between 等关键词，只检索给定范围的行
    • ref（非唯一索引扫描）：返回数据返回可能是多条。因为虽然使用了索引，但该索引列的值并不唯一，有重复。
    • eq_ref（唯一索引扫描）：多用于主键或唯一索引的多表联查
    • const（结果只有一条的主键或唯一索引扫描）：主键或唯一索引与常量比较
  • extra额外信息，常见指标如下：
    • Using filesort ：当查询语句中包含 group by 操作，而且无法利用索引完成排序操作的时候， 这时不得不选择相应的排序算法进行，甚至可能会通过文件排序，效率是很低的，所以要避免这种问题的出现。
    • Using temporary：使了用临时表保存中间结果，MySQL 在对查询结果排序时使用临时表，常见于排序 order by 和分组查询 group by。效率低，要避免这种问题的出现。
    • Using index：所需数据只需在索引即可全部获得，不须要再到表中取数据，也就是使用了覆盖索引，避免了回表操作，效率不错。

三、事务篇

3.1 ACID特性

  • 原子性（Atomicity）：事务要么全成功要么全失败，回滚靠undo log实现。
  • 一致性（Consistency）：事务前后数据满足完整性约束，由原子性、隔离性、持久性共同保证。
  • 隔离性（Isolation）：并发事务互不干扰，靠MVCC或锁实现。
  • 持久性（Durability）：事务提交后数据永久生效，靠redo log实现。

3.2 并发事务问题

严重性排名：脏读 > 不可重复读 > 幻读

  • 脏读：读到其他事务未提交的数据，该事务可能回滚，导致读到无效数据。
  • 不可重复读：同事务内多次读同一行，数据被其他事务修改提交，导致前后两次结果不一致。
  • 幻读：同事务内多次按相同条件查询，记录数被其他事务插入/删除，导致前后结果集行数不一致。

3.3 事务隔离级别

隔离性水平排名：串行化 > 可重复读 > 读已提交 > 读未提交

要解决脏读现象，就要将隔离级别升级到读已提交以上的隔离级别，要解决不可重复读现象，就要将隔离级别升级到可重复读以上的隔离级别。

隔离级别 脏读 不可重复读 幻读 说明
读未提交 可能 可能 可能 能读到其他事务未提交的数据
读已提交 不可能 可能 可能 只能读到其他事务已提交的数据，每次查询生成新Read View
可重复读（InnoDB默认） 不可能 不可能 大概率避免，未完全解决 事务启动时生成Read View，全程复用，快照读靠MVCC避免幻读，当前读靠next-key锁避免幻读
串行化 不可能 不可能 不可能 读写都加锁，性能极差

读提交与可重复读均依靠MVCC实现，通过ReadView字段和记录隐藏列比对管控并发数据可见性，核心差异在于ReadView创建时机：
①读提交每次执行select都会新建视图，易出现同一事务多次读取数据不一致；
②可重复读仅在事务开启时生成一次视图，全程复用，保障重复读数据统一；
③同时可重复读中普通select是无锁快照读，而select ... for update为会加临键锁（next-key lock 锁）、读取最新数据的当前读。

  • 针对快照读（普通 select 语句），是通过 MVCC方式解决了幻读。
  • 针对当前读（select … for update 等语句），是通过 next-key lock（记录锁+间隙锁）方式解决了幻读。

3.4 MVCC多版本并发控制

  • 核心组件：Read View（快照，记录当前活跃事务列表、最小事务ID、全局事务 id 计数器的当前值）、undo log版本链（通过roll_pointer串联所有历史版本，每条版本记录trx_id标识修改它的事务）。

  • InnoDB 存储引擎的数据库表，它的聚簇索引记录中都包含下面两个隐藏列：

    • trx_id，当一个事务对某条聚簇索引记录进行改动时，就会把该事务的事务 id 记录在 trx_id 隐藏列里；
    • roll_pointer，每次对某条聚簇索引记录进行改动时，都会把旧版本的记录写入到 undo 日志中，然后这个隐藏列是个指针，指向每一个旧版本记录，于是就可以通过它找到修改前的记录。

  • 可见性判断规则：

    1. trx_id < Read View.min_trx_id：版本是事务启动前提交的，可见。

    2. trx_id >= Read View.max_trx_id：版本是事务启动后新启动的，不可见。

    3. min_trx_id <= trx_id < max_trx_id：在活跃事务列表中则不可见，否则可见。

      如果当前版本不可见，InnoDB 会通过 Undo Log 版本链向下（向历史版本）回溯，直到找到第一个对当前事务可见的版本。

  • 读已提交每次查询生成新Read View，可重复读事务启动时生成一次Read View。

3.5 幻读未完全解决的特殊场景

可重复读下仍有两种幻读场景：

  1. 事务A查不到id=5的记录，事务B插入id=5提交，事务A更新id=5的记录，再查询就能看到该记录，发生幻读（更新操作会让该记录的trx_id变成事务A的ID，后续快照读可见）。
  2. 事务A先快照读，事务B插入新记录提交，事务A再执行当前读（如select ... for update），会读到新记录，发生幻读。

所以，MySQL 可重复读隔离级别并没有彻底解决幻读，只是很大程度上避免了幻读现象的发生。要避免这类特殊场景下发生幻读的现象的话，就是尽量在开启事务之后，马上执行 select … for update 这类当前读的语句，因为它会对记录加 next-key lock，从而避免其他事务插入一条新记录。

3.6 事务其他注意点

  • autocommit参数默认开启，单条增删改语句会自动提交事务；大事务会导致锁占用久、undo log量大、主从延迟，建议拆小。

四、锁篇

可通过**select * from performance_schema.data_locks\G;**这条语句，查看事务执行 SQL 过程中加了什么锁。

4.1 锁的分类（按粒度）

表锁和行锁是满足读读共享、读写互斥、写写互斥的。

4.1.1 全局锁

  • 命令：flush tables with read lock，整库进入只读状态，增删改、表结构变更都会被阻塞，用于全库逻辑备份；MySQL 8.0后备份可使用mysqldump --single-transaction（可重复读隔离级别），利用MVCC实现热备份，无需全局锁。

4.1.2 表级锁

  • 表锁：lock tables 表名 read/write，read锁允许其他线程读不允许写，write锁完全阻塞其他线程的所有操作，本线程也只能操作锁定的表，解锁用unlock tables，连接断开自动释放。

  • 元数据锁（MDL）：自动加锁，CRUD操作加MDL读锁，表结构变更加MDL写锁，读读不互斥，读写、写写互斥，事务提交才释放；长事务持有MDL读锁会导致后续DDL（DDL：alter/drop/ 加字段）拿不到写锁而阻塞，只要 DDL 在排队，这张表后续所有CRUD。

  • 意向锁：加行锁前先加表级意向锁（IS意向共享、IX意向排他），方便快速判断表是否有行锁，无需遍历所有行；意向锁和表锁互斥，和行锁不互斥。

  • 作用于AUTO_INCREMENT 修饰的字段的锁有两种。①AUTO-INC锁：自增主键专用；②轻量级锁：赋值一个自增的值立刻释放，不需要等待整个插入语句执行完后才释放锁。

    innodb_autoinc_lock_mode控制：

    • 0用AUTO-INC锁（语句结束释放）；
    • 1普通insert用轻量级锁（申请完自增值就释放），批量insert用AUTO-INC锁；
    • 2全用轻量级锁，需搭配binlog_format = row格式避免主从不一致。

4.1.3 行级锁（仅InnoDB支持）

  • 如果 LOCK_MODE 为 X，说明是 next-key 锁；
  • 如果 LOCK_MODE 为 X, REC_NOT_GAP，说明是记录锁；
  • 如果 LOCK_MODE 为 X, GAP，说明是间隙锁；

通常来说，如果 LOCK_MODE 是 next-key 锁或者间隙锁，那么 LOCK_DATA 就表示锁的范围最右值，而锁范围的最左值为 LOCK_DATA 的上一条记录的值。

  • begin;
    //对读取的记录加共享锁
    select … lock in share mode;
    commit; //锁释放
  • begin;
    //对读取的记录加排他锁
    select … for update;
    commit; //锁释放

  • Record Lock（记录锁）：锁单条记录，分S锁（共享，读读兼容）、X锁（排他，写写、读写互斥）。

  • Gap Lock（间隙锁）：锁两个记录之间的间隙，防止其他事务插入数据，只有X型，间隙锁之间完全兼容（目的是防区间插入，无S/X区别，因此可以共存）。

  • Next-Key Lock（临键锁）：Record Lock + Gap Lock，锁记录+前面的间隙，前开后闭区间，是可重复读隔离级别的默认加锁单位，部分场景会退化。

    在 MySQL 的可重复读隔离级别下，针对当前读的语句会对索引加记录锁+间隙锁，这样可以避免其他事务执行增、删、改时导致幻读的问题。

  • 插入意向锁：插入记录时如果目标间隙（插入不存在的记录、无唯一键冲突时）有 间隙锁/next-key锁 ，会生成插入意向锁并进入等待，插入意向锁和间隙锁互斥。不是意向锁，它是一种特殊的间隙锁，属于行级别锁。如果说间隙锁锁住的是一个区间，那么「插入意向锁」锁住的就是一个点。

    每插入一条新记录，都需要看一下待插入记录的下一条记录上是否已经被加了间隙锁，如果已加间隙锁，此时会生成一个插入意向锁，然后锁的状态设置为等待状态（PS：MySQL 加锁时，是先生成锁结构，然后设置锁的状态，如果锁状态是等待状态，并不是意味着事务成功获取到了锁，只有当锁状态为正常状态时，才代表事务成功获取到了锁），现象就是 Insert 语句会被阻塞。

4.2 加锁规则（可重复读隔离级别）

InnoDB 表里同时存在二级索引 + 聚簇主键索引，二级索引查询最终会关联主键，双索引都会上锁；但是对主键索引加锁的时候，只有满足查询条件的记录才会对它们的主键索引加锁。

加锁基本单位是next-key lock，扫描到的索引记录才加锁：

对于唯一索引「等值部分」能退化就退化，「范围部分」老老实实加 Next-Key Lock；对于非唯一索引，由于可能有重复值，得防幻读，全加 Next-Key Lock。

  1. 唯一索引等值查询：

    • 记录存在：next-key锁退化为Record Lock，仅锁该记录（唯一索引能确定，只锁单行，不用锁间隙）。
    • 记录不存在：next-key锁退化为Gap Lock，锁第一条大于该记录的间隙（唯一索引不确定，但为了防止插入导致幻读，得锁间隙）。

  2. 唯一索引范围查询：

    • 等值部分记录存在：等值部分退化为Record Lock，范围部分加next-key锁；
    • 等值部分记录不存在：等值部分退化为Gap Lock，范围部分加next-key锁；
    • 扫描到末尾的supremum伪记录：加(最后一条记录, +∞]的next-key锁。

  3. 非唯一索引等值查询：

    • 记录存在：该记录加next-key锁，对应主键加Record Lock，第一条不满足条件的记录加Gap Lock。
    • 记录不存在：第一条不满足条件的记录加Gap Lock。

  4. 非唯一索引范围查询：所有扫描到的记录都加next-key锁，不会退化。

  5. 无索引的 查询/update/delete：全表扫描，所有记录加next-key锁，相当于锁全表，非常危险；可通过sql_safe_updates=1限制无索引的更新/删除，或用force index强制走索引。

    如果发现即使在 where 条件中带上了索引列，优化器走的还是全表扫描，这时我们就要使用 force index([index_name]) 可以告诉优化器使用哪个索引。

4.3 死锁

  • 成因：两个事务循环等待对方持有的锁，例如两个事务都持有间隙锁，又分别向对方的间隙插入记录，插入意向锁互相等待，满足死锁条件（互斥、占有且等待、不可剥夺、循环等待）。
  • 避免：拆小事务，按相同顺序访问表和行，降低隔离级别，设置innodb_lock_wait_timeout锁等待超时，开启innodb_deadlock_detect死锁检测。

4.4 锁与幻读的关系

  • 快照读（普通select）：靠MVCC避免幻读，事务全程用同一个Read View，看不到其他事务新插入的记录。
  • 当前读（select ... for update、update、delete）：靠next-key锁避免幻读，锁住扫描到的记录和间隙，其他事务无法在该范围内插入/删除，防止幻读。
  • 记录锁+间隙锁（next-key锁）可以防止删除导致的幻读：对扫描到的记录加next-key锁后，其他事务无法删除该范围内的记录，也不会有新记录插入，因此不会出现幻读。

4.5 INSERT 加锁

INSERT 的执行顺序（唯一索引专属）

不是直接插入加 X 锁，而是分两步：

  • 第一步：校验唯一性（查这条记录是否已存在 → 读操作）
  • 第二步：校验通过 → 真正插入 → 加 X 锁 / 插入意向锁

插入阶段：执行 INSERT 时，先检查唯一二级索引的唯一性。若无冲突，则在聚簇索引和唯一索引上插入新记录，不立即加显式锁，仅通过事务 ID 形成隐式锁保护记录。

冲突检测阶段：若另一事务插入相同键值，会先对已有记录尝试加 S 型 Next-Key 锁 以验证唯一性。

锁转换与阻塞阶段：检测到隐式锁存在时，将其升级为显式 X 型记录锁，导致后续事务的 S 锁请求阻塞，直到原事务提交释放锁。

如果非唯一索引，那么两个INSERT事务，前后执行相同的 Insert 语句，是不会发生阻塞的。

如果无索引，则会全表扫描级别的锁。

五、日志篇

更新语句的流程基于查询语句之外还会涉及到 undo log（回滚日志）、redo log（重做日志） 、binlog （归档日志）这三种日志，具体流程如下：

  • 经过连接器、解析器、优化器生成执行计划，执行器通过索引找到目标数据页；
  • 数据页不在 Buffer Pool 则从磁盘加载至内存，再修改该行数据；
  • InnoDB 先写入undo log（记录修改前数据，内存缓存），用于事务回滚与 MVCC；
  • 修改内存中的数据页并标记为脏页，同时将数据页物理变更写入redo log buffer；
  • 变更逻辑日志写入服务层 binlog cache；
  • 事务开启两阶段提交：先将 redo log 持久化落盘并设为 prepare 状态，再把 binlog 落盘，最后标记 redo log 为 commit，事务提交完成；
  • 脏页、undo 页后续由后台线程异步刷盘，无需实时落盘。

redo log 记录了此次事务「修改后」的数据状态，记录的是更新之后的值，主要用于事务崩溃恢复，保证事务的持久性。
undo log 记录了此次事务「修改前」的数据状态，记录的是更新之前的值，主要用于事务回滚，保证事务的原子性。

5.1 undo log（InnoDB引擎层）

  • 作用：保证事务原子性（回滚用），实现MVCC（和Read View配合做快照读）。

    快照读：undo log 为每条记录保存多份历史数据，MySQL 在执行快照读（普通 select 语句）的时候，会根据事务的 Read View 里的信息，顺着 undo log 的版本链找到满足其可见性的记录。

  • 内容：记录事务修改前的数据，插入操作记主键，删除操作记整行数据，更新操作记被修改列的旧值；每次修改生成一个undo log，通过roll_pointer串成版本链，trx_id标识修改它的事务。

  • 刷盘：undo页存在Buffer Pool中，修改undo页也会记录redo log，靠redo log保证持久化。

    InnoDB 的 Buffer Pool 以16KB数据页为单位缓存数据，核心缓存数据页、索引页，同时还会缓存undo页、插入缓存、自适应哈希索引、锁信息等各类引擎运行所需页与资源，减少磁盘IO、提升读写性能。

    当修改数据时，如果数据存在于 Buffer Pool 中，那直接修改 Buffer Pool 中数据所在的页，然后将其页设置为脏页（该页的内存数据和磁盘上的数据已经不一致），为了减少磁盘I/O，不会立即将脏页写入磁盘，后续由后台线程选择一个合适的时机将脏页写入到磁盘。

5.2 redo log（InnoDB引擎层）

  • 作用：保证事务持久性（crash-safe），WAL（Write-Ahead Logging）技术：先写日志再写磁盘，把随机写转为顺序写，提升写性能。

  • 内容：redo log buffer（内存）->redo log文件（磁盘），物理日志，记录数据页的修改（哪个表空间、哪个页、哪个偏移量修改成什么值）；循环写，有write pos（当前写入位置）和checkpoint（当前擦除位置），写满后会阻塞，需要刷脏页腾出空间。

  • 刷盘时机：

    • MySQL 正常关闭时；

    • 当 redo log buffer 中记录的写入量大于 redo log buffer 内存空间的一半时，会触发落盘；

    • InnoDB 的后台线程每隔 1 秒，将 redo log buffer 持久化到磁盘。（基础、全局性的保底机制）

    • 每次事务提交时都将缓存在 redo log buffer 里的 redo log 直接持久化到磁盘

  • 刷盘控制：innodb_flush_log_at_trx_commit参数：

    • 0：事务提交时（不做write、fsync ）写redo log buffer，后台每秒刷盘，宕机可能丢1秒数据。

    • 1（默认）：事务提交时（write+fsync）立即刷盘，最安全，性能最差。

    • 2：事务提交时（只做 write）写操作系统page cache（操作系统的文件缓存），后台每秒刷盘，OS崩溃可能丢1秒数据，MySQL宕机不丢数据。

      • 数据安全性：参数 1 > 参数 2 > 参数 0
      • 写入性能：参数 0 > 参数 2> 参数 1

  • 默认情况下， InnoDB 存储引擎有 1 个重做日志文件组( redo log Group），「重做日志文件组」由有 2 个 redo log 文件组成，这两个 redo 日志的文件名叫 ：ib_logfile0 和 ib_logfile1 。重做日志文件组是以循环写的方式工作的，从头开始写，写到末尾就又回到开头，相当于一个环形。redo log 文件满了，这时 MySQL 不能再执行新的更新操

  • 为什么不直接将 buffer pool 的数据写入到磁盘，而是选择更新redo log file文件：redo log file 是顺序写入的，buffer pool 的内存数据是随机分散在磁盘各处的，顺序写磁盘性能是随机写的几十倍，所以很多存储系统在写数据时都会搞个日志来记录操作，方便服务重启后进行数据对账，确保数据的一致性和完整性，这类操作就是所谓的 Write-Ahead Logging (WAL) 。

5.3 binlog（Server层，所有引擎通用）

  • 作用：数据备份、主从复制，记录所有数据变更（增删改），不记录查询语句。

  • 格式：

    • STATEMENT：记录原始SQL，主从复制可能因动态函数（如now()）导致数据不一致。
    • ROW（MySQL 5.7.7+默认）：记录行变更内容，主从一致，但大批量更新会产生大量日志。
    • MIXED：混合模式，自动选择STATEMENT或ROW。

  • 刷盘时机：

    • 事务执行时binlog暂存于线程独立的binlog cache；
    • 事务提交时统一一次性write写入binlog文件（多线程写入同一个）（落入操作系统页缓存）；
    • 真正物理落盘由sync_binlog控制：设为1则每次提交强制fsync落盘，设为N则累积N个事务批量fsync，设为0完全交由系统自动刷盘。

  • 刷盘控制：sync_binlog参数：

    • 0（默认）：仅write不fsync，由操作系统决定刷盘时机。
    • 1：每次事务提交都fsync，最安全。
    • N：每N个事务fsync一次，可能丢N个事务的数据。

5.4 redo log 和 bin log 的区别

  • 所属层级：redo log 是 InnoDB 引擎专属日志；binlog 是 MySQL Server 层通用日志，所有引擎均可使用。
  • 日志类型：redo log 为物理日志，记录数据页的物理修改；binlog 为逻辑日志，记录 SQL 语句或行数据变更。
  • 写入方式：redo log 是循环覆盖写，空间固定、循环复用；binlog 是追加写入，永久留存、不会覆盖。
  • 核心作用：redo log 依托 WAL 实现崩溃恢复，保障事务持久性、降低随机 IO；binlog 用于主从复制与全量数据备份恢复。
  • 生成时机：redo log 事务执行更新时实时生成；binlog 在事务提交阶段统一写入。

5.5 主从复制

MySQL 主从复制基于 binlog 实现：主库记录所有数据与表结构变更的binlog；从库通过IO线程拉取主库binlog并写入中继日志，再由SQL线程回放执行日志内容，同步数据变更；支持异步、半同步、同步三种模式，核心用于读写分离、数据备份、故障容灾、集群扩容。

  • 异步复制（默认）：主库写入 binlog 后直接响应客户端，无需等待从库同步，性能高；主库宕机易丢失未同步数据。
  • 半同步复制：主库需等待至少一台从库接收日志并返回响应后，再提交返回，平衡性能与数据安全性，防止单点数据丢失。
  • 同步复制：主库等待所有从库完整回放同步完成后才响应，数据强一致，但阻塞严重、性能极低，线上基本不用。

5.6 redo log 和 binlog持久化–两阶段提交

  • 目的：保证redo log和binlog逻辑一致，避免主从数据不一致。
  • 流程：
    1. Prepare阶段：redo log写入磁盘，状态改为prepare。
    2. 写binlog到磁盘。
    3. Commit阶段：redo log状态改为commit。
  • 崩溃恢复：prepare状态的redo log，若对应binlog完整（binlog 中查找到与 redo log 相同的 XID）则提交事务，否则回滚。

事务没提交的时候，redo log 也是可能被持久化到磁盘的。MySQL崩溃时，没提交的事务，因为bin log为持久化，redo log和bin log中的XID不一致，会进行回滚。因此，bin log 必须在事务提交之后，才可以持久化到磁盘。

  • 问题：
    • IO 开销高：双 1 配置下，每次事务提交需两次 fsync 刷盘，磁盘 IO 频繁，拖慢性能；
    • 锁竞争严重：早期依赖全局锁保证日志提交顺序，并发场景下争抢锁资源，并发能力受限；
    • 依赖组提交机制才能缓解刷盘与锁带来的性能问题。

5.7 组提交

  • 目的：合并多个事务的redo log/binlog刷盘操作，减少磁盘IO次数，提升性能。
  • 实现：组提交将事务提交拆分为 flush、sync、commit 三阶段，通过队列 + 组长机制批量处理事务；binlog 组提交合并多事务单次 fsync 落盘，减少 binlog 刷盘 IO；redo 组提交（5.7 新增）合并 redo prepare 阶段批量刷盘；同时缩小锁粒度、摒弃全局锁，在保证 redo 与 binlog 日志一致的前提下，大幅降低磁盘 IO、提升并发性能。
  • 相关参数：binlog_group_commit_sync_delay（延迟多久刷盘，合并更多事务）、binlog_group_commit_sync_no_delay_count（事务数达到该值就立刻刷盘）。

5.8 其他日志

  • relay log：从库的中继日志，存储主库同步过来的binlog，用于回放。
  • 慢查询日志：记录执行时间超过阈值的SQL，用于性能分析。

六、内存篇

6.1 Buffer Pool（缓冲池）

  • 作用：缓存磁盘中的数据页、索引页，减少磁盘IO，是InnoDB提升性能的核心组件。
  • 配置：默认128MB，建议设置为物理内存的60%-80%，通过innodb_buffer_pool_size参数调整；以16KB页为单位划分，启动时申请连续虚拟内存，访问后才分配物理内存。
  • 缓存内容：数据页、索引页、undo页、插入缓存、自适应哈希索引、锁信息等。
  • 控制快、缓存页：为了更好的管理这些在 Buffer Pool 中的缓存页，InnoDB 为每一个缓存页都创建了一个控制块，控制块信息包括「缓存页的表空间、页号、缓存页地址、链表节点」等等。控制块也是占有内存空间的，它是放在 Buffer Pool 的最前面，接着才是缓存页。控制块和缓存页之间可能有碎片空间，取决于在尽可能分配完控制快和缓存页后还有没有剩不可用空间（大小不够分）。

6.2 Buffer Pool管理链表

  • Free链表（空闲页链表）：管理空闲缓存页，需要加载新页时从链表中取空闲页。Free链表节点有指针指向控制块，控制块有指针指向缓存页。

  • LRU链表（管理脏页+干净页）：管理已使用的页，优化为young区域（前半部分，热点数据）+ old区域（后半部分，默认占37%，由innodb_old_blocks_pct控制）：

    • 预读的页先放入old区域，被访问后才进入young区域，解决预读失效问题。

  • old区域的页需要停留超过innodb_old_blocks_time（默认1000ms）才进入young区域，避免全表扫描的大量页冲掉热点数据（Buffer Pool污染问题）。

    • young区域前1/4被访问不会移到头部，减少热点数据频繁移动的开销。

  • Flush链表（脏页链表）：管理脏页（被修改过、内存和磁盘数据不一致的页），后台线程定期刷脏页到磁盘。

6.3 脏页刷盘时机

  1. redo log满了，触发checkpoint刷脏页腾空间。
  2. Buffer Pool空间不足，淘汰页是脏页时，先刷盘再淘汰。
  3. 后台线程定期刷盘。
  4. MySQL正常关闭时刷所有脏页。

6.4 其他内存组件

  • 自适应哈希索引：对热点数据页，用哈希表存储（查询值→数据页地址），查询复杂度O(1)，提升等值查询性能。
  • Change Buffer：缓冲非唯一二级索引的更新操作，等对应索引页被读到Buffer Pool时再合并，减少磁盘IO；仅适用于非唯一二级索引（唯一索引需要立刻判断唯一性，不能延迟合并）。
  • redo log buffer：存储事务产生的redo log，默认16MB，由innodb_log_buffer_size控制，事务提交时刷盘。

七、性能/调优篇

7.1 查询性能优化

  • 执行计划分析：用explain查看SQL执行计划，核心指标：

    • type：访问类型，从差到好为all（全表扫描）<index（全索引扫描）<range（索引范围扫描）<ref（非唯一索引扫描）<eq_ref（唯一索引扫描）<const（主键/唯一索引等值查询）。
    • key：实际使用的索引，NULL表示未走索引。
    • rows：预估扫描的行数。
    • extra：Using filesort（文件排序，需优化）、Using temporary（临时表，需优化）、Using index（覆盖索引，性能好）。

  • 索引优化：为where/group by/order by字段建索引，避免索引失效，尽量用覆盖索引，联合索引遵循最左匹配，主键自增。

  • 分页优化：

    • 深度分页问题：limit offset, size的offset越大，需要扫描并丢弃的数据越多，性能越差。

      MySQL 会查询 offset+size 条数据并丢弃前 offset 条；主键索引分页会冗余读取完整行数据，非主键索引分页易产生大量回表操作，offset 过高还会触发全表扫描。

    • 优化方案：如果是全量导出，用主键分批查询（where id > 上次最大id limit size）；如果是前端分页，限制分页深度，采用瀑布流（上一页/下一页），避免跳转深页；可用子查询先查主键再关联（select * from t where id >= (select id from t order by id limit 6000000,1) order by id limit 10）。

7.2 计数优化

  • count(*)和count(1)性能一致：InnoDB会优先选择最小的二级索引扫描，没有二级索引则扫主键索引，MySQL官方说明二者没有性能差异。
  • count(字段)性能最差：会全表扫描（即使字段有索引，也要判断NULL值），建议给该字段建二级索引。
  • 大表计数：精确计数可单独建计数表维护；近似计数可用show table status或explain的rows估算。

7.3 写性能优化

  • 日志参数调优：非核心业务可将innodb_flush_log_at_trx_commit设为2、sync_binlog设为大于1，减少fsync次数；用binlog组提交参数合并刷盘。

  • 避免大事务：拆小事务，减少锁占用时间、undo log量、主从延迟。

  • Change Buffer：非唯一二级索引的更新先存Change Buffer，减少磁盘IO。

    二级索引：

    • 通常非唯一，插入顺序相对随机
    • 索引页可能不在 Buffer Pool 中，需要从磁盘读取

    延迟更新策略：不立即加载磁盘中的二级索引页，而是将变更操作暂存到 Change Buffer 中，等该索引页被其他操作读入 Buffer Pool 时，再一次性合并所有变更，从而将多次随机 I/O 转化为一次批量操作，大幅提升写性能。

7.4 架构优化

  • 主从复制：读写分离（写主库，读从库），提升读性能，减轻主库压力。
  • 分库分表：垂直分库（按业务拆分数据库）、垂直分表（按字段热度拆分宽表）、水平分库/分表（按时间/范围拆分大表），解决单表数据量过大问题。

7.5 其他优化

  • 开启sql_safe_updates=1，防止无索引的update/delete锁全表。
  • 用force index强制SQL走指定索引，避免优化器选错索引。
  • 避免select *，只查需要的字段，减少数据传输和回表开销。

八、架构篇

8.1 整体架构

MySQL采用Server层+存储引擎层的插件式架构：

  • Server层（所有引擎共用）：

    • 连接器：管理连接、权限验证。
    • 解析器：词法、语法分析。
    • 预处理器：校验表/字段、扩展select *。
    • 优化器：生成执行计划、选择索引。
    • 执行器：调用存储引擎接口执行SQL。
    • 公共功能：内置函数、存储过程、触发器、视图、binlog（Server层日志）。

  • 存储引擎层：

    • InnoDB（默认）：支持事务、行锁、外键、crash-safe，核心组件有Buffer Pool、redo log buffer、自适应哈希索引、Change Buffer。
    • MyISAM：不支持事务、表锁、外键，仅支持表级锁。
    • Memory：内存存储，支持哈希索引，重启数据丢失。

server 层和存储引擎层共同构成了一个完整的数据库，它就是我们常说的 Mysql 数据库。并且，server 层和存储引擎层是通过接口函数进行解耦的，换句话说就是，只要实现了接口函数，就能作为存储引擎与 Server 层对接。

8.2 InnoDB物理存储架构

  • 内存部分：Buffer Pool、redo log buffer、自适应哈希索引（自动为热点高频访问的数据页构建哈希索引）、Change Buffer。
  • 磁盘部分：
    • .ibd：表数据、索引文件。
    • ib_logfile0/1：redo log文件，循环写。
    • undo log文件：存储undo log，用于回滚和MVCC。
    • binlog：Server层日志，存储在磁盘，追加写，用于备份和主从复制。

两部分共同构成了innodb存储引擎，并对外提供一系列函数接口。比如操作数据行的 write_row(), update_row()，以及操作数据表的 create(), drop()等等接口。平时写的 SQL 语句经过Server层，最终都会转换成 InnoDB 提供的这些接口函数调用。

8.3 主从复制架构

  • 作用：读写分离提升性能、高可用基础、数据备份。
  • 流程（异步复制，默认）：
    1. 主库写binlog，提交事务。
    2. 从库IO线程连接主库，拉取binlog写入本地relay log。
    3. 从库SQL线程回放relay log，更新本地数据。
  • 复制模式：
    • 异步复制：主库提交事务无需等待从库确认，性能高，主库宕机可能丢数据。
    • 半同步复制（MySQL 5.7+）：主库等待至少一个从库收到binlog才返回，兼顾性能和安全。
    • 同步复制：主库等待所有从库收到binlog才返回，性能极差，几乎不用。
  • 主从延迟优化：强制大事务走主库，避免从库延迟过大。

8.4 核心流程

  • 查询流程：连接器→（查询缓存）→解析器→预处理器→优化器→执行器→存储引擎（查Buffer Pool，未命中则加载磁盘页）→返回结果。
  • 更新流程：连接器→解析器→预处理器→优化器→执行器→存储引擎（写undo log→更新Buffer Pool脏页→写redo log prepare→写binlog→redo log commit），两阶段提交保证redo log和binlog一致。

九、思考篇

一、普通的update语句在innodb中会自动上事务然后上锁？

InnoDB 默认自动提交事务（autocommit=1），单条普通 update 会隐式开启独立事务，语句执行完立刻自动提交。

更新过程中必然自动加排他锁：命中索引加行锁；无索引 / 索引失效，会降级锁住整张表。

锁和事务生命周期一致，事务自动提交后，锁即刻释放。
