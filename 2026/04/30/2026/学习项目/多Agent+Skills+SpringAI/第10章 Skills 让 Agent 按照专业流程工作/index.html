第10章 Agent进阶：从助手到自主协作的进化

一、AI Agent的5个进化阶段

从最初的语言大模型到能够自主协作的复杂系统，Agent的发展共经历5个关键阶段，逐步解决决策能力、复杂任务处理、幻觉抑制等核心问题：

阶段 核心特征 解决的问题 未解决的缺陷
1. 语言大模型阶段 仅接收提示词，返回文本/多模态内容，无自主决策能力 提供知识查询、内容生成的基础能力 不会自主拆分任务，无工具调用能力，存在幻觉问题
2. 基础Agent阶段 引入工具调用能力，可对接本地软件、第三方API解决实际问题 突破纯文本交互限制，完成具体落地任务 各厂商工具调用标准不统一，无自主决策能力，复杂多步骤任务无法处理，幻觉问题未解决
3. 框架+工作流阶段 引入ReAct思维框架（思考→执行→验证→动态调整），通过人工编排Workflow实现多Agent节点协同 赋予Agent自主决策能力，支持复杂多步骤任务拆解 工作流需人工封装节点，灵活性不足，幻觉问题仅部分缓解
4. 自主协作系统阶段 无需人工介入，Agent自主理解意图、拆分目标、分配子任务，多Agent自动协同（代表框架：LangGraph） 实现全流程自动化，减少人工编排成本 专业领域任务处理效率不足，固定流程场景冗余度高
5. 专业领域+技能体系阶段 细分专业领域Agent，搭配SubAgent处理单场景任务，通过Agent Skills固化固定流程，形成完整工程化体系 大幅提升专业场景执行效率，通过多层自我反思最大限度缓解幻觉问题 系统复杂度提升，需配套技能管理、加载机制

二、核心概念解析

2.1 MCP：AI调用外部工具的统一标准

定义与价值

MCP（Model Context Protocol）是Cloud Code提出的外部工具连接标准，统一了不同大模型调用第三方工具的规范：只要工具服务端遵循MCP标准开发，DeepSeek、千问、OpenAI等各类大模型都可以直接调用该工具，无需重复开发适配逻辑，避免了此前各厂商工具调用标准不兼容的问题。

局限性

  • MCP仅提供工具本身，不规定工具的使用方法、适用场景
  • 传统MCP会将服务端所有工具一并返回给Agent，占用大量上下文窗口
  • Agent不知道如何选择正确工具，需要反复请求大模型确认，消耗额外Token

2.2 SubAgent：专注单场景的「牛马小弟」

定义与设计思路

SubAgent（子代理）是Cloud Code三层多Agent架构中最底层的执行单元，专门负责单个专业场景的任务，设计思路完全仿照IT部门分工：

  • 总协调Agent：仅负责制定计划、划分需求，将不同领域任务分配给对应主管
  • 领域Agent主管：将领域内任务拆分，分配给具备对应专业知识的SubAgent执行
  • SubAgent：仅处理单一专业场景（如SQL编写、CRUD操作、数据清洗、登录接口开发等）

核心特性

  1. 独立上下文执行：每个SubAgent在独立容器、独立上下文中运行，不会污染主线任务的上下文
  2. 错误隔离：单个SubAgent执行失败不会影响主线任务和其他SubAgent的运行
  3. 并行处理：多个SubAgent可同时处理不同专业场景，大幅提升执行效率
  4. 执行即释放：任务执行完成后SubAgent会被释放，不支持跨任务复用

2.3 Agent Skills：可复用的专业领域技能系统

Agent Skills是Cloud Code提出的智能体技能标准，不是单一工具调用能力，而是包含「专业知识+固定工作流程+工具调用逻辑」的完整SOP体系，是区别于MCP、SubAgent的核心能力。

与MCP、Workflow、SubAgent的差异

对比维度 MCP Agent Skills Workflow SubAgent
核心定位 外部工具连接标准 专业领域完整技能体系 人工封装的流程节点 单场景执行单元
包含内容 仅工具 专业知识+固定流程+工具逻辑 流程节点+业务逻辑（含分支/并行/总结） 仅执行能力
流程灵活性 无流程 固定SOP，无分支 支持自定义分支、并行逻辑 按分配的技能执行
加载方式 全量返回工具 渐进式加载（见下文） 全量加载流程节点 任务触发时创建
可复用性 工具可被多个Agent调用 以文件夹形式移植，多Agent复用 绑定特定工作流，复用性低 执行完即释放，不可复用
调用决策 需大模型选择工具 Agent自主识别场景、选择工具 人工触发或按节点执行 由上层Agent分配任务

渐进式加载机制（解决上下文占用问题）

Agent Skills包含大量知识、流程模板，若全量加载会占满上下文窗口，因此采用三步渐进式加载：

  1. 元数据索引加载：仅加载SKILL.md文件的索引信息（技能名称、描述、步骤索引），文件极小，仅占用少量上下文，让Agent知道技能的存在和适用场景
  2. 工具按需加载：Agent根据索引中的工具清单，仅要求MCP返回当前场景需要的工具，无需加载无关工具
  3. 附加资源使用时加载：脚本、参考文档、示例等资源仅在执行到对应步骤时才加载，避免提前占用上下文

SKILL.md文件结构（分三部分）：

  1. 基础信息字段（YAML Frontmatter）

    字段 必要性 说明
    name 可选（默认用文件夹名） 技能唯一标识，仅支持小写字母、数字、连字符，最长64字符
    description 强烈推荐 技能功能+适用场景，Claude以此判断自动触发时机，最长1024字符，建议细节丰富提升匹配准确率
    disable-model-invocation 可选 设为true则仅允许用户手动调用，Claude不会自动触发
    user-invocable 可选 设为false则仅Claude可调用，隐藏在用户/菜单中
    allowed-tools 可选 技能激活时无需审批即可使用的工具列表
    context 可选 设为fork则在独立SubAgent上下文中运行，隔离主对话历史
    paths 可选 Glob模式，仅处理匹配文件时自动加载技能

  2. 正文：执行步骤
    明确技能的目标、固定执行步骤、输出格式，例如产品经理画原型图技能的步骤：

    1. 跟需求方沟通确认需求，输出需求文档
    2. 调研竞品信息，整理参考案例
    3. 使用墨刀/即时设计工具绘制原型图
    4. 与需求方确认原型，调整细节
    5. 输出带尺寸标注、功能说明的最终原型图，同步给前后端开发

  3. 指令结构
    约定执行的前置条件、参数规范、错误处理、返回格式模板，例如百度地图路线查询技能的指令：

    • 前置条件：已配置百度地图API Key
    • 入参规范：支持城市名、经纬度两种参数格式
    • 优先规则：优先选择距离最短的路线
    • 错误处理：API调用失败则返回「无法查询到路线，请检查地址是否正确」
    • 返回格式：按JSON模板组装路线距离、耗时、步骤信息

推荐的Skill内容结构（6部分）

根据官方最佳实践，SKILL.md正文建议包含以下6个部分，条理更清晰：

  1. 概述：简要介绍Skill的核心功能、适用场景
  2. 前置条件：使用Skill前需要满足的条件（如API Key配置、依赖环境等）
  3. 执行步骤：详细说明每个步骤的输入、输出、操作细节
  4. 示例：提供具体使用场景和操作示例
  5. 错误处理：列出常见错误及对应解决方法
  6. 局限说明：明确Skill无法实现的功能边界

Claude Code中的Skills特性

结合Claude Code官方文档，Skills还支持以下扩展能力：

  • 多级别存储：企业级（组织全部用户）、个人级（~/.claude/skills/，全部项目可用）、项目级（.claude/skills/，仅当前项目可用）、插件级（插件目录/skills/，随插件启用）
  • 实时变更检测：Skills目录文件修改后无需重启会话，实时生效
  • 嵌套自动发现：Monorepo场景下会自动发现子包内的.claude/skills/目录
  • 动态上下文注入：支持通过command语法在执行前运行Shell命令，将结果注入技能上下文
  • 字符串替换：支持$ARGUMENTS（全部参数）、$0/$1（位置参数）、${CLAUDE_SESSION_ID}（会话ID）等动态变量

四、实践建议

  1. 场景选型指南
    • 仅需调用外部工具、无固定流程：选择MCP
    • 自定义灵活流程、有分支逻辑：选择Workflow
    • 固定专业场景、可复用标准流程：选择Agent Skills
    • 单专业场景、执行完即释放：选择SubAgent
  2. Skills编写技巧
    • description字段尽量丰富细节，包含触发关键词，提升自动匹配准确率
    • SKILL.md控制在500行以内，详细参考文档、示例放到支持文件中，通过链接引用
    • 需手动触发的技能（如部署、提交代码）设置disable-model-invocation: true，避免Agent自动执行
  3. 性能优化建议
    • 生产环境使用持久化CheckPointer存储对话记忆，而非内存存储
    • 通过paths字段限制技能生效的文件范围，减少不必要的上下文加载
    • 复杂技能可设置context: fork，在独立SubAgent中运行，避免污染主上下文

五、章节总结

本章完整梳理了Agent从单一工具调用到复杂自主协作系统的进化路径，核心结论如下：

  • MCP解决了工具调用标准不统一的问题，是Agent连接外部能力的桥梁
  • SubAgent通过单场景隔离、并行执行提升了复杂任务的执行效率和稳定性
  • Agent Skills通过可复用的专业SOP体系，固化了固定流程场景的最佳实践，配合渐进式加载机制解决了上下文占用问题
  • Spring AI Alibaba 1.1.2已提供完整的Skills集成能力，开发者可快速基于Java生态构建具备专业领域能力的Agent系统

最终形成的「专业领域Agent + SubAgent + Agent Skills」协作体系，相比传统单Agent、人工编排Workflow，能够通过多层自我反思有效缓解大模型幻觉，大幅提升复杂任务的解决能力。
