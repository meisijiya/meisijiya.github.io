第13章 旅游规划多Agent系统工程化实践

📌 章节总览

第13章以旅游规划实战项目为载体，系统讲解智能体应用从功能开发到工程化落地的全流程，核心覆盖两大知识点：

  1. 多Agent协同：包含工作流编排、AgentScope企业级框架两种实现路径

  2. 智能体自主能力：自主思考、复杂任务拆分、思考-行动-反思闭环

技术栈覆盖：AgentScope Java、Langfuse可观测平台、Docker容器化、百度地图MCP服务、阿里千问/DeepSeek等大模型。

注：本章文稿为课程语音转写，存在部分名称转录误差（如nfuse→Langfuse、deep see kr 1→DeepSeek R1），已结合官方文档标注修正。

13-1 敏感资源处理方案

核心概要

解决原项目敏感信息（阿里千问API Key、百度地图MCP地址/API Key等）硬编码的安全风险，实现敏感信息与程序代码完全分离。

详细实现

  1. 敏感信息集中存储在项目根目录创建.env文件，统一存放所有敏感配置，示例：

  2. 模块配置引用
    各功能模块（主管Agent、行程规划、路线制定等）的application.yml中通过占位符引用.env变量，且必须配置.env文件导入规则，否则无法解析占位符。

  3. 配置读取与代码调整

    • 使用@Value注解注入配置，原工具类静态方法（static）无法读取Spring注入的配置，需全部改为非静态方法，通过注入方式获取配置。
    • Common模块特殊处理：Common模块无resource目录、无独立配置文件，作为嵌入模块运行在其他功能模块（主管/行程规划/路线制定）中，通过读取宿主模块配置获取敏感信息：Common模块创建配置类，用@Value注入配置并提供非静态get方法，工具类通过注入配置类获取敏感信息。

注意事项

  • 所有需读取配置的模块必须配置.env导入，否则占位符解析失败
  • 原静态工具方法必须改为非静态，否则配置读取为空

13-2 主管Agent暴露用户交互接口

核心概要

实现主管Agent作为用户交互核心节点，通过Web API接收旅游规划请求，完成端口分配、接口开发验证。

详细实现

  1. Controller开发：在主管模块创建Controller类，添加@RestController注解，开发旅游规划提交接口，指定接口地址，需配置UTF-8编码支持中文提交。

  2. 智能体注解配置：为主管Agent类添加@Component注解，确保可被Spring容器注入。

  3. 端口分配：各智能体节点分配独立端口避免冲突：

    智能体节点 端口号
    主管Agent 8081
    路线制定Agent 8082
    行程规划Agent 8085
    其他节点 8848

  4. 运行验证：启动主管模块后通过接口发送请求，可成功打印API Key、使用的大模型（如qwen3-max）信息，验证敏感信息配置生效、接口正常暴露（仅验证基础能力，后续流程未执行）。

13-3 Docker搭建Agent跟踪与Token消费分析（Langfuse平台）

核心概要

引入Langfuse开源可观测平台（文稿转录为nfuse/nem fields，以官方名称为准），实现大模型与智能体运行的实时监控，覆盖响应时间、Token消费、MCP调用、工具调用、上下文等核心指标。

Langfuse官方特性补充

Langfuse是集可观测性、提示管理、评估于一体的AI开发平台，支持OpenAI、LangChain、AgentScope等多框架集成，支持云服务和自部署。

部署与使用流程

  1. 环境要求：需提前安装Docker、Docker Compose。
  2. 启动命令：进入Langfuse目录执行启动命令，首次启动会自动下载Redis、ClickHouse等依赖组件，耗时较长。
  3. 访问与配置：
    • 启动后访问localhost:3000注册登录（密码需含大小写+特殊符号）
    • 创建项目后，在Settings→API Keys页面获取Public Key（pk-lf-xxx）和Secret Key（sk-lf-xxx），用于后续项目集成
  4. 核心功能：
    • 追踪智能体/大模型完整调用链路
    • 统计Token消费、响应时间、调用成功率
    • 监控MCP调用、工具调用、上下文传递

13-5 测试主管Agent接收Prompt及结构化输出

核心概要

实现主管Agent接口返回JSON结构化输出，兼容老模型，解决大模型返回格式不可控问题。

详细实现（结合AgentScope结构化输出官方文档）

  1. 请求格式约束：Common模块创建PromptSchema，规定用户提交的Prompt必须放在prompt字段下，接口需指定UTF-8编码支持中文。

  2. 响应格式定义：Common模块创建ResponseSchema，规定大模型返回内容必须放在response字段下，必须提供无参构造函数（AgentScope强制要求）。

  3. 智能体结构化输出配置：构建主管Agent时，通过structuredOutputReminder()配置输出模式：

    模式 特点 适用场景
    TOOL_CHOICE 强制调用工具，单API调用 支持tool_choice的模型（qwen3-max、gpt-4等）
    PROMPT（本项目选用） 提示词引导，可能多次调用 兼容DeepSeek等老模型
    配置示例：

  4. 响应解析：智能体运行时，通过response.getStructuredData(ResponseSchema.class)自动将返回内容匹配到ResponseSchema，提取response字段内容。

  5. 接口验证：提交{"prompt": "制定深圳到惠州最优驾车路线"}格式的请求，返回结果为JSON，内容自动存入response字段。

注意事项

  • 部分老模型（如DeepSeek早期版本）不支持结构化输出，会报错，需选用支持的模型（如qwen3系列）
  • 响应Schema必须有无参构造函数，否则解析失败

13-6 打印Agent深度思考过程

核心概要

实现大模型思考链（CoT）可视化，便于分析推理过程，基于AgentScope框架实现。

详细实现

  1. 模型选择：阿里百炼平台提供的模型分为「思考模式」和「仅输出模式」，需选择支持思考模式的模型，本项目选用DeepSeek R1（文稿转录为deep see kr 1，以官方名称为准），免费版可选开源支持思考模式的模型。
  2. 开启思考模式：Common模块的智能体工具类中，调用enableThinking(true)方法开启思考模式。
  3. 验证：提交问题（如“咖啡馆里想要背星巴克，星巴克指代什么”），日志中可打印完整思考链，再输出最终结果。

注意事项

  • 部分模型思考过程默认输出英文，教学场景无影响，生产环境可配置输出语言
  • 需确认所用模型在阿里百炼平台支持思考模式，否则无法打印思考过程

13-7 测试路线制定专员规划最优驾车路线

核心概要

实现主管Agent自动将用户需求拆分，传递给远程路线制定Agent，无需手动编写子Agent调用Prompt。

详细实现

  1. 远程Agent方法定义：在主管Agent中定义调用路线制定Agent的方法，添加@ToolParam注解，两个核心参数：
    • type：工具类型，标注为“擅长制定最优驾车路线的agent”
    • description：参数描述（如“驾车的起点和终点”），大模型自动从用户Prompt中提取对应信息
  2. Prompt自动传递：
    • Common模块创建PromptBuilder工具类，将用户提交的字符串Prompt转换为AgentScope要求的Msg对象
    • 主管Agent匹配到路线制定需求后，自动提取起点、终点，组装为Msg对象，通过call方法传递给远程路线制定Agent
  3. 结果返回：远程Agent（路线制定）调用百度地图MCP获取路线数据，执行完成后返回结果，主管Agent阻塞等待结果，最终组装为JSON返回前端。
  4. 验证：提交“制定深圳到惠州最优驾车路线”，路线制定Agent日志显示接收到请求，调用百度地图MCP后返回结果，主管Agent日志显示成功获取远程结果。

13-8 课程总结

核心知识点复盘

第13章两大核心知识点，通过旅游规划项目完整演示：

1. 多Agent协同

实现路径分为两类：

  • 工作流编排设计思想：

    实现方式 核心组件 特点
    Graph引擎 StateGraph、OverOrState 状态与数据在OverOrState中传输，Graph编排节点与连线
    FlowAgent基类 ParallelAgent、LmRoutingAgent、SequentialAgent 并行执行、路由分发、顺序执行
    （非重点）Jmanus Java版Manus框架 轻量多Agent实现

  • AgentScope企业级框架（重点）：

    核心特性：

    • 分布式Agent基于A2A协议通信，通过Nexus注册中心注册节点

    • MCP服务部署为单独节点（本项目用百度地图远程MCP服务）

    • Skills工程化、单独子任务由子Agent独立执行

    • 支持Docker分布式部署

2. 智能体自主决策

  • 复杂任务自动拆分：将整体需求拆分为简单子任务，自动分发给对应子Agent
  • 思考-行动-反思闭环：执行任务时遵循「思考→行动→反思验证」流程，不符合目标时自主调整执行步骤
  • 基于ReAct思考框架实现

附录：工程化部署补充（Docker，来自官方资料）

智能Agent可通过Docker实现生产级部署，核心优势：环境一致、资源隔离、快速扩缩容，性能提升显著（平均响应时间从320ms降至98ms，TPS从450提升至1120）。

部署流程

  1. 编写轻量Dockerfile（建议用alpine/slim基础镜像，多阶段构建优化体积）
  2. 构建镜像推送至仓库（公共/私有）
  3. 运行容器，配置持久化存储、健康检查、资源限制（CPU/内存）
  4. 集成ELK+Prometheus监控体系，实现全链路可观测

📝 整体注意事项

  1. 模型兼容性：结构化输出、思考模式需提前确认模型支持，避免运行时报错
  2. 工程化规范：敏感信息必须放.env、工具方法非静态、Langfuse采样率按环境配置
