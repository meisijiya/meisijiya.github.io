Java虚拟机---面试

作者: 老江湖
日期: 四月 19, 2026  10:52:36
分类: Java面试篇

Hi，我是 meisijiya ，也是 老江湖 ，感谢你的阅读，我们一起进步。

一、整体核心考点

JVM是Java后端面试核心考点，重点考察内存结构、垃圾回收、类加载机制、内存问题排查四大模块，答好可加分，答不上易拉低印象分。

二、JVM内存模型（JDK 8及以后）

1. 运行时内存分区

  • 程序计数器：线程私有，记录当前线程执行的字节码行号，用于本次CPU时间片用完后 下一轮CPU时间片到来时能够继续执行未完成部分的代码，唯一无OOM的区域。
  • Java虚拟机栈：线程私有，存方法栈帧（局部变量、操作数栈等），抛StackOverflowError/OOM。
  • 本地方法栈：为Native方法服务，HotSpot中与虚拟机栈合并，抛StackOverflowError/OOM。
  • Java堆：线程共享、最大内存区，存对象实例；分新生代（Eden+S0/S1）、老年代，OOM高发区。
  • 元空间（方法区实现）：替代永久代，使用本地内存，存类元数据、常量、静态变量、JIT编译后代码，运行时常量池（在类加载后，存字面量/符号引用，支持运行时新增常量）
  • 直接内存：堆外内存，NIO使用，受物理内存限制，可触发OOM。

2. 核心内存特性

  1. 堆 vs 栈

    1. 堆：存所有类的实例和数组；线程共享、动态扩容、存取慢、空间大；堆由 GC 回收；
    2. 栈：存基本类型 + 对象引用、方法调用上下文；线程私有、空间固定、存取快。

    栈帧中保存方法执行的完整上下文，包括：

    • 局部变量表：存储方法参数和内部定义的变量，基本类型数据直接存储在栈帧的局部变量表中的。
    • 操作数栈：字节码指令执行时的操作数临时存储区（如计算、赋值）。
    • 动态链接：指向运行时常量池的方法引用，支持多态调用。
    • 方法返回地址：方法执行完毕后，返回上层调用者的字节码位置。

  2. 堆分区比例：

    • 新生代（Young Generation）：分为Eden Space和Survivor Space，Eden:S0:S1=8:1:1；对象创建 -> Eden区 -> 满了触发Minor GC（新生代垃圾回收）-> 其中一个Survivor区 -> 一次或多次Minor GC后 -> Old。
    • 老年代（Old Generation/Tenured Generation）：空间通常比新生代大。存放过一次或多次Minor GC仍存活的长期对象会被移动到老年代。Major GC（Full GC）涉及老年代的垃圾回收，发生频率低，执行时间长。

  3. 大对象分配：在 G1 垃圾收集器中，任何超过 Region 一半大小的对象直接进入老年代，避免新生代GC 的 Eden/Survivor 间复制，减少 GC 开销和内存碎片。

  4. String存储：存于字符串常量池（堆中）；new String("meisijiya")会创建堆对象+常量池对象（常量不存在时）、当常量池存在“meisijiya”，则String s = "meisijiya"会复用String对象、String s = "meisi" + "jiya"在编译期会被优化为“meisijiya”，复用常量池、主动调用new String.intern()，手动入池复用。

    特别强调，String s这个引用是存储在虚拟机栈中的。

  5. 引用类型：强引用（不回收）>软引用（内存不足回收）>弱引用（下次GC必回收）>虚引用（最弱，配合队列管理堆外内存）。

    弱引用通过Java.lang.ref.WeakReference类实现。

    弱引用的使用场景：缓存、对象池、ThreadLocal，避免无用对象长期占用内存导致泄漏。

三、类加载与对象机制

1. 类加载器（双亲委派模型）

  1. 加载器层级：启动类加载器→平台/扩展类加载器→应用类加载器→自定义类加载器。
  2. 双亲委派：加载请求逐级委派父加载器，父加载失败子加载；保证类唯一性、防止核心类被篡改。

2. 类加载过程

加载→验证→准备→解析→初始化→使用→卸载

  • 准备：为静态变量分配内存并赋默认值；
  • 初始化：执行静态代码块、静态变量赋值。

3. 对象相关

  1. 对象创建：类加载检查（查常量池中类的符号引用的代表类，没有被加载过、解析和初始化过则执行相应类加载过程）→分配内存（在堆中划分同等于 类加载完成后确定的内存大小 的一块内存）→初始化零值（除了对象头，分配到的内存空间都初始化为零值）→设置对象头→执行构造方法。

    在HotSpot虚拟机中，对象在内存中存储的布局可以分为3块区域：对象头（Header）、实例数据（Instance Data）和对齐填充（Padding）。HotSpot虚拟机的对象头包括两部分信息，第一部分用于存储对象自身的运行时数据，如哈希码（HashCode）、GC分代年龄、锁状态标志、线程持有的锁、偏向线程ID、偏向时间戳等。对象头的另外一部分是类型指针，即对象指向它的类元数据的指针，虚拟机通过这个指针来确定这个对象是哪个类的实例。

  2. 对象生命周期：创建→使用→GC回收销毁。

四、垃圾回收（GC）

1. GC基础

  • 定义：自动回收无用对象内存，避免内存泄漏/溢出。
  • 触发条件：内存不足、手动调用System.gc()、分代内存阈值达标。

2. 垃圾判断算法

  1. 引用计数法：简单，无法解决循环引用问题。
  2. 可达性分析：Java主流，以GC Roots为起点追溯引用链，无引用链则判定为垃圾。

3. 垃圾回收算法

  • 标记-清除：效率低、产生内存碎片。
  • 复制算法：无碎片，浪费内存，适用于新生代。
  • 标记-整理：无碎片，适用于老年代。
  • 分代回收：新生代用复制算法，老年代用标记-清除/整理。

4. GC类型

  • Minor GC：回收新生代，Eden区满触发，频繁、速度快。
  • Major GC：回收老年代，频率低、耗时久。
  • Full GC：回收整堆+元空间，STW时间最长；触发于老年代/元空间不足、手动GC等。

5. 核心回收器

  1. CMS：老年代回收器，低延迟，标记-清除算法，有内存碎片，已废弃。
  2. G1：JDK9默认全堆回收器，分区设计、停顿时间可控、标记-整理无碎片。
  3. ZGC/Shenandoah：超低停顿，适用于超大堆、低延迟场景。

6. STW关键阶段

初始标记、重新标记、内存清理、对象转移阶段会触发STW。

五、内存问题排查

1. 核心概念

  • 内存泄漏：无用对象被强引用，无法被GC回收，逐步占用内存。

    内存泄漏诱因：

    • 静态集合泄漏：静态 List 持续添加对象，GC 无法回收；
      解决：减少静态集合，及时清理元素；
    • 资源未关闭：IO / 数据库连接未释放；
      解决：finally 关闭或 try-with-resources；
    • ThreadLocal 泄漏：线程未销毁，value 无法回收（key 为 null 的 Entry 的 value 就会一直存在一条强引用链：Thread Ref → Thread → ThreadLocalMap → Entry → value，永远无法回收，造成内存泄漏。）；
      解决：在finally块调用remove()清理。

    通用优化：及时关闭资源、避免滥用static、手动清理ThreadLocal、合理配置JVM参数。

  • 内存溢出（OOM）：申请内存无足够空间，JVM抛出OutOfMemoryError。

2. 四种内存溢出情况

堆内存溢出

  • 异常：java.lang.OutOfMemoryError: Java heap space
  • 原因：大对象分配、内存泄漏，多次 GC 后仍无足够堆内存分配新对象。
  • 解决：加 JVM 参数-XX:+HeapDumpOnOutOfMemoryError -XX:HeapDumpPath=./heapdump.hprof在发生OOM时生成堆快照→用 MAT/JProfiler 分析泄漏点→修复引用问题；调整-Xms/-Xmx扩大堆内存；优化代码分批处理数据。

虚拟机栈 / 本地方法栈溢出

  • 异常：StackOverflowError（递归过深压栈过量）或 OOM（栈空间扩展失败）
  • 原因：无限递归、方法调用层级过深、单个栈帧占用过大。
  • 解决：修复递归终止条件，递归改迭代；通过-Xss调整栈内存大小（谨慎调整，总内存固定时，单个线程栈越大，能创建的线程数越少）。

元空间溢出

  • 异常：java.lang.OutOfMemoryError: Metaspace
  • 原因：类 / 第三方包过多、动态大量生成类，导致元空间内存占用超限。

直接内存溢出

  • 异常：java.lang.OutOfMemoryError: Direct buffer memory
  • 原因：NIO 框架（如 Netty）分配堆外内存时，直接内存申请超出限制。
