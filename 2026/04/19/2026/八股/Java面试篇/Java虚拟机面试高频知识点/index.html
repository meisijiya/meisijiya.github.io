Java虚拟机--面试

作者: 老江湖
日期: 四月 19, 2026  10:52:36
分类: Java面试篇

Hi，我是 meisijiya ，也是 老江湖 ，感谢你的阅读，我们一起进步。

一、整体核心考点

JVM是Java后端面试核心考点，重点考察内存结构、垃圾回收、类加载机制、内存问题排查四大模块，答好可加分，答不上易拉低印象分。

二、JVM内存模型（JDK 8及以后）

1. 运行时内存分区

  • 程序计数器：线程私有，记录当前线程执行的字节码行号，用于本次CPU时间片用完后 下一轮CPU时间片到来时能够继续执行未完成部分的代码，唯一无OOM的区域。
  • Java虚拟机栈：线程私有，存方法栈帧（局部变量、操作数栈等），抛StackOverflowError/OOM。
  • 本地方法栈：为Native方法服务，HotSpot中与虚拟机栈合并，抛StackOverflowError/OOM。
  • Java堆：线程共享、最大内存区，存对象实例；分新生代（Eden+S0/S1）、老年代，OOM高发区。
  • 元空间（方法区实现）：替代永久代，使用本地内存，存类元数据、常量、静态变量、JIT编译后代码，运行时常量池（在类加载后，存字面量/符号引用，支持运行时新增常量）
  • 直接内存：堆外内存，NIO使用，受物理内存限制，可触发OOM。

2. 核心内存特性

  1. 堆 vs 栈

    1. 堆：存所有类的实例和数组；线程共享、动态扩容、存取慢、空间大；堆由 GC 回收；
    2. 栈：存基本类型 + 对象引用、方法调用上下文；线程私有、空间固定、存取快。

    栈帧中保存方法执行的完整上下文，包括：

    • 局部变量表：存储方法参数和内部定义的变量，基本类型数据直接存储在栈帧的局部变量表中的。
    • 操作数栈：字节码指令执行时的操作数临时存储区（如计算、赋值）。
    • 动态链接：指向运行时常量池的方法引用，支持多态调用。
    • 方法返回地址：方法执行完毕后，返回上层调用者的字节码位置。

  2. 堆分区比例：

    • 新生代（Young Generation）：分为Eden Space和Survivor Space，Eden:S0:S1=8:1:1；对象创建 -> Eden区 -> 满了触发Minor GC（新生代垃圾回收）-> 其中一个Survivor区 -> 一次或多次Minor GC后 -> Old。
    • 老年代（Old Generation/Tenured Generation）：空间通常比新生代大。存放过一次或多次Minor GC仍存活的长期对象会被移动到老年代。Major GC（Full GC）涉及老年代的垃圾回收，发生频率低，执行时间长。

  3. 大对象分配：在 G1 垃圾收集器中，任何超过 Region 一半大小的对象直接进入老年代，避免新生代GC 的 Eden/Survivor 间复制，减少 GC 开销和内存碎片。

  4. String存储：存于字符串常量池（堆中）；new String("meisijiya")会创建堆对象+常量池对象（常量不存在时）、当常量池存在“meisijiya”，则String s = "meisijiya"会复用String对象、String s = "meisi" + "jiya"在编译期会被优化为“meisijiya”，复用常量池、主动调用new String.intern()，手动入池复用。

    特别强调，String s这个引用是存储在虚拟机栈中的。

  5. 引用类型：强引用（不回收）>软引用（内存不足回收）>弱引用（下次GC必回收）>虚引用（最弱，配合队列管理堆外内存）。

    弱引用通过Java.lang.ref.WeakReference类实现。

    弱引用的使用场景：缓存、对象池、ThreadLocal，避免无用对象长期占用内存导致泄漏。

三、类加载与对象机制

1. 类加载器（双亲委派模型）

  1. 加载器层级：JVM 类加载器为层级委派结构，父子为组合委派关系，非 Java 类继承关系，共 4 个固定层级如下

    • 启动类加载器（Bootstrap，顶层根加载器）

      核心：加载 JDK核心类库（JDK8 及之前：rt.jar；JDK9+：模块化核心模块）；C++ 实现，JVM 内置，Java 层面无对应对象，getClassLoader()返回 null，无法被 Java 代码直接引用。

    • 平台类加载器（JDK9+）/ 扩展类加载器（JDK8 及之前，第二层）

      核心：JDK8 及之前加载扩展目录类；JDK9 + 加载 JDK 非核心平台模块类；逻辑父委派节点为启动类加载器。

    • 应用程序类加载器（AppClassLoader，第三层，开发者默认加载器）

      核心：加载用户 ClassPath / 模块路径的业务代码、第三方 Jar 包；可通过ClassLoader.getSystemClassLoader()获取，是日常开发默认使用的加载器。

    • 自定义类加载器（最下层，可扩展）

      核心：继承ClassLoader实现，可自定义类加载逻辑，支持加密加载、类隔离、热部署等场景。

  2. 双亲委派：类加载请求优先逐层向上委派给父加载器处理，直到顶层启动类加载器；仅当父加载器无法完成加载时，子加载器才会自行加载。（注：“双亲” 指层级委派关系，非类继承）；保证类唯一性、防止核心类被篡改。；为默认规则而非强制规则，可通过重写loadClass()打破委派（典型场景：JDBC SPI、Tomcat 类隔离、热部署）

2. 类加载过程

类从加载到 JVM 内存，到卸载出内存，完整生命周期共 7 个固定阶段：

加载 → 连接（验证、准备、解析） → 初始化 → 使用 → 卸载

关键规则：加载、验证、准备、初始化、卸载的执行顺序固定；解析阶段可延后至初始化之后执行，以支持 Java 动态绑定特性。

  1. 加载阶段（用户可自定义干预的唯一核心阶段）

    • 核心动作：

      1. 通过类的全限定名，获取该类.class 文件的二进制字节流（来源可自定义：本地文件、网络、加密流、数据库等）；

      2. 将字节流的静态存储结构，转化为方法区的运行时数据结构；

      3. 在内存中生成该类唯一的java.lang.Class对象，作为该类所有数据的全局访问入口。

        重点：该阶段是双亲委派模型的核心执行环节，可通过自定义类加载器干预类的加载逻辑。

  2. 连接阶段（JVM 自动执行，分 3 个子阶段）

    核心作用：将加载后的类数据，校验、整理、解析为 JVM 可直接运行的状态，分为 3 个串行子阶段：

    • 验证阶段（JVM 安全保障核心）

      核心：校验.class 字节流完全符合 JVM 规范，确保类的正确性，避免恶意 / 不合规代码危害虚拟机安全。

      核心校验项：文件格式校验、元数据验证、字节码验证、符号引用验证。

    • 准备阶段（静态变量内存分配核心）

      核心：为类的static 静态变量在方法区分配内存，并设置类型默认初始值（如 int=0、引用类型 = null）。

      重点：final 修饰的 static 常量，编译期已确定固定值，此阶段直接完成显式赋值，不设置默认初始值；该阶段不会为实例变量分配内存（实例变量随对象在堆中分配）。

    • 解析阶段（引用转换核心）

      核心：将常量池中的符号引用（无歧义描述目标的字面量，与内存布局无关），替换为直接引用（可直接定位目标的内存指针、相对偏移量等，与 JVM 内存布局强绑定，目标必定已在内存中）。

  3. 初始化阶段（类加载最后一步，静态逻辑执行核心）

    核心：执行编译器自动生成的类构造器<clinit>()方法，完成静态变量的显式赋值、静态代码块的逻辑执行。

    核心规则（重点）：

    1. <clinit>() ≠ 实例构造器<init>()，仅处理类静态内容，无需开发者手动定义；
    2. 父类的<clinit>() 优先于子类执行，保证父类静态逻辑先完成初始化；
    3. JVM 会保证<clinit>() 在多线程环境下被正确加锁、全局仅执行一次，确保类的唯一初始化；
    4. 仅类被「主动使用」时才会触发初始化（如 new 对象、调用类静态成员、反射访问、启动主类等）。

3. 对象相关

  1. 对象创建：类加载检查（查常量池中类的符号引用的代表类，没有被加载过、解析和初始化过则执行相应类加载过程）→分配内存（在堆中划分同等于 类加载完成后确定的内存大小 的一块内存）→初始化零值（除了对象头，分配到的内存空间都初始化为零值）→设置对象头→执行构造方法。

    在HotSpot虚拟机中，对象在内存中存储的布局可以分为3块区域：对象头（Header）、实例数据（Instance Data）和对齐填充（Padding）。
    HotSpot虚拟机的对象头包括两部分信息，第一部分Mark Word用于存储对象自身的运行时数据，如哈希码（HashCode）、GC分代年龄、锁状态标志、线程持有的锁、偏向线程ID、偏向时间戳等。对象头的另外一部分是类型指针（Klass Pointer），即对象指向它的类元数据（Klass 实例）的指针，虚拟机通过这个指针来确定这个对象是哪个类的实例。

  2. 对象生命周期：创建→使用→GC回收销毁。

四、垃圾回收（GC）

1. GC基础

  • 定义：自动回收无用对象内存，避免内存泄漏/溢出。
  • 触发条件：堆内存不足、手动调用System.gc()、分代内存满、元空间不足。
  • 回收目标：
    • 回收核心：Java 堆
    • 新生代（YGC）、老年代（OGC/FGC）、Humongous 大对象区。
    • 回收次要：方法区（元空间）
    • Full GC 附带回收：废弃常量 + 无用类元数据。
    • 不回收：程序计数器、虚拟机栈、本地方法栈（随线程自动释放）。
    • 不直接回收：堆外直接内存（靠 DirectByteBuffer 间接触发）。

2. 垃圾判断算法

  1. 引用计数法：简单地通过每个对象被分配的引用计数器在对象被引用或失效时进行加减一实现，无法解决循环引用问题。

  2. 可达性分析：Java主流，以GC Roots为起点追溯引用链，无引用链（不可达）则判定为垃圾。

    GC Roots对象包括：虚拟机栈（栈帧中的本地变量表）中引用的对象、方法区中类静态属性引用的对象、本地方法栈中JNI（Java Native Interface）引用的对象、活跃线程的引用等。

3. 垃圾回收算法

  • 标记-清除：效率低、产生内存碎片。

  • 复制算法：无碎片，浪费内存，适用于新生代。

  • 标记-整理：无碎片，适用于老年代。

  • 分代回收：新生代用复制算法，老年代用标记-清除/整理。

4. GC类型

  • Minor GC：回收新生代，Eden区满触发，频繁、速度快。
  • Major GC：回收老年代，频率低、耗时久。
  • Full GC：回收整堆（年轻代 + 老年代）+元空间（无用的类元数据进行卸载），STW时间最长；触发于老年代/元空间不足、手动GC等。

5. 核心回收器

JDK 21+ 官方认证生产级主流 GC 共 4 款，G1 为开箱即用的默认 GC，分代 ZGC 是官方主推低延迟方案，Parallel、Shenandoah 为场景化备选；CMS、ParNew 等已正式移除，该版本完全不可用。

  1. G1 GC【默认首选，通用平衡型】

    • 核心定位：JDK 9+ 全版本官方默认 GC，兼顾吞吐量与可控停顿的服务端通用型方案
    • 核心特性：堆划分为等大 Region，弱化传统分代边界；整体标记 - 整理、局部复制算法，无连续内存碎片；Mixed GC 优先回收垃圾收益最高的 Region，支持通过-XX:MaxGCPauseMillis精准设置目标停顿时间
    • 适用场景：绝大多数线上业务（微服务、中间件等），无需复杂调优即可稳定运行

  2. ZGC【官方主推，超低延迟首选】

    • 核心定位：JDK 21+ 官方主推亚毫秒级全并发低延迟 GC，JDK 21 默认开启分代优化，解决非分代版本内存占用问题
    • 核心特性：基于染色指针 + 读屏障，几乎所有 GC 阶段与用户线程并发；停顿时间稳定≤1ms，与堆大小无关，支持 TB 级超大堆，无内存碎片，启用参数-XX:+UseZGC
    • 适用场景：金融交易、高并发在线业务、实时计算等对延迟要求极高的场景

  3. Parallel GC【吞吐量优先】

    • 核心定位：JDK 8 默认 GC，JDK 21+ 全量支持的极致吞吐量优先并行 GC
    • 核心特性：新生代复制算法（Parallel Scavenge）、老年代标记 - 整理算法（Parallel Old），核心 GC 阶段全多线程并行；最大化 CPU 利用率，可精准控制吞吐量目标
    • 适用场景：后台批处理、大数据计算、离线任务等对延迟不敏感、追求执行效率的场景

  4. Shenandoah GC【低延迟备选方案】

    • 核心定位：RedHat 主导开发的超低延迟并发 GC，与 ZGC 同级的低延迟备选方案
    • 核心特性：基于Brooks 转发指针 + 读屏障，实现标记、整理、回收全流程与用户线程并发；停顿时间与堆大小无关，支持超大堆
    • 适用场景：RHEL/CentOS 等 Linux 发行版自带 OpenJDK 的低延迟业务场景

新生代：Serial、ParNew、Parallel Scavenge；

老年代：Serial Old、Parallel Old、CMS（已废弃）；

全堆：G1（JDK9 默认）、ZGC、Shenandoah。

6. STW(stop the world)关键阶段

G1 GC 的标准流程：G1 通过“短暂 STW 初始标记”锁定根对象，随后“并发标记”遍历全堆找存活对象，接着“短暂 STW 最终标记”修正引用变动，最后“STW 复制转移”将存活对象移入新 Region 并清空旧内存来完成回收。

G1 GC 的 STW（Stop The World）关键阶段集中在 初始标记、最终标记（Remark）、筛选回收（Evacuation） 三个核心环节：

  • 初始标记全程 STW、耗时极短，仅标记 GC Roots 直接关联的存活对象，G1 通常借 Young GC 的 STW 窗口完成该阶段，几乎无额外停顿；

  • 最终标记全程 STW、是三大阶段中耗时最短的环节，核心是处理并发标记期间 SATB 写屏障记录的引用变更，修正并最终确认存活对象的标记结果；

    SATB（Snapshot-At-The-Beginning，起始快照）写屏障是 G1 用来保证并发标记正确性的核心机制，用来解决“标记过程中用户线程还在改引用，导致漏标对象”的问题。

  • 筛选回收全程 STW 且时长可控，会筛选垃圾收益最高的 Region 组成回收集，并行完成存活对象的复制转移与旧 Region 的内存回收，可通过参数精准控制停顿时长；

特别提醒，并发标记阶段全程与用户线程并发执行，无任何 STW 停顿。

五、内存问题排查

1. 核心概念

  • 内存泄漏：无用对象被强引用，无法被GC回收，逐步占用内存。

    内存泄漏诱因：

    • 静态集合泄漏：静态 List 持续添加对象，GC 无法回收；
      解决：减少静态集合，及时清理元素；
    • 资源未关闭：IO / 数据库连接未释放；
      解决：finally 关闭或 try-with-resources；
    • ThreadLocal 泄漏：线程未销毁，value 无法回收（key 为 null 的 Entry 的 value 就会一直存在一条强引用链：Thread Ref → Thread → ThreadLocalMap → Entry → value，永远无法回收，造成内存泄漏。）；
      解决：在finally块调用remove()清理。

    通用优化：及时关闭资源、避免滥用static、手动清理ThreadLocal、合理配置JVM参数。

  • 内存溢出（OOM）：申请内存无足够空间，JVM抛出OutOfMemoryError。

2. 四种内存溢出情况

堆内存溢出

  • 异常：java.lang.OutOfMemoryError: Java heap space
  • 原因：大对象分配、内存泄漏，多次 GC 后仍无足够堆内存分配新对象。
  • 解决：加 JVM 参数-XX:+HeapDumpOnOutOfMemoryError -XX:HeapDumpPath=./heapdump.hprof在发生OOM时生成堆快照→用 MAT/JProfiler 分析泄漏点→修复引用问题；调整-Xms/-Xmx扩大堆内存；优化代码分批处理数据。

虚拟机栈 / 本地方法栈溢出

  • 异常：StackOverflowError（递归过深压栈过量）或 OOM（栈空间扩展失败）
  • 原因：无限递归、方法调用层级过深、单个栈帧占用过大。
  • 解决：修复递归终止条件，递归改迭代；通过-Xss调整栈内存大小（谨慎调整，总内存固定时，单个线程栈越大，能创建的线程数越少）。

元空间溢出

  • 异常：java.lang.OutOfMemoryError: Metaspace
  • 原因：类 / 第三方包过多、动态大量生成类，导致元空间内存占用超限。

直接内存溢出

  • 异常：java.lang.OutOfMemoryError: Direct buffer memory
  • 原因：NIO 框架（如 Netty）分配堆外内存时，直接内存申请超出限制。
