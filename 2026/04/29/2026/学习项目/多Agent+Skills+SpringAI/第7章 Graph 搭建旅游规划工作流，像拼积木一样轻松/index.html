第7章 Spring AI Alibaba 多智能体工作流编排实战

📌 本章概述

本章围绕Spring AI Alibaba 多智能体协同工作流展开，覆盖两种官方编排方案：

  1. Graph引擎：基于可视化图结构编排节点与流程，适合流程固定、节点关系明确的场景
  2. Flow Agent基类：基于智能体自主决策编排，适合需要智能体动态分配任务、调用工具的场景

内容从核心概念、基础组件、版本差异、实战案例到高级特性（持久化、流式输出、内存管理）逐层深入，所有内容基于Spring AI Alibaba 1.1最新版本，同时标注1.0版本的差异，确保内容不过时。

7.1 工作流编排核心概念与场景选型

7.1.1 多智能体协同的两种设计思想

根据7-1文稿，多智能体协同有两种底层设计模式：

设计模式 核心特点 代表框架
Agent自主代理模式 智能体自主决定执行步骤、工具使用，无需预定义流程 AgentScope、Spring AI Alibaba Flow Agent
工作流编排模式 基于Graph引擎预定义节点、边、协作关系，流程可视化可控 Spring AI Alibaba Graph引擎

7.1.2 典型场景：旅行规划工作流选型

7-1文稿中旅行规划场景涉及5个智能体，需根据协作关系选择合适的流程模式：

智能体类型 核心功能 协作关系
主管智能体 用户意图识别、任务拆解分发 流程起点，按任务类型分发到路线/行程智能体
路线规划智能体 路线规划、距离计算、经纬度转换 与行程智能体并行执行
行程规划智能体 天气查询、景点信息搜索 与路线智能体并行执行
费用统筹智能体 费用计算、表格制作，汇总路线+行程费用 接收路线、行程结果，输出总费用
汇总智能体 整合所有结果，生成含交通、行程、开销明细的可视化方案 流程终点，返回最终结果给用户

选型结论：旅行规划适合「并行+汇总」混合流程：

主管分发任务 → 路线/行程智能体并行执行 → 结果汇总到费用统筹 → 总费用<1000则到汇总智能体，否则退回行程规划重排 → 返回用户

7.1.3 两种编排方案对比

根据7-13总结，Spring AI Alibaba 1.1支持两种编排方式，可按场景选择：

对比项 Graph引擎 Flow Agent基类
核心实现 StateGraph定义节点、边、状态 继承FlowAgent，组合子智能体
执行单元 函数/方法的节点（Node） 具备思考、工具使用能力的智能体
流程定义 通过边（Edge）预定义流程 智能体自主决策流程，支持并行/顺序/路由分发
适用场景 流程固定、节点关系明确、需要可视化编排 需要智能体自主决策、动态任务分配
✅ 建议：需要智能体自主决策优先选Flow Agent；流程固定的场景优先选Graph引擎。

7.2 Graph引擎核心组件（Spring AI Alibaba 1.1 最新版本）

Graph引擎核心三要素：状态（State）、节点（Node）、边（Edge），1.0版本差异将在7.5节单独说明。

7.2.1 全局状态：OverAllState

对应7-2文稿+官方文档核心概念：

  • 作用：贯穿整个工作流的共享内存，所有节点可读写，存储上下文数据

  • 存储格式：Key-Value结构，Key为String类型，Value需指定更新策略

  • 更新策略（KeyStrategy）：

    策略类 作用 适用场景
    ReplaceStrategy 新值完全覆盖旧值 只需保留最新结果的场景（如分类结果、最终响应）
    AppendStrategy 新值追加到旧值后（如List类型追加元素） 需要保留历史记录的场景（如多轮对话消息、节点日志）
    RemoveByHash 根据hashCode删除指定值 AppendStrategy场景下删除特定历史值

  • 状态策略定义：通过KeyStrategyFactory定义所有Key的更新策略，1.1版本无需手动注册，工厂类自动生效：

💡 注意：7-2文稿提到的1.0版本使用OverallStateFactory手动注册策略，1.1版本已废弃该方式。

7.2.2 工作流框架蓝图：StateGraph

核心概念：

  • 作用：工作流的载体，用于添加节点、边，定义工作流结构

  • 1.1版本构造方法：

    构造方式 参数说明 适用场景
    最简构造 仅需传入KeyStrategyFactory（状态策略为必填项） 基础工作流
    带名称构造 工作流名称 + KeyStrategyFactory 需要标识工作流场景
    自定义序列化器构造 工作流名称 + KeyStrategyFactory + 自定义序列化器 需要兼容自定义类型的场景

  • 序列化器（官方文档补充）：默认使用Jackson序列化器，支持三种自定义方式：

    1. 为自定义类添加Jackson注解（如@JsonCreator、@JsonProperty）
    2. 定制StateGraph默认Jackson序列化器的ObjectMapper
    3. 完全自定义序列化器（如Gson、JDK序列化器）

7.2.3 执行单元：Node（节点）

核心概念：

  • 作用：工作流的执行单元，实现具体业务逻辑，输入输出均基于OverAllState

  • 节点类型：

    接口 作用 适用场景
    NodeAction 同步节点，入参OverAllState，返回Map更新状态 普通同步业务逻辑
    AsyncNodeAction 异步节点，入参OverAllState，返回Map更新状态 异步执行的节点
    AsyncNodeActionWithConfig 异步节点，入参OverAllState+RunnableConfig 需要访问配置、长期内存的节点

  • 节点添加：通过StateGraph.addNode(String nodeId, AsyncNodeAction nodeAction)添加，建议使用单独类实现节点逻辑，避免Lambda导致代码混乱：

7.2.4 流程顺序：Edge（边）

核心概念：

  • 作用：定义节点之间的流转关系

  • 边的类型：

    边类型 方法 作用
    普通边 addEdge(String fromNodeId, String toNodeId) 固定从fromNodeId流转到toNodeId
    条件边 addConditionalEdges(String fromNodeId, AsyncEdgeAction router, Map<String, String> routingMap) 根据router函数返回值，匹配routingMap中的节点ID流转
    特殊节点边 使用START（开始节点）、END（结束节点）作为特殊ID 定义工作流入口和出口

7.5 版本差异：Spring AI Alibaba 1.0 vs 1.1

核心概念完全通用，仅以下差异：

对比项 1.0版本 1.1版本（最新） 迁移建议
状态策略定义 使用OverallStateFactory手动注册Key-Strategy到OverAllState 使用KeyStrategyFactory，无需手动注册，工厂类自动生效 替换为KeyStrategyFactory，删除手动注册代码
StateGraph构造 最少2个参数（名称、OverallStateFactory） 最少1个参数（KeyStrategyFactory，必填） 调整构造方法，移除OverallStateFactory参数
核心依赖 agent-framework spring-ai-alibaba-graph（Graph引擎独立模块） 更新Maven依赖，移除旧模块
✅ 建议：新项目直接使用1.1版本，老项目按上述差异调整即可，工作流核心代码无需大幅修改。

7.6 高级特性

7.6.1 持久化与检查点（Checkpointer）

  • 作用：保存工作流执行状态，支持断点续跑、历史回溯、人机协作、多轮对话记忆

  • 核心概念：

    概念 说明
    会话（Thread） 唯一ID（threadId）标识一次工作流执行的生命周期，所有检查点关联到对应会话
    检查点（Checkpoint） 每个节点执行后保存的状态快照，包含当前状态、下一个执行节点、配置等
    检查点器（Saver） 检查点存储实现，内置MemorySaver（测试用），支持自定义RedisSaver等生产级实现

7.7 实战案例

7.7.1 案例1：基础链式工作流

目标：搭建两个节点的链式工作流，实现用户输入处理→结果返回

  1. 定义状态策略：KeyStrategyFactory，定义k1（覆盖）、k2（追加）两个Key
  2. 定义两个节点：分别读取k1、修改后返回
  3. 搭建StateGraph：添加节点，连线START→Node1→Node2→END
  4. 编译运行：使用invoke方法验证结果

7.7.2 案例2：旅行规划工作流

目标：搭建5智能体旅行规划工作流，支持费用超1000时退回行程规划重排

  1. 定义5个节点：主管（任务分发）、路线规划、行程规划、费用统筹、汇总
  2. 定义状态：总体费用（覆盖策略）
  3. 搭建边：
    • START→主管节点
    • 主管条件分发：路线相关→路线规划，行程相关→行程规划
    • 路线/行程并行执行→费用统筹节点
    • 费用统筹条件判断：总费用<1000→汇总节点→END；≥1000→行程规划节点（重排）
  4. 编译运行：测试深圳到惠州的旅行需求，验证费用判断逻辑

7.7.3 案例3：客服邮件处理工作流（官方文档）

目标：搭建自动处理客服邮件的工作流，支持分类、文档搜索、人工审核、自动回复

  1. 定义节点：读取邮件、分类意图、文档搜索、Bug跟踪、起草回复、人工审核、发送回复
  2. 定义状态：邮件内容、分类结果、搜索结果、草稿回复等，按场景配置覆盖/追加策略
  3. 搭建边：按邮件处理流程连线，分类节点条件路由，起草回复后条件路由到人工审核/发送回复
  4. 配置中断点：在人工审核节点前中断，支持人工编辑回复后恢复发送

7.8 本章总结

  1. Spring AI Alibaba支持两种工作流编排方式：Graph引擎（固定流程）、Flow Agent（智能体自主决策）
  2. Graph引擎核心三要素：OverAllState（状态）、Node（节点）、Edge（边），1.1版本使用KeyStrategyFactory定义状态策略，无需手动注册
  3. 工作流需编译为CompiledGraph后运行，支持阻塞（invoke）和流式（stream）两种模式，可图形化验证结构
  4. 高级特性支持持久化检查点、人机协作、短期/长期内存、原生流式输出，可满足复杂业务场景
  5. 版本建议：新项目直接使用Spring AI Alibaba 1.1版本，老项目迁移仅需调整状态策略定义和StateGraph构造方法
