第8章 多Agent协同框架设计与实现

第一部分 多Agent核心设计基础

8-1 多Agent设计思路：角色分工

核心执行流程（双流程并存）

  1. 多Agent协同流程：用户提需求 → 大模型意图识别 → 主管级Agent拆解复杂任务 → 细分任务分配给对应专业Agent → 各Agent执行生成中间结果 → 主管Agent整合结果 → 返回用户
  2. 单Agent执行流程：步骤规划 → 按序执行步骤 → 每步完成后监测中间结果 → 评估是否符合预期 → 符合则执行下一步，不符合则动态调整后续步骤 → 循环至所有步骤完成

注意：多Agent协同流程包含单Agent执行流程，两类流程共同构成完整执行逻辑。

多Agent协同框架三大核心组成部分

  1. 主管级智能体：负责复杂任务划分、任务分派、结果整合
  2. 多智能体协同操作与环境：所有智能体处于同一环境；基于同大模型的智能体不适用A2A通信协议（A2A仅适用于不同大模型的智能体通信）
  3. 中间结果存储整合模块：负责存储各Agent中间结果、整合为统一答案

框架三大基础类

类名 别名 功能说明 子类/方法
环境类 Env类 存储框架全局记忆信息、参数、包含的所有智能体角色信息 -
动作类 Action类 所有智能体调用工具、调用LLM大模型的统一入口 方法1：思考方法（步骤规划） 方法2：执行方法（按规划执行）
角色类 - 存储智能体角色信息、执行状态、中间结果 子类1：RoleSetting（角色配置类，存角色、擅长工具等信息） 子类2：RoleContext（角色状态类，存执行步骤、中间结果、当前状态等信息）

8-2 多Agent设计思路：冲突协商

前序内容回顾

  • 双核心执行流程、两大全局类（Env环境类、Action动作类）、角色类结构
  • 主管Agent通过角色分配算法，根据各Agent的角色能力匹配分配对应任务

智能体协作两大模块

1. 结构化通信

同大模型的智能体之间采用结构化通信方案，如RPC、WebSocket等，不适用A2A协议。

2. 冲突调解模块

智能体执行任务时不可避免产生冲突，调解机制包含4部分内容：

  1. 多议题协商：并行解决同时发生的多个冲突，对应多议题协商冲突网络模型
  2. 动态角色转换：冲突发生时，涉事Agent暂抛原角色，加入临时协调群转为辩论角色（如正方/反方），完成冲突调解后恢复原角色
  3. 信用分配：根据Agent信用分采纳方案，信用分高的经验丰富Agent的方案优先被采用
  4. 共识形成：冲突调解后通过WSLL算法达成共识，避免冲突持续

冲突调解整体抽象为冲突调解模型，包含多议题协商、动态角色转换、信用规则等子模型，最终通过共识形成模块输出一致结果。

8-3 多Agent设计思路：SOP管理机制

前序模块回顾

  1. 全局模块：Env环境类、Action动作类
  2. 角色分工模块：角色类、角色分配算法
  3. 通信与冲突协调模块：结构化通信、冲突调解模型、WSLL共识算法

第四大核心模块：SOP（标准流程）模块

作用

统筹智能体行动顺序，明确任务执行先后、依赖关系，避免协同混乱，提升整体效率。类比团队中主管的全局统筹，明确每个成员的任务时间、前后依赖。

实现方式
  • 核心为SOP类，包含State状态节点类，整体为树状结构流程图
  • 每个状态节点对应Agent的位置、需执行的任务、步骤、后续流向
  • SOP生成方式：
    1. 主管Agent预设：提前手动配置完整流程
    2. 大模型动态生成：根据Agent完成情况、用户需求实时调整节点内容、分配Agent

完整多Agent框架组成总结

  1. 双核心流程：
    • 多Agent协同流程：意图识别→任务规划→任务执行→结果整合
    • 单Agent执行流程：步骤规划→执行→监测→评估→调整（循环）
  2. 五大模块：
    全局模块、任务切分模块、角色模块、通信及冲突协调模块、SOP标准流程模块

第二部分 主流多Agent框架解析

8-4 Manus多Agent的技术架构

Manus核心双骨架流程

  1. 主管Agent流程：理解用户需求→拆解复杂任务为简单子任务→根据角色能力分配子任务→整合所有子任务结果→返回用户
  2. 单Agent执行流程：基于LLM推理能力+MCP协议调用外部工具（第三方API、数据库、浏览器等）完成子任务，每步执行后监测结果、动态调整

核心组件

组件 定位 功能
任务规划器 团队主管 1. 初始化需求理解，拆解任务为子任务 ；2. 初始化隔离容器（每个Agent调用工具需在独立容器中，基于MCP协议，支持第三方API、本地命令、浏览器操作等）
任务执行调度器 项目经理 1. 分配子任务给对应Agent ；2. 协调Agent工作、解决矛盾 3. 汇总所有子任务结果，生成用户友好输出

隔离容器特点

  • 每个Agent的工具调用在独立容器完成，避免数据混乱
  • 容器与用户本地文件/服务器文件隔离，出错不影响核心文件，保障安全
  • Manus为不开源框架，遵循通用多Agent拆解-分配-工具调用闭环逻辑

行业背景

  • 国外代表框架：资料原文提及为“基于拍摄的love train”（为语音转写误差），此外还有LangChain等
  • 国内代表框架：Manus、京东、蚂蚁集团推出的多Agent框架
  • 趋势：多智能体协同是未来必然方向，单Agent无法完成所有复杂任务

8-5 Manus是自主决策的AI Agent

Manus两大AI自循环

  1. 全局计划层自循环：每执行完一个子任务→监测结果→评估是否符合预期→动态调整后续计划
  2. 单Agent执行层自循环：单Agent执行子任务时，每完成一个步骤→监测结果→评估→动态调整后续步骤

Manus三大核心特点

  1. 全自主决策：从需求理解、任务拆解、分配、执行、反馈调整到结果汇总，全流程无需人类介入，仅用户提交需求即可自动完成
  2. 多Agent协同：隔离容器保障Agent独立工作，任务可同步/异步执行
  3. 工具调用能力：基于MCP协议调用浏览器、数据库、第三方API等外部工具，真正形成生产力，区别于仅支持对话的传统大模型

行业影响

Manus推出时引发广泛关注，核心是突破了传统大模型仅能对话的局限，实现了自主决策+工具调用+全流程自动化的生产力落地。

8-6 OpenManus复刻Manus的架构思路

框架关系

  • Manus：首创多Agent协同+自主决策+隔离容器+工具调用的闭环架构

  • OpenManus：Python版Manus复刻框架，完全继承Manus设计思路

  • JManus：Java版OpenManus复刻，基于Spring Boot + Spring AI（阿里版）开发

OpenManus三大核心特点（继承Manus）

  1. 智能体模块化：每个Agent为独立功能单元，支持即插即用，新增Agent不影响原有系统运行

  2. 工具驱动：采用工具箱设计，支持MCP协议，新增工具无需复杂配置，直接放入工具集合即可被Agent调用

  3. 复杂任务处理

    ：双模块架构

    • 规划模块：全局任务分解、流程设计
    • 执行模块：子任务独立执行，支持动态调整

两大智能体思考框架

  1. Plan-and-Execute（PlanAct）：处理复杂任务

  2. Reason-and-Action（ReAct）：处理简单任务

第三部分 智能体思考框架

8-7 PlanAct是全局流程规划的主管

OpenManus架构定位

多智能体协同开发框架，核心目标为实现Agent自由创建与集群协作。

三大核心模块

  1. 模块化Agent：实现新Agent简单创建、集成，即插即用
  2. 工具驱动：外部工具集成，支持MCP协议，工具箱可动态扩展
  3. 复杂任务处理：规划+执行双主干，规划模块负责全局任务分解，执行模块负责子任务实施

Plan-and-Execute（PlanAct）思考框架

  • 适用场景：有明确依赖关系的复杂任务，类比团队主管工作模式
  • 两阶段结构：
    1. 规划阶段：将复杂需求分解为子任务，制定执行顺序、优先级排序
    2. 执行阶段：分配子任务给对应Agent，监控执行情况
  • 动态调整机制：实时评估子任务完成质量，根据结果调整后续任务顺序、资源分配，保障最终结果符合原始意图

知识小结

知识点 核心内容
OpenManus架构思路 多智能体协同开发框架，含三大核心模块： 1. 智能体模块化（自由创建/加入集群） 2. 工具驱动（外部工具集成，支持MCP协议） 3. 复杂任务处理（规划+执行双主干）
Plan and Execute思考框架 分两阶段： 1. 规划阶段：拆解复杂任务为子任务并排序 2. 执行阶段：动态监测子任务完成情况并调整
Reason and Act思考框架 先理性归因 / 推演（Reason），再落地行动迭代（Act），是一套从思考到执行、闭环落地的极简决策框架。
智能体自主决策特性 基于两大思考框架实现： - 整体规划自主思考 - 子任务执行自主决策

8-8 ReAct是灵活应变的基层执行者

Reason-and-Action（ReAct）思考框架

  • 适用场景：线性单一简单任务，类比人类“先谋后动”的工作方式
  • 双步骤流程：
    1. 思考阶段：确定任务执行步骤顺序、选择适用工具集、建立预期目标标准
    2. 行动阶段：按计划执行具体操作、持续进行结果校验、及时修正执行偏差
  • 特点：强调执行过程中的实时反馈机制

两大思考框架对比

对比维度 Plan-and-Execute（PlanAct） Reason-and-Action（ReAct）
任务复杂度 多层级嵌套复杂任务 线性单一简单任务
资源分配 需要多智能体协作 通常单智能体完成
调整频率 动态调整频率高，侧重全局计划优化 侧重执行过程优化
思考范围 全局任务规划 单任务步骤规划

知识小结

知识点 核心内容
Plan-and-Execute思考框架 分两阶段（规划+执行），适用于复杂任务： 1. 规划：需求分解为子任务，分配执行顺序 2. 执行：监控子任务完成情况，动态调整顺序
Reason-and-Action思考框架 分两步骤（思考+行动），适用于简单任务： 1. 思考：明确步骤与工具 2. 行动：执行中持续反思目标对齐性
框架对比维度 任务复杂度：Plan-and-Execute > Reason-and-Action 灵活性：Plan-and-Execute支持动态调整，Reason-and-Action侧重线性执行

第四部分 JManus实战：从架构到代码实现

8-9 JManus的文件架构以及Prompt提交入口

JManus定位

OpenManus的Java复刻版，架构思路完全一致，基于Spring Boot + Spring AI（阿里版）开发。

文件架构（按名称推断功能）

文件夹/文件 功能说明
agent 存放智能体属性、创建相关工具类
config 配置文件夹
exception 异常报错处理文件夹
i18n 多语言支持文件夹
llm 调用大模型的文件夹
toolbox 工具箱，存放自定义工具
workflow 工作流相关逻辑
prompt 提示词工程文件夹
JManusApplication 整个JManus的入口文件（Spring Boot启动类）
event 任务事件相关文件夹
recorder 实体类文件夹
dynamic 自定义Agent、MCP、Prompt等动态扩展内容存放处
planning 核心文件夹，存放计划相关逻辑，为整个架构的核心

Prompt提交入口

  • 用户请求通过planning/controller/ManusController的/sq/query（POST请求）方法提交
  • 流程：接收用户Prompt→生成计划ID、记忆ID→初始化执行上下文→通过计划工厂执行后续流程

8-10 JManus的核心：计划协调器

JManus执行总流程

用户提交请求→初始化执行上下文（ExecutionContext，存储计划ID、记忆ID等属性）→创建计划协调器（PlanningCoordinator）→异步执行计划全生命周期→立即返回200状态码给用户

计划协调器创建流程

  1. 入口：ManusController的/sq/query方法调用PlanningFactory.createPlanningCoordinator()

  2. 实际构造：PlanningCoordinator构造方法需传入三个核心参数，管理计划全生命周期

    参数 类名 功能
    计划开始对象 PlanCreator 负责任务意图分析、初步计划创建
    计划执行对象 PlanExecutorFactory 负责根据计划类型创建对应执行者、执行计划步骤
    计划结束对象 PlanFinalizer 负责所有计划执行完成后结果总结

特点

  • 核心业务逻辑：所有任务的分配、执行、完成均依赖计划协调器
  • 异步执行：用户提交请求后无需等待，系统自动完成后续所有操作（数据库查询、文档调用、API请求等）

8-11 JManus向大模型请求计划创建

计划协调器生命周期两阶段

  1. 计划阶段：创建计划、意图分析
  2. 执行计划阶段：执行计划步骤、汇总结果

计划开始（PlanCreator）核心逻辑

  1. 调用ChatClient（Spring AI 阿里版大模型客户端）向语言大模型发起请求
  2. Prompt组装要求：必须封装两部分信息
    • JManus拥有的Agent信息：角色、擅长工具、属性等
    • 用户需求信息
  3. 结果处理：提取大模型返回的计划内容，封装为PlanInterface对象，写入执行上下文（ExecutionContext）

8-12 根据计划类型创建不同的执行者

计划执行对象创建规则

由PlanExecutorFactory采用工厂模式，根据计划类型创建对应执行对象：

计划类型 对应执行对象 适用场景
simple（简单计划） PlanExecutor 任务简单，无需进一步拆解，直接分配Agent执行
advanced（复杂计划） MapReducePlanExecutor 任务复杂，需主管Agent进一步拆解为子任务后，分配给多个子Agent执行

创建流程

计划协调器调用executePlan()方法→从执行上下文提取PlanInterface计划对象→判断计划类型→工厂类创建对应执行对象

8-13 不同的执行者调用不同的执行流程

两类执行对象的执行流程

1. PlanExecutor（简单计划执行对象）
  • 流程：提取计划→拆分为单个步骤→for循环按顺序执行每个步骤→为每个步骤分配匹配的Agent→调用Agent的run()方法执行步骤→调用setState()记录Agent执行状态
2. MapReducePlanExecutor（复杂计划执行对象）
  • 流程：提取计划→拆分为多个分支（节点）→for循环执行每个分支→每个分支再拆分为单个步骤→for循环执行步骤→分配Agent执行→记录状态

自定义Agent要求

需继承BaseAgent抽象类，实现两个核心方法：

  1. run()：智能体执行任务的方法
  2. setState()：记录智能体执行过程状态信息的方法

本章核心要点汇总

  1. 多Agent框架核心为双流程（多Agent协同、单Agent执行）+ 五大模块（全局、任务切分、角色、通信冲突、SOP）
  2. Manus首创自主决策+隔离容器+工具调用的多Agent闭环架构，OpenManus、JManus均为其复刻实现
  3. 复杂任务用Plan-and-Execute（PlanAct）框架，简单任务用Reason-and-Action（ReAct）框架
  4. JManus核心为计划协调器，管理计划全生命周期，根据计划类型创建不同执行者，支持简单/复杂任务的差异化执行
