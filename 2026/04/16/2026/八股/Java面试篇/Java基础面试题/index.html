Java基础--面试

作者: 老江湖
日期: 四月 16, 2026  9:05:23
分类: Java面试篇

Hi，我是 meisijiya ，也是 老江湖 ，感谢你的阅读，我们一起进步。

一、整体概述

Java基础是面试基础门槛，答不好易暴露基本功问题；本文覆盖高频核心考点，侧重原理理解，重点掌握：面向对象、int/Integer、String系列、equals/hashCode、Java8/17/21新特性、反射。

二、核心概念

  1. Java特点：跨平台（JVM+字节码）、纯面向对象、自动GC内存管理
  2. 优劣
    1. 优势：跨平台、生态完善、自动GC、内置多线程、安全稳定
    2. 劣势：性能低于C++/Rust、语法繁琐、JVM内存占用高
  3. 跨平台原理：源码编译为字节码，不同平台JVM将字节码翻译为对应机器码；JVM不跨平台，Java程序跨平台
  4. JVM/JDK/JRE：JDK(开发工具包) ⊃ JRE(运行环境) ⊃ JVM(虚拟机)+核心类库
  5. 执行类型：Java为编译+解释混合执行（先编译字节码，JVM解释+JIT即时编译）
  6. 编译型vs解释型：编译型一次编译终身执行（快、跨平台差）；解释型逐行执行（跨平台好、效率低）
  7. Java vs Python：Java静态类型、混合执行；Python动态类型、解释执行
  8. 参数传递：Java只有值传递；基本类型传值副本，引用类型传引用副本

三、数据类型

  1. 8种基本类型：byte/short/int/long/float/double/char/boolean
  2. int/long：int(32位4字节)、long(64位8字节)，可互转，long强转int易溢出
  3. 类型转换：自动转换（小范围→大范围）；强制转换（大范围→小范围，易溢出/精度丢失）
  4. 精度问题：金钱计算必须用BigDecimal，避免double二进制精度丢失
  5. 装箱/拆箱：基本类型与包装类自动转换，循环中自动装箱会产生冗余对象
  6. Integer：int包装类，适配泛型/集合；保留int是因效率更高、内存占用更小
  7. Integer缓存：默认-128~127对象复用，避免重复创建

四、面向对象

  1. 三大特性：封装（隐藏内部细节）、继承（代码复用）、多态（同一接口不同实现）

  2. 多态体现：方法重载（编译期，同类同名不同参）、方法重写（运行期，子类覆写父类）、接口实现

  3. 设计原则：单一职责、开放封闭、里氏替换、接口隔离、依赖倒置、迪米特法则

    1. 单一职责原则（SRP）

      一个类 / 模块只做一件事，只承担一个责任。职责过多会导致修改牵一发而动全身。

    2. 开放封闭原则（OCP）

      对扩展开放，对修改关闭。新增功能通过扩展实现，不改动原有稳定代码。

    3. 里氏替换原则（LSP）

      子类必须能完全替换父类而不破坏程序逻辑。子类不能削弱父类约定，不能随便重写破坏原有行为。

    4. 接口隔离原则（ISP）

      接口要小而专，不要大而全。避免客户端被迫实现它用不到的方法。

    5. 依赖倒置原则（DIP）

      高层模块不依赖低层模块，二者都依赖抽象；抽象不依赖细节，细节依赖抽象。简单说：面向接口 / 抽象编程，不直接依赖具体实现。

    6. 迪米特法则（LoD，最少知识原则）

      一个对象只和直接朋友通信，不和陌生人说话。减少类之间的依赖耦合，只调用必要的方法。

  4. 重载vs重写：重载→同类多方法；重写→子类修改父类方法

  5. 抽象类vs接口

    1. 抽象类：单继承，可含成员变量/构造器/普通方法，不可实例化
    2. 接口：多实现，仅常量（Java8+支持默认/静态/私有方法），无构造器

  6. static成员：归属类、全局共享、无需实例；静态方法不能直接访问非静态成员

  7. 内部类：非静态内部类依赖外部实例，可直接访问外部成员；静态内部类无依赖，仅能访问外部静态成员

五、关键字

  1. final：类不可继承、方法不可重写、变量不可二次赋值（引用不可变，对象内容可变）
  2. static：修饰变量/方法/代码块/内部类，实现类级共享与全局调用

六、深拷贝&浅拷贝

  1. 浅拷贝：仅复制对象本身，引用类型字段共享同一对象
  2. 深拷贝：递归复制所有引用对象，新旧对象完全独立
  3. 实现方式：重写clone()、序列化反序列化、手动递归复制
    1. 重写clone()：Object 自带的 native 方法，子类重写并实现Cloneable接口，调用super.clone()完成拷贝。默认浅拷贝，深拷贝需手动在clone()里把所有引用类型也逐个 clone；
    2. 序列化反序列化：把对象转成二进制流（序列化），再从流还原成新对象（反序列化）。天然深拷贝。
    3. 手动递归复制：自己写构造器 / 工具方法，遍历对象所有字段，基本类型直接赋值，引用类型递归 new 新对象复制。

七、泛型

  1. 作用：编译期类型校验，避免类型转换异常，实现代码复用
  2. 约束：泛型仅支持引用类型，不支持基本类型

八、对象操作

  1. 创建方式：new关键字、反射、clone()、反序列化、工厂模式

  2. 垃圾回收：GC通过可达性分析（GC Roots判定）回收无引用对象

    • GC（垃圾回收）：GC 从 GC Roots（虚拟机栈引用（栈帧中的局部变量表）、静态变量（类变量）、常量（运行时常量池）、本地方法栈引用、JNI内部和全局引用、被同步锁持有的对象 等）出发，沿着对象引用链遍历，能遍历到的对象标记为存活，遍历不到的判定为垃圾，可回收。
      • 能被遍历到 = 可达对象 → 存活，不回收
      • 遍历不到 = 不可达对象 → 无引用 / 用不到，判定为垃圾，回收
    • GC解决了简单引用计数法的循环引用无法回收问题（GC中：A 引用 B、B 引用 A，无外部引用时，依然会被判定为不可达）

  3. 私有成员访问：对外提供getter方法；或通过反射暴力访问

九、反射

  1. 定义：运行时获取类结构，动态创建对象、调用方法、修改属性

    • 反射获取Class对象的三种方式：

      • Class.forName (“全限定类名”)：运行时通过类的完整包名 + 类名（字符串）加载类并获取 Class 对象，会触发类的初始化（执行静态代码块、静态变量赋值）。
      • 类名.class：编译期就能获取，通过类名直接获取 Class 对象，不会触发类的初始化，只有类第一次被实例化时才会完成初始化。
      • 对象.getClass ()：运行时通过已实例化的对象获取 Class 对象，继承自 Object 类，是获取实例运行时真实类型的常用方式。

    • 反射的四大核心能力：

      • 运行时获取类的完整结构：

        • getXxx()：获取当前类及父类中所有 public 修饰的成员（如 getMethods ()、getFields ()、getConstructors ()）
        • getDeclaredXxx()：获取当前类自身声明的所有成员，包括 private/protected/default 修饰的，无视访问权限，但不包含父类继承的成员（如 getDeclaredMethods ()、getDeclaredFields ()、getDeclaredConstructors ()）

      • 动态创建对象：

        • 通过构造器实例化（推荐，JDK9 + 官方唯一推荐方式）：可以调用任意修饰符的构造器，支持带参构造。
        • Class.newInstance ()（已过时）：只能调用 public 无参构造器，若构造器不存在或无访问权限，直接抛出异常，JDK9 后已标记为过时。

      • 动态调用方法：运行时动态调用对象的任意方法，无视访问权限，核心 API 是Method.invoke()

        • // 1. 获取Class对象与实例
Class<?> userClazz = Class.forName("com.example.User");
Object userInstance = userClazz.getDeclaredConstructor().newInstance();
// 2. 获取目标方法：方法名+参数类型（区分重载）
Method setNameMethod = userClazz.getDeclaredMethod("setName", String.class);
// 3. 关闭权限检查，可调用private方法
setNameMethod.setAccessible(true);
// 4. 执行方法调用：实例 + 方法入参
setNameMethod.invoke(userInstance, "meisijiya");

  2. 应用：JDBC驱动加载、Spring IOC、配置文件动态实例化Bean

    1. JDBC 是 Java 官方定义的数据库访问规范（一套接口）,各厂商提供对应的驱动实现类。为了实现解耦，只需要在运行时通过字符串传入驱动类的全限定名，来完成加载。Class.forName()会触发驱动类的初始化，执行驱动类中的静态代码块。
    2. Spring 容器启动执行 refresh () 方法，先扫描解析各类配置封装为 BeanDefinition，通过专属类加载器获取 Bean 的 Class 反射元数据，经构造器推断后反射实例化原始 Bean，并将其 ObjectFactory 存入三级缓存提前暴露以解决循环依赖，再执行依赖注入（依赖未就绪时递归完成其全生命周期），随后按序执行 Aware 接口回调、BeanPostProcessor 前后置处理、自定义初始化逻辑（AOP 动态代理在此环节生成），最终将完全就绪的成熟 Bean 存入一级单例池全局复用，容器关闭时执行对应销毁逻辑完成 Bean 生命周期闭环。

    以下为 Spring 默认singleton单例、lazy-init=false关闭懒加载的 Bean 标准详细流程。

    1. 步骤 1：容器启动，解析配置，封装 BeanDefinition

      • Spring 容器启动时执行核心refresh()方法，先扫描 xml 配置、类注解（@Component/@Service/@Controller等）、JavaConfig 配置，解析出所有需容器管理的 Bean 的全限定类名、作用域、依赖关系、初始化 / 销毁方法、懒加载属性等全量元数据，封装成一个个BeanDefinition对象，注册到BeanDefinitionMap中，完成 Bean 元数据的预处理与校验。

    2. 步骤 2：类加载，获取 Bean 的 Class 反射元数据

      • Spring 通过自身的BeanClassLoader类加载器，调用ClassUtils.forName(全限定类名, classLoader)完成 Bean 的类加载（**非直接调用 JDK 原生Class.forName()**），提前获取并缓存 Bean 的 Class 对象，解析出类的构造器、方法、字段、注解等全量反射元信息，为后续实例化做准备，同时兼容 Tomcat、SpringBoot 等环境的类加载器隔离机制。

    3. 步骤 3：反射实例化，创建 Bean 原始对象

      • Spring 根据BeanDefinition信息，通过构造器推断逻辑筛选出合适的构造器（优先级：@Autowired标记的构造器 > 无参构造器 > 唯一匹配的带参构造器），通过反射获取对应的Constructor对象；若构造器非public修饰，会调用constructor.setAccessible(true)关闭权限检查，最终调用constructor.newInstance(入参)完成 Bean 实例化，创建出仅分配内存、属性未注入、未执行初始化的原始 Bean 对象。

      细节必背：Spring 全程不使用 JDK9 已废弃的Class.newInstance()，该方法仅支持 public 无参构造，灵活性与安全性均不满足框架要求。

    4. 步骤 4：提前暴露对象，放入三级缓存（循环依赖核心设计）

      • 实例化完成后，Spring 会立即将该原始 Bean 的ObjectFactory工厂对象，添加到 三级缓存singletonFactories 中，提前暴露对象引用，用于解决属性填充阶段的循环依赖问题；此时绝对不会将原始对象放入单例池，避免多线程环境下拿到未初始化的不完整 Bean，引发空指针等致命问题。

        三级缓存:

        • 核心定义

          Spring 内置 3 个线程安全 Map，核心作用：解决单例 Bean setter / 字段注入的循环依赖，兼容 AOP 逻辑且不破坏 Bean 原生生命周期。 固定查找优先级：一级缓存 → 二级缓存 → 三级缓存

          三级缓存核心信息

          层级 官方名称 存储内容 核心作用
          一级（单例池） singletonObjects 全流程完成的成熟可用 Bean 全局复用，对外提供 Bean 的唯一出口
          二级 earlySingletonObjects 仅实例化的早期半成品 Bean（含 AOP 早期代理） 避免循环依赖中重复创建早期对象
          三级 singletonFactories Bean 对应的 ObjectFactory 工厂 延迟生成 AOP 早期代理，兼容循环依赖场景

        • 核心工作顺序

          1. 反射实例化 Bean 生成原始对象，向三级缓存存入 ObjectFactory 提前暴露
          2. 执行 DI 依赖注入，触发循环依赖时按固定优先级逐级查找
          3. 命中三级缓存后，生成早期对象移入二级缓存，清空对应三级缓存记录
          4. 完成 DI 与 Bean 全流程初始化，成熟 Bean 存入一级缓存，清空二、三级缓存对应记录

        • 必记易错点

          1. 仅单例 Bean 支持该机制，原型 Bean、构造器注入的循环依赖无法解决
          2. 三级缓存核心价值是延迟生成 AOP 早期代理，无 AOP 场景两级缓存即可解决基础循环依赖
          3. 二、三级缓存仅为临时中转载体，最终仅一级缓存永久留存成品 Bean

    5. 属性填充（依赖注入 DI，反射核心执行环节）

      • Spring 根据BeanDefinition解析的依赖信息，执行populateBean方法完成依赖注入，三种注入方式的反射实现细节如下：

        1. 构造器注入：已在实例化阶段完成，无需二次处理
        2. Setter 方法注入（Spring 原生默认方式）：解析 Bean 的 setter 方法，通过反射获取对应的Method对象，调用method.invoke(当前Bean实例, 已就绪的依赖Bean对象)完成注入
        3. 字段注入（@Autowired字段注解常用方式）：解析标记了注入注解的字段，通过反射获取对应的Field对象，先调用field.setAccessible(true)关闭权限检查，再调用field.set(当前Bean实例, 已就绪的依赖Bean对象)完成注入

        细节必背：若依赖的 Bean 尚未创建，会递归触发该依赖 Bean 的完整实例化→依赖注入（DI）→初始化流程，确保注入的是可正常使用的 Bean 对象。

    6. Bean 初始化全流程执行

      • 属性填充完成后，执行initializeBean方法完成 Bean 的完整初始化，严格按以下顺序执行：

        1. Aware 接口回调：执行BeanNameAware、BeanFactoryAware、ApplicationContextAware等 Aware 接口的回调方法，向 Bean 注入容器相关资源

          Aware 接口回调，是 Spring 在 Bean 属性填充完成后、初始化执行前，对实现了对应 Aware 子接口的 Bean，自动调用接口规定的 set 方法注入容器核心资源，使 Bean 能够主动感知并获取 Spring 容器的相关环境与元数据。

        2. BeanPostProcessor 前置处理：遍历所有注册的BeanPostProcessor实现类，执行postProcessBeforeInitialization方法，可对 Bean 实例进行定制化修改

        3. 自定义初始化逻辑执行：先执行InitializingBean接口的afterPropertiesSet方法，再执行BeanDefinition中指定的自定义init-method方法

        4. BeanPostProcessor 后置处理：遍历所有注册的BeanPostProcessor实现类，执行postProcessAfterInitialization方法，Spring AOP 的动态代理生成，就在此环节完成

    7. 放入单例池，完成生命周期注册

      • 初始化完成后，生成完全装配完毕、可正常使用的成熟 Bean 实例，Spring 会将该实例放入 一级缓存singletonObjects（即单例池） 中，同时从二级缓存earlySingletonObjects、三级缓存singletonFactories中移除该 Bean 的相关记录。
      • 细节必背：单例池本质是一个ConcurrentHashMap，key 为 beanName，value 为成熟的 Bean 实例，线程安全、全局唯一，后续所有获取该 Bean 的请求，都直接从单例池返回复用，不会重复创建。

    8. 容器关闭，执行 Bean 销毁逻辑

      • 当 Spring 容器关闭时，会遍历单例池中的所有 Bean，先执行DisposableBean接口的destroy方法，再执行自定义的destroy-method方法，完成 Bean 的资源释放与销毁。

十、注解

  1. 本质：注解都是继承了Annotation接口的特殊接口，给 Java 代码（类、方法、字段、参数等）附加元数据标记，运行时通过动态代理解析

  2. 作用域：由@Target指定（类、字段、方法、参数等）

    • 全量ElementType枚举值详解

      枚举值 可标注的位置 典型使用场景
      TYPE 类、接口、枚举、注解类型 Spring 的@Controller、@Service、@Component
      FIELD 成员字段（包括枚举常量） Spring 的@Autowired、@Value、MyBatis 的@TableId
      METHOD 成员方法 Spring 的@RequestMapping、@GetMapping、@PostConstruct
      PARAMETER 方法入参 Spring 的@RequestParam、@PathVariable、@RequestBody
      CONSTRUCTOR 类的构造方法 Spring 的@Autowired构造器注入
      LOCAL_VARIABLE 局部变量 很少使用，如@SuppressWarnings局部变量抑制警告
      ANNOTATION_TYPE 注解本身（元注解专用） @Target、@Retention自身都标注了该值
      PACKAGE 包名 极少使用，用于包级别的元数据声明
      TYPE_PARAMETER 泛型类型参数声明（JDK8+） 比如public class Test<T extends @MyAnno String>
      TYPE_USE 任意类型使用处（JDK8+） 几乎所有类型场景，如变量声明、强制类型转换、泛型参数等

  3. 保留策略：源码级/类文件级/运行时级（仅运行时可被反射解析）

十一、异常

  1. 体系：Throwable → Error（虚拟机错误）、Exception（程序异常）

  2. 分类：受检异常（Java 编译器强制校验的异常，编译期必须处理）、运行时异常（RuntimeException及其子类，非受检，逻辑错误，仅在程序运行时触发）

    • 常见受检异常类：IOException、FileNotFoundException、SQLException、ClassNotFoundException、InterruptedException
    • 常见运行时异常类：NullPointerException(NPE)、ArrayIndexOutOfBoundsException、ClassCastException、ArithmeticException、IllegalArgumentException

  3. 处理方式：try-catch-finally、throw（手动抛异常）、throws（方法声明异常）

    • 困惑的点，main方法中不try-catch，而是由main方法自己throws异常，那么到底谁来处理这个异常。基于谁调用谁处理原则，答案是最终由JVM接收异常并处理。
      JVM收到异常后的默认行为：
      1.中断当前线程（主线程）
      2.在控制台打印异常堆栈信息
      3.直接终止程序

  4. 执行规则：finally代码块必定执行；finally中return会覆盖try中的return

十二、Object类与字符串

  1. Object核心方法：equals、hashCode、toString、getClass、clone、wait/notify、finalize

    • equals：用于判断两个对象的 “内容” 是否逻辑相等，其重写必须满足的 5 个特性

      • 自反性：x.equals(x) 必须返回 true
      • 对称性：若 x.equals(y) 为 true，则 y.equals(x) 也必须为 true
      • 传递性：若 x.equals(y) 为 true 且 y.equals(z) 为 true，则 x.equals(z) 也必须为 true
      • 一致性：只要对象未修改，多次调用 equals() 结果必须一致
      • 非空性：x.equals(null) 必须返回 false

    • hashCode ()：对象的哈希码值

    • toString ()：对象的字符串表示

    • getClass ()：获取对象的运行时类

    • clone ()：对象的克隆（副本创建），默认实现为 “浅拷贝”

      • native 方法：由 JVM 底层实现，直接复制对象的内存块；
      • protected 修饰：子类必须重写为 public 才能被外部调用；
      • 异常要求：若类未实现 Cloneable 接口，调用 clone() 会抛出 CloneNotSupportedException。

    • wait () /notify () /notifyAll ()：线程通信的核心方法

      • wait()：让当前线程进入等待状态，释放对象的监视器锁；
      • notify()：随机唤醒一个正在该对象上等待的线程；
      • notifyAll()：唤醒所有正在该对象上等待的线程。

  2. == vs equals：==比较地址/基本类型值；equals默认比地址，重写后比内容

  3. hashCode&equals：相等对象hashCode必相同，反之不成立

  4. 字符串对比

    1. String：不可变，常量池复用对象
    2. StringBuilder：单线程，高效无锁
    3. StringBuffer：多线程，线程安全（synchronized）

十三、JDK 8、17、21 核心新特性总结（三大 LTS 版本）

三个版本均为 Oracle 官方长期支持（LTS） 版本，是 Java 开发的核心里程碑：

  • JDK 8（2014）：史上最经典、市占率最高的 LTS，开启 Java 函数式编程时代
  • JDK 17（2021）：企业级稳定 LTS，整合 9-16 成熟特性、清理废弃功能、GC 正式成熟
  • JDK 21（2023）：最新 LTS，云原生 / 高并发终极优化，解决 Java 并发核心痛点

一、JDK 8 核心新特性

1. 编程范式革命（最核心）

  • Lambda 表达式：简化匿名内部类，支持函数式编程
  • 函数式接口：@FunctionalInterface 约束，仅允许一个抽象方法
  • 方法引用：:: 语法，极致简化 Lambda 代码
  • 接口默认 / 静态方法：接口可直接实现方法，解决接口扩展兼容问题

2. 核心 API 重构

  • Stream API：流式集合操作（过滤、映射、聚合），告别繁琐 for 循环
  • **Optional**：优雅空指针处理，彻底消灭NullPointerException
  • 全新日期时间 API（java.time）：线程安全、API 简洁，取代老旧Date/Calendar

3. 并发与集合优化

  • CompletableFuture：异步编程神器，支持链式回调、组合异步任务
  • HashMap：链表长度≥8 转为红黑树，查询性能指数级提升
  • ConcurrentHashMap：锁机制优化，并发性能大幅增强

4. JVM 底层

  • 永久代（PermGen）移除：替换为元空间（Metaspace），使用本地内存，避免 OOM

二、JDK 17 核心新特性

JDK 17 是企业从 JDK8 迁移的首选版本，主打稳定、精简、高性能，整合了 JDK9-16 的所有成熟能力。

1. 语法糖（正式版，大幅简化代码）

  • Switch 表达式：支持返回值、省略 break，代码极简
  • 文本块（Text Blocks）：""" 原生支持多行字符串（JSON/XML/SQL）
  • 密封类（Sealed Classes）：sealed关键字限制类继承，提升代码安全性
  • instanceof 模式匹配（预览）：自动类型强转，告别冗余强转代码

2. 生态清理（减负）

  • JPMS 模块化系统：Java 平台模块化，解决类路径冲突，支持按需加载
  • 彻底废弃 / 移除：Applet、Security Manager、RMI 激活机制

3. 虚拟机 GC（重大突破）

  • ZGC 正式版：微秒级低延迟垃圾回收，支持 TB 级大堆内存
  • Shenandoah GC 正式版：无停顿 GC，低延迟业务首选
  • G1 GC 持续优化：默认回收器，吞吐量与延迟平衡

4. 架构支持

  • 原生支持 ARM64/AArch64（macOS M 系列、ARM 服务器）

三、JDK 21 核心新特性

最新 LTS，Java 云原生 / 高并发时代的标杆版本，解决了 Java 数十年的并发编程瓶颈。

1. 语言特性（正式版）

  • Switch 模式匹配：Switch + 类型匹配 + 空值处理，代码极致简洁
  • instanceof 模式匹配：正式转正，彻底告别手动强转
  • 记录模式（Record Patterns）：一键解构 Record 对象，简化数据解析

2. 并发编程革命（⭐核心特性）

  • 虚拟线程（Virtual Threads）：Project Loom 核心成果
    • 轻量级线程（可创建百万级并发），无额外开销
    • 同步代码实现高并发，彻底告别异步回调地狱
    • 兼容现有 API，零成本迁移

3. GC 终极优化

  • 分代 ZGC（Generational ZGC）正式版：兼顾微秒级停顿 + 分代回收，内存利用率拉满，云原生大内存场景首选

4. 集合与 API

  • 序列集合（Sequenced Collections）：统一有序集合接口，支持首尾快速操作
  • 废弃Finalization（对象终结器），推荐更安全的资源回收方案

5. 预览 / 孵化特性

  • 结构化并发：统一管理多线程生命周期
  • 作用域值：线程局部变量的安全替代方案
  • 向量 API：硬件加速的高性能数值计算

十四、其他高频考点

  1. 序列化：对象与字节流互转，实现Serializable接口，用于跨JVM传输
    • Java的Serializable是标记接口，作为Java的安全机制仅允许实现该接口的对象被JVM执行序列化与反序列化操作（实现对象与字节流的双向转换），以此支撑对象的跨JVM传输，使用时需注意通过serialVersionUID保证版本兼容，用transient关键字排除敏感字段。
  2. 设计模式：单例（volatile+synchronized）、代理、适配器、责任链、策略模式
    • 单例模式：核心是构造器私有化，双重校验锁必须加 volatile 防止指令重排，枚举单例是最安全的实现；
    • 代理模式：核心是无侵入式增强，Spring AOP 底层是 JDK 动态代理（基于接口）和 CGLIB 动态代理（基于继承）；
    • 适配器模式：核心是接口兼容转换，解决不兼容问题，Spring MVC 的 HandlerAdapter 是经典实现；
    • 责任链模式：核心是请求沿链传递，解耦处理逻辑，Servlet Filter、Spring 拦截器是最常用的场景；
    • 策略模式：核心是封装可互换的算法，消除 if-else 分支，符合开闭原则，是业务开发中最常用的优化手段。
  3. IO模型：BIO(同步阻塞)、NIO(同步非阻塞)、AIO(异步非阻塞)，Netty基于NIO
  4. 排序：通过Comparable/Comparator实现多条件排序
    • 排序的核心依赖 Comparable 与 Comparator 两个接口：单条件排序时，若为类的固有唯一排序规则，优先让类实现 Comparable 并重写 compareTo，使用 Integer.compare/Double.compare 等包装类静态方法避免手动减法的整数溢出；若需灵活切换规则或无法修改业务类代码，则使用 Comparator，JDK8+ 推荐通过 Comparator.comparingInt/comparing 结合方法引用的简洁写法。多条件排序的核心是 “前序条件相等时再触发后续比较”，JDK8+ 最佳实践为通过 thenComparing/thenComparingInt 链式组合多个比较器，逻辑清晰易维护，同时可搭配 Comparator.nullsFirst/nullsLast 处理空值场景，且需保证比较结果与 equals 逻辑一致，避免基于比较的集合出现异常。
  5. Native方法：调用非Java代码，实现Java与操作系统交互
    • JNI（Java Native Interface）是 Java 官方定义的标准化跨语言交互接口，是 Java 与 C/C++ 等非 Java 本地代码交互的核心桥梁，专门支撑由native关键字修饰、无 Java 方法体的 Native 方法，实现 Java 对操作系统底层能力、本地高性能原生库的调用；其完整调用流程为：先在 Java 类中声明 Native 方法并编写加载本地动态库的逻辑，通过javac编译 Java 类生成字节码的同时导出符合 JNI 规范的头文件，再由 C/C++ 代码实现头文件中约定的、符合 JNI 命名规范的本地函数，编译生成对应操作系统平台的动态链接库（Windows 为.dll、Linux 为.so、Mac 为.dylib），Java 程序运行时 JVM 会加载该动态库，调用 Native 方法时通过 JNI 环境指针完成 Java 与本地代码的类型映射、内存交互与方法执行，最终实现 Java 与底层系统、原生代码的双向交互。
