一、Spring 6 基础概述与快速入门

1.1 Spring 6 核心定义与基础规范

  • Spring核心定位：Java EE 轻量级开源框架， Java 企业级开发框架，核心是IoC 容器和AOP，还提供事务管理、SpringMVC 等能力，可降低代码耦合，简化企业应用开发。

    IoC：控制反转，由 Spring 容器创建、管理 Bean 生命周期和依赖，替代手动 new 对象，解耦组件。

    AOP：面向切面编程，封装日志、事务等横切逻辑，不侵入业务代码，基于动态代理实现。

  • 版本硬性要求：Spring 6 最低要求 JDK 17，从 Spring Framework 6.0 开始，已全面升级至 Jakarta EE 9+，核心包名由javax全面改为jakarta。

  • 广义与狭义区分

    • 狭义：特指 Spring Framework，即 Spring 核心基础框架，是分层的、面向切面的一站式轻量级企业级开发解决方案。
    • 广义：完整 Spring 技术栈，包含 Spring Framework、Spring Boot、Spring Cloud、Spring Data、Spring Security 等全系列生态组件。

1.2 Spring 6 核心模块与功能

核心模块分类 包含子包与核心功能
Spring Core（核心容器） 提供 IoC、DI、Bean 配置与装载的核心实现，核心子包：spring-core（IoC/DI 底层基础）、spring-beans（BeanFactory 与 Bean 装配）、spring-context（ApplicationContext 容器）、spring-expression（SpEL 表达式）
Spring AOP 面向切面编程核心实现，核心子包：spring-aop（整合 ASM、CGLib、JDK Proxy）、spring-aspects（集成 AspectJ）、spring-instrument（动态类加载）
Spring Data Access 数据访问与事务支持，核心子包：spring-jdbc（JDBC 极简封装）、spring-orm（ORM 框架整合）、spring-oxm（对象 - XML 映射）、spring-jms（消息服务）、spring-tx（声明式事务管理）
Spring Web Web 应用开发支持，核心子包：spring-web（基础 Web 能力）、spring-webmvc（MVC 框架实现）、spring-websocket（全双工通信）、spring-webflux（响应式 Web 框架，Spring 5.0 + 新增）
Spring Message 消息传递核心能力（spring-messaging，Spring 4.0 + 新增）
Spring Test 集成测试支持（spring-test，核心封装 JUnit 测试框架）

1.3 Spring 6 核心特性

  • 非侵入式：对应用程序结构影响极小，领域模型零污染，业务组件仅需少量注解即可接入容器。
  • 控制反转（IoC）：翻转资源获取方向，由容器主动注入依赖，彻底解耦对象间的依赖关系。
  • 面向切面（AOP）：分离业务核心逻辑与系统级服务（日志、事务、安全），实现组件高内聚低耦合。
  • 容器化管理：统一管理 Bean 的配置与全生命周期，屏蔽对象创建与销毁的底层细节，降低开发门槛。
  • 组件化拼装：通过 XML 或注解即可组合简单组件，搭建复杂企业级应用系统。
  • 一站式解决方案：在 IoC 和 AOP 基础上，无缝整合各类开源框架与第三方类库，无需额外适配。

1.4 Spring 6 快速入门

1.4.1 环境与依赖要求

  • 基础环境：JDK 17+、Maven 3.6+、Spring Framework 6.0.2+
  • 核心依赖：spring-context（引入后自动包含 Spring 基础核心依赖，无需额外引入零散包）
  • 可选测试依赖：JUnit 5（junit-jupiter-api）

1.4.2 基础开发流程

  1. 创建 Maven 模块
  2. 在pom.xml中添加上述核心依赖
  3. 编写业务 Bean 类（如HelloWorld）
  4. 在resources/目录下创建beans.xml配置文件，完成 Bean 的声明配置
  5. 通过ApplicationContext context = new ClassPathXmlApplicationContext("beans.xml");获取 Spring 容器
  6. 通过context.getBean("helloWorld")获取 Bean 实例，调用对应业务方法

1.4.3 入门级底层核心原理

  • 对象创建机制：Spring 底层通过反射机制调用 Bean 的无参构造器完成对象实例化。

    反射：运行时获取类信息、动态创建对象 （clazz.getDeclaredConstructor().newInstance()）/ 调用方法（Method类的invoke()）、访问和修改字段值（Field类的get()和set()）
    反射使用场景：Spring IoC 实例化 Bean、AOP 动态代理、框架配置解析。

  • 容器存储结构：Bean 最终存储在 Spring 容器中，底层是DefaultListableBeanFactory类中的ConcurrentHashMap(256)，Key 通常为类名首字母小写，Value 为BeanDefinition（Bean 的全量定义信息对象）。

  • Bean 加载流程：容器启动时加载 Bean 配置，将 Bean 的全量描述信息存入beanDefinitionMap，后续按需完成实例化与初始化。

1.5 Log4j2 日志框架集成

1.5.1 日志级别（优先级从低到高）

  • TRACE：追踪程序全流程执行细节
  • DEBUG：开发调试阶段最低级别，输出调试信息
  • INFO：业务核心运行信息，生产环境默认级别
  • WARN：程序警告，不影响正常运行但需关注
  • ERROR：程序运行错误，部分功能异常
  • FATAL：严重系统错误，会导致程序崩溃退出

1.5.2 完整集成步骤

  1. 引入依赖：需同时添加log4j-core（日志核心实现）和log4j-slf4j2-impl（SLF4J 桥接实现）依赖
  2. 配置文件：在resources/目录下创建log4j2.xml核心配置文件，配置三大核心模块：Loggers（日志记录器）、Appenders（输出目的地）、Layouts（输出格式）
  3. 滚动日志配置：可通过RollingFile实现按文件大小或时间滚动归档，例如配置SizeBasedTriggeringPolicy size="50MB"和DefaultRolloverStrategy max="20"，避免单日志文件过大。

1.5.3 核心输出形式与 Spring 桥接机制

  • 支持三种核心输出形式：控制台输出（Console）、文件输出（File）、滚动日志归档（RollingFile）
  • Spring 框架日志桥接：从 Spring 5.0 开始，内置spring-jcl模块，优先选用 Log4j 2.x 或 SLF4J 实现，无对应实现时回退至 JUL（java.util.logging）；应用代码建议直接使用 Log4j 2.x、SLF4J 或 JUL 标准 API。

二、Spring 核心原理深度解析

2.1 核心重点：IoC 控制反转容器

2.1.1 核心概念

  • IoC（控制反转）：一种核心设计思想，将对象的创建、依赖管理、生命周期控制权，从业务代码完全交给 Spring 容器，从根本上实现代码的松耦合。

    实现机制：反射 + 工厂模式 + 依赖注入，通过 BeanFactory/ApplicationContext 管理 Bean。

    IoC设计：Bean 生命周期管理、依赖注入实现、Bean 作用域、AOP 支持、异常处理、配置加载。

  • DI（依赖注入）：IoC 思想的具体实现方式，在运行时由容器动态将依赖对象注入到程序中，彻底解耦对象间的依赖关系。

  • DI 三种实现方式与优劣对比

    • 构造器注入：通过类的构造函数传递依赖，保证 Bean 初始化时依赖完全就绪，Spring 官方推荐首选
    • Setter 注入：通过 Setter 方法设置依赖，灵活性高，支持可选依赖与循环注入，但依赖可能未完全初始化
    • 字段注入：通过@Autowired直接注解字段，写法最简洁，但会隐藏依赖关系，不推荐生产环境使用

2.1.2 IoC 容器核心接口

  • BeanFactory：Spring 最底层核心接口，面向 Spring 框架本身，提供最基础的 IoC 能力（如核心的getBean()方法），启动时不会实例化 Bean，首次获取时才实例化。
  • ApplicationContext：BeanFactory的子接口，面向应用开发者，在基础 IoC 能力之上，提供 AOP 集成、国际化、事件驱动、资源加载等高级特性，容器启动时默认完成所有单例 Bean 的实例化与初始化。
    • 常用实现类：
      • ClassPathXmlApplicationContext：从类路径加载 XML 配置文件
      • FileSystemXmlApplicationContext：从文件系统绝对路径加载 XML 配置文件
      • AnnotationConfigApplicationContext：基于注解配置类加载，全注解开发核心容器

2.1.3 Bean 全生命周期管理

Bean 加载 / 销毁前后执行逻辑的方式：

  • @PostConstruct（初始化）、@PreDestroy（销毁）
  • 实现InitializingBean/DisposableBean接口
  • XML/@Bean指定 init-method/destroy-method

  1. 实例化（Instantiation）

    容器通过反射调用 Bean 的构造器（或无参构造器）创建对象实例。此时仅分配内存空间，尚未进行属性赋值。

  2. 属性赋值（Populate）

    Spring 将配置文件中定义的值或通过 @Autowired 注入的其他 Bean，填充到当前 Bean 的对应属性中（依赖注入）。

  3. Aware 接口回调

    若 Bean 实现了特定的 Aware 接口，Spring 会回调对应的方法，按优先级顺序执行：

    • 实现 BeanNameAware：调用 setBeanName()，传入 Bean 的 ID。
    • 实现 BeanFactoryAware：调用 setBeanFactory()，传入 BeanFactory 容器实例。

  4. BeanPostProcessor 前置处理

    若 Bean 实现了 BeanPostProcessor 接口（或其子类），Spring 调用 postProcessBeforeInitialization() 方法。

    • 注：ApplicationContextAware 接口的 setApplicationContext() 方法，正是由 ApplicationContextAwareProcessor 在此阶段触发。

  5. 初始化（Initialization）

    执行 Bean 的初始化逻辑，顺序如下：

    • 若实现 InitializingBean 接口：调用 afterPropertiesSet() 方法。
    • 若配置了自定义初始化方法（如 init-method 或 @PostConstruct）：调用指定的方法。

  6. BeanPostProcessor 后置处理

    再次调用 BeanPostProcessor 的 postProcessAfterInitialization() 方法。

    • 注：AOP 代理对象的生成（如 JDK 动态代理或 CGLIB）通常发生在这一阶段。

  7. Bean 就绪

    此时 Bean 已完成所有生命周期回调，处于可用状态，驻留在应用上下文中等待调用。

  8. 销毁（Destruction）

    当容器关闭时，若 Bean 实现了 DisposableBean 接口或定义了 destroy-method（或 @PreDestroy），Spring 会调用相应的销毁方法释放资源。

关键注意：prototype多例作用域的 Bean，Spring 容器仅负责创建，不管理其完整生命周期，不会执行销毁阶段的方法。

2.1.4 Bean 核心作用域

  • singleton（默认）：单例模式，容器启动时创建一次，全局共享一个实例
  • prototype：多例模式，每次从容器获取 Bean 时，都会创建一个全新的实例，可通过@Scope("prototype")设为多例。
  • Web 环境专属作用域：request（单次请求）、session（单次会话）、application（全局 ServletContext）、websocket（单次 WebSocket 连接）
  • 支持自定义作用域：通过实现Scope接口完成自定义作用域开发

2.1.5 Bean 管理方式

（1）XML 管理 Bean（了解）
  • Bean 获取方式：按 ID 获取（getBean("id")）、按类型获取（需容器中该类型 Bean 唯一）、按 ID + 类型精准获取
  • 核心依赖注入实现：
    • Setter 注入：<property name="属性名" value/ref="属性值/Bean引用"/>
    • 构造器注入：<constructor-arg index/name="参数索引/名称" value/ref="属性值/Bean引用"/>
  • 支持注入的数值类型：字面量、null 值、XML 特殊字符（CDATA 节）、外部 Bean / 内部 Bean、级联属性、数组 / List/Map/Set 集合类型、外部属性文件（${key}取值）
  • 自动装配：支持byType（按类型匹配）、byName（按属性名匹配 Bean ID）两种模式
  • 特殊 Bean：FactoryBean，用于封装复杂对象的创建逻辑，整合第三方框架，通过getObject()方法返回实际业务对象。
（2）注解管理 Bean（重点，生产环境首选）
  • 组件标记注解（功能完全一致，仅做分层语义区分）：
    • @Component：通用组件标记，无明确分层的通用 Bean 使用
    • @Repository：数据访问层（DAO）组件，支持数据库异常转译（转为 Spring 统一的DataAccessException）
    • @Service：业务逻辑层组件
    • @Controller：Web 控制层组件（Spring MVC）
  • 开启组件扫描：
    • XML 方式：<context:component-scan base-package="com.example.xxx"/>，支持通过exclude-filter/include-filter排除 / 包含指定组件
    • 注解方式：在配置类上添加@ComponentScan("com.example.xxx")，全注解开发使用
  • 核心依赖注入注解：
    • @Autowired（Spring 原生注解）：默认按类型注入，可标注在构造器、方法、参数、属性上；通过required属性设置依赖是否必选；配合@Qualifier("beanName")可实现按名称精准注入
    • @Resource（JSR-250 标准注解）：默认按名称匹配，找不到对应 Bean 时回退到按类型匹配，跨框架通用性更强；JDK 11+ 或 JDK 8 以下需额外引入jakarta.annotation-api依赖
  • 全注解开发模式：使用@Configuration标记配置类，替代 XML 配置文件；结合@ComponentScan指定扫描包路径；通过AnnotationConfigApplicationContext加载配置类启动容器。

2.2 核心重点：AOP 面向切面编程

2.2.1 基本概念

  • 核心定义：在不修改业务源代码的前提下，动态为业务方法统一添加额外功能（日志、事务、权限校验等），实现业务核心逻辑与横切关注点的彻底分离， 解决横切逻辑冗余问题（日志、事务、权限），避免代码侵入业务，补充面向对象编程的不足。

  • 设计目标：实现代码复用、逻辑模块化、降低系统耦合度、提升代码可维护性与可扩展性。

  • 实现机制与边界：动态代理，实现接口用 JDK 动态代理，无接口用 CGLIB 代理；仅作用于 Spring 容器 Bean；同类 this 调用不生效，仅支持方法增强；核心是无侵入封装横切逻辑。

    静态代理：编译期确定代理类，仅代理单个类，代码冗余。

    动态代理：运行期生成代理类，可代理接口下多个实现类，灵活无冗余。

  • AOP在Spring的应用：声明式事务（@Transactional）、日志记录、权限校验、统一异常处理。

2.2.2 AOP 核心术语详解

  • 横切关注点：跨越应用多个模块的通用功能，如日志、事务、安全校验、性能监控等。
  • 通知（Advice）：切面的增强逻辑，即要执行的额外代码，Spring AOP 支持 5 种通知类型：
    • 前置通知（@Before）：目标方法执行前执行
    • 返回通知（@AfterReturning）：目标方法成功执行并返回后执行
    • 异常通知（@AfterThrowing）：目标方法抛出异常后执行
    • 后置通知（@After）：目标方法最终结束后执行，无论方法正常返回还是抛出异常都会执行
    • 环绕通知（@Around）：包围整个目标方法，可控制目标方法是否执行、何时执行，可自定义执行前后逻辑，功能最强大
  • 切面（Aspect）：仅作为一个概念；通知与切入点的结合，封装增强逻辑的类，即切面类；特别的AspectJ是一个独立的AOP框架，SpringAOP只是借用了它的注解（如@Aspect）
  • 目标对象（Target）：被代理增强的原始业务对象。
  • 代理对象（Proxy）：应用通知增强后，AOP 框架动态创建的对象，用于控制对目标对象的访问。Spring AOP 中有两种代理，分别是JDK动态代理和CGLIB动态代理。
  • 连接点（Join Point）：程序执行过程中可插入切面的点，Spring AOP 仅支持方法级别的连接点。
  • 切入点（Pointcut）：匹配连接点的表达式，用于定义哪些方法需要被增强，哪些方法不需要。在 Spring AOP 中，仅支持方法级别的连接点。

2.2.3 AOP 底层代理机制

Spring AOP 基于动态代理实现，支持两种动态代理模式，默认根据目标类自动选择：

  • JDK 动态代理：基于接口实现，使用java.lang.reflect.Proxy和InvocationHandler实现，要求目标类必须实现至少一个接口，生成接口的实现类作为代理对象。
  • CGLIB 动态代理：CGLIB（Code Generation Library）是一个第三方代码生成库，基于继承实现，生成目标类的子类作为代理对象，适用于无接口的类，无法代理 final 修饰的类和方法。
  • Spring AOP 默认策略：若目标类实现了接口，优先使用 JDK 动态代理；若目标类未实现任何接口，自动切换为 CGLIB 动态代理。

2.2.4 AOP 实现方式

（1）注解方式实现 AOP（重点，生产环境首选）

完整实现步骤：

  1. 引入依赖：spring-context已包含 AOP 核心依赖，无需额外引入
  2. 定义切面类：类上添加@Aspect + @Component注解，标记为切面类并交给 Spring 容器管理
  3. 配置通知与切入点：在切面方法上添加对应通知注解（@Before、@Around等），通过切入点表达式指定要增强的方法
  4. 开启 AOP 自动代理：XML 配置<aop:aspectj-autoproxy />，或在配置类上添加@EnableAspectJAutoProxy注解
  • 核心切入点表达式语法：execution([权限修饰符] 返回值类型 包名.类名.方法名(参数列表))，支持通配符*（匹配任意字符）、..（匹配任意层级包 / 任意数量参数）
  • 通知信息获取：
    • 连接点信息：在通知方法参数中添加JoinPoint对象，可获取目标方法、参数、类名等信息
    • 返回值：在@AfterReturning中通过returning="result"参数接收方法返回值
    • 异常信息：在@AfterThrowing中通过throwing="ex"参数接收方法抛出的异常
（2）XML 方式实现 AOP（了解）

通过<aop:config>、<aop:aspect>、<aop:pointcut>、<aop:before>等标签，在 XML 配置文件中完成切面、切入点、通知的全量配置，无需注解。

三、Spring 全家桶生态体系

3.1 Spring MVC

3.1.1 MVC 分层架构

  • Model（模型）：负责业务数据承载、业务逻辑处理，封装核心业务规则
  • View（视图）：负责数据的页面渲染与展示，将模型数据呈现给用户
  • Controller（控制器）：负责请求接收与调度，承接用户请求，调用对应模型处理数据，最终匹配视图完成响应

3.1.2 核心执行流程

用户请求
↓
DispatcherServlet（前端控制器，统一接收所有请求）
↓
HandlerMapping（处理器映射器，根据 URL 映射到对应 Controller，生成处理器执行链HandlerExecutionChain(包括处理器对象和处理器拦截器)）
↓
HandlerExecutionChain给到DispatcherServlet
↓
HandlerAdapter（处理器适配器，适配不同类型 Controller）
↓
Controller（业务处理器，处理核心业务逻辑）
↓
返回ModelAndView给到HandlerAdapter（封装模型数据与视图名）
↓
ViewResolver（视图解析器，解析视图名得到对应视图）
↓
返回View给到HandlerAdapter（视图渲染，将模型数据填充到视图）
↓
HandlerAdapter把 响应结果返回给用户

3.1.3 核心组件

  • DispatcherServlet：前端控制器，整个 Spring MVC 流程的核心调度中心，统一处理所有请求与响应
  • HandlerMapping：负责请求 URL 与 Controller 处理方法的映射匹配，找到对应请求的处理器
  • HandlerAdapter：按照特定规则适配执行 Controller 中的处理方法，屏蔽不同处理器的执行差异
  • ViewResolver：负责将视图名解析为具体的视图对象，完成视图定位

3.1.4 过滤器（Filter） vs 拦截器（Interceptor）

特性 过滤器（Filter） 拦截器（Interceptor）
规范归属 Servlet 原生规范 Spring MVC 组件
拦截范围 所有进入容器的请求 仅拦截 Controller 层的请求
Spring 依赖 不依赖 Spring 容器，无法注入 Spring Bean 由 Spring 容器管理，支持依赖注入
执行时机 请求进入 Servlet 之前、响应离开 Servlet 之后 请求到达 Controller 之前、Controller 执行之后、视图渲染之后

3.2 Spring Boot

3.2.1 核心优势

  • 约定大于配置：提供大量默认配置，大幅减少开发者的配置工作量
  • 自动装配：根据引入的依赖自动完成框架配置，无需手动编写配置文件
  • Starter 起步依赖：封装各类场景的依赖包，一站式解决依赖冲突与版本管理问题
  • 内嵌 Web 容器：内置 Tomcat、Jetty、Undertow 等 Web 容器，无需单独部署 WAR 包
  • 自带监控能力：提供 Actuator 组件，支持应用健康检查、指标监控、运行状态查看

3.2.2 自动装配核心原理

  1. 核心注解@SpringBootApplication是复合注解，核心包含@EnableAutoConfiguration（开启自动装配）
  2. @EnableAutoConfiguration通过@Import(AutoConfigurationImportSelector.class)加载自动配置类选择器
  3. AutoConfigurationImportSelector读取META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports文件（Spring Boot 2.7+），获取所有预设的自动配置类
  4. 根据条件注解（@ConditionalOnClass、@ConditionalOnMissingBean、@ConditionalOnProperty等）筛选，匹配当前项目环境的自动配置类
  5. 将符合条件的配置类注册到 Spring 容器中，完成自动装配

3.2.3 Starter 起步依赖

Starter 是 Spring Boot 提供的开箱即用的场景化依赖包，封装了对应场景所需的所有依赖与自动配置，无需开发者手动引入多个零散依赖、解决版本冲突问题。

  • 官方 Starter 示例：spring-boot-starter-web（Web 开发）、spring-boot-starter-data-jpa（数据访问）、spring-boot-starter-test（测试）
  • 第三方 Starter 示例：mybatis-spring-boot-starter、spring-boot-starter-data-redis

3.2.4 核心常用注解

@SpringBootApplication：SpringBoot 启动入口核心复合注解，整合配置类、自动配置、组件扫描三大核心能力，一键开启 SpringBoot 基础功能。

@RestController：SpringMVC REST 接口专用注解，整合@Controller+@ResponseBody，标识类为 HTTP 接口控制器，类中方法返回值直接作为 HTTP 响应体（默认序列化为 JSON）。

@RequestMapping：SpringMVC 请求映射基础注解，可作用于类 / 方法，用于绑定 HTTP 请求的 URL 路径、请求方法、请求头等匹配规则，是@GetMapping/@PostMapping等派生注解的父注解。

@Value：Spring 单属性注入注解，标注在类字段上，从配置文件、系统环境变量、SpEL 表达式中读取单个值，注入到 Bean 的对应属性中。

@ConfigurationProperties：Spring 批量配置绑定注解，通过指定前缀匹配配置文件，批量将配置项映射并注入到 Java Bean 中，支持松散绑定、类型转换与配置校验。

@Bean：Spring Bean 注册注解，标注在配置类的方法上，将方法返回的对象注册为 IoC 容器管理的 Bean，用于自定义 Bean 实例化逻辑，或注册第三方组件到容器中。

@Conditional 系列：Spring 条件装配核心注解，标注在配置类或@Bean方法上，仅当满足指定前置条件时，才会注册对应的 Bean；是 SpringBoot 自动配置的底层核心，内置@ConditionalOnClass、@ConditionalOnMissingBean等大量场景化派生注解。

3.3 MyBatis & MyBatis-Plus

3.3.1 MyBatis 相对原生 JDBC 的核心优势

  • SQL 与 Java 代码解耦，SQL 写在 XML 文件中，便于统一管理与维护
  • 自动完成 Java 对象与数据库结果集的 ORM 映射，减少大量冗余的重复代码
  • 内置强大的动态 SQL 功能，可根据条件灵活拼接 SQL 语句
  • 屏蔽 JDBC 底层繁琐的 API 操作，简化数据库开发流程
  • 兼容多种主流数据库，适配性强

JDBC 连接数据库步骤：加载驱动→建立连接→创建 Statement→执行 SQL→处理结果集→关闭资源。

MyBatis查询步骤：

  1. 配置MyBatis： 在项目中配置MyBatis的数据源、SQL映射文件等。
  2. 创建实体类： 创建用于映射数据库表的实体类。
  3. 编写SQL映射文件： 创建XML文件，定义SQL语句和映射关系。
  4. 编写DAO接口： 创建DAO接口，定义数据库操作的方法。
  5. 编写具体的SQL查询语句： 在DAO接口中定义查询方法，并在XML文件中编写对应的SQL语句。
  6. 调用查询方法： 在服务层或控制层调用DAO接口中的方法进行查询。

3.3.2 #{} 与 ${} 核心区别

  • #{} ：预编译处理（效率高），采用占位符赋值方式，底层会将传入的参数当作字符串处理，调用 PreparedStatement 的 set 方法来赋值，有效防止 SQL 注入，生产环境首选
  • ${}：直接字符串拼接，将传入的参数原封不动的拼接到 SQL 语句中，存在 SQL 注入风险，仅适用于表名、排序字段等无法使用预编译的场景

3.3.3 MyBatis-Plus 核心能力

MyBatis-Plus 是 MyBatis 的增强工具，在 MyBatis 的基础上只做增强不做改变，核心能力：

  • 内置 BaseMapper，单表 CRUD 操作无需编写 SQL，零配置即可实现
  • 内置强大的条件构造器，可灵活拼接查询条件
  • 内置分页插件、代码生成器、主键自动生成、逻辑删除、乐观锁等常用功能
  • 完全兼容 MyBatis 原生的所有功能，原有项目可无缝切换

3.3.4 核心设计模式

  • 建造者模式：隔离复杂对象的创建与表示，分步完成实例构建，典型实现：SqlSessionFactoryBuilder、各类 XML 配置 / 映射解析 Builder、CacheBuilder。
  • 工厂模式：解耦对象创建与使用逻辑，提供统一的实例化入口，典型实现：SqlSessionFactory、MapperProxyFactory、ObjectFactory。
  • 单例模式：保证类的全局唯一实例，避免资源重复创建，典型实现：全局上下文 ErrorContext、日志工厂 LogFactory。
  • 代理模式：MyBatis 核心底层模式，通过动态代理实现无侵入的功能增强与访问控制，典型实现：JDK 动态代理的 Mapper 接口代理、连接日志代理，CGLIB/javassist 实现的延迟加载代理。
  • 组合模式：以树形结构统一处理整体与部分对象，支撑动态 SQL 节点的解析与拼接，典型实现：SqlNode 及其各类 SQL 节点子类。
  • 模板方法模式：固定核心流程骨架，由子类自定义实现步骤细节，标准化 SQL 执行与类型处理流程，典型实现：BaseExecutor 与子类执行器、BaseTypeHandler 与各类型处理器子类。
  • 适配器模式：统一接口规范，兼容多框架异构实现，用于适配第三方日志框架，典型实现：Log 接口对接 jdbc、log4j 等日志框架的适配类。
  • 装饰器模式：透明式叠加功能增强，不改动原有对象，用于缓存能力的分层扩展，典型实现：cache.decorators 包下的缓存装饰器实现。
  • 迭代器模式：封装集合遍历逻辑，不暴露底层数据结构，用于配置属性的解析遍历，典型实现：PropertyTokenizer。

3.4 Spring Cloud 微服务体系

3.4.1 与 Spring Boot 的核心区别

  • Spring Boot：专注于快速、便捷地开发单个单体应用，是微服务开发的基础
  • Spring Cloud：基于 Spring Boot 实现，专注于微服务架构的全局治理，协调管理多个微服务实例，提供服务注册发现、配置管理、网关、熔断降级等微服务全场景能力

3.4.2 微服务核心组件

  • 服务注册中心：Nacos、Eureka
  • 配置中心：Nacos、Spring Cloud Config
  • 负载均衡组件：Ribbon、Spring Cloud LoadBalancer
  • 服务网关：Spring Cloud Gateway
  • 熔断降级组件：Sentinel、Hystrix
  • 服务间调用：OpenFeign、Dubbo
  • 链路追踪：Sleuth + Zipkin、SkyWalking

3.4.3 核心负载均衡算法

轮询、加权轮询、随机、加权随机、一致性哈希（哈希后的数值取模映射出同一台服务器）、最小活跃数、最短响应时间

3.4.4 熔断 & 降级核心概念

  • 熔断：当下游服务不可用、响应超时或异常率过高时，上游服务直接切断对该服务的调用，快速返回失败结果，防止请求堆积导致服务雪崩，是应对服务故障的被动防御机制
  • 降级：在系统高峰期，为了保障核心业务的可用性，主动舍弃非核心业务（如关闭推荐、评论等非核心接口），释放系统资源，保障核心业务平稳运行，是应对流量高峰的主动限流策略

四、Spring 高频面试核心考点汇总

4.1 面试必背 5 大核心考点

IoC/AOP 核心原理、循环依赖 & 三级缓存、Spring 事务失效场景、Spring Boot 自动装配原理、Bean 全生命周期

4.2 核心原理高频考点

  1. 依赖倒置、依赖注入、控制反转分别是什么：
    • 控制反转：程序执行流程控制权从开发者转给框架。
    • 依赖注入：外部创建依赖对象，通过构造器 / Setter 注入类中，是 IoC 的实现方式。
    • 依赖倒置：高层不依赖低层，二者共同依赖抽象，细节依赖抽象。
  2. IoC 与 DI 的区别：IoC 是设计思想，核心是控制权的转移；DI 是 IoC 思想的具体实现方式，是实现 IoC 的手段。
  3. BeanFactory 与 ApplicationContext 的区别：BeanFactory 是 Spring 底层容器，提供基础 IoC 能力，懒加载；ApplicationContext 是其子接口，提供更多高级特性，启动时预加载单例 Bean。
  4. AOP 的动态代理两种实现方式的区别：JDK 动态代理基于接口，CGLIB 基于继承；JDK 只能代理实现接口的类，CGLIB 无此限制，但无法代理 final 类和方法。
  5. Spring 循环依赖与三级缓存
    1. 支持场景：仅支持单例 Bean 的 Setter / 字段注入，构造器注入、多例 Bean 的循环依赖会直接报错
    2. 三级缓存结构（DefaultSingletonBeanRegistry中）：
      • 一级缓存singletonObjects：存放完全初始化完成的单例 Bean，可直接使用
      • 二级缓存earlySingletonObjects：存放已实例化但未完成初始化的 Bean 早期引用，用于解决循环依赖
      • 三级缓存singletonFactories：存放 Bean 的ObjectFactory工厂，用于在循环依赖时生成 Bean 的早期引用，核心解决 AOP 代理场景下的循环依赖
    3. 核心解决流程：实例化 A → 放入三级缓存 → 填充 A 的属性时发现依赖 B → 实例化 B → 填充 B 的属性时从三级缓存拿到 A 的工厂，生成 A 的早期引用（AOP 场景下提前生成代理对象）并移至二级缓存 → B 完成初始化并放入一级缓存 → A 完成属性注入与初始化，放入一级缓存
    4. 二级缓存无法单独解决循环依赖的原因：无法处理 AOP 代理场景，会导致同一个 Bean 生成多个代理实例，破坏单例原则。
  6. Spring 框架的核心扩展点
    1. BeanFactoryPostProcessor：Bean 实例化之前，修改 Bean 的定义信息
    2. BeanPostProcessor：Bean 初始化前后，对 Bean 实例进行定制修改
    3. PropertySource：用于定义不同的属性源，如文件、数据库等，以便在Spring应用中使用
    4. ApplicationListener：监听 Spring 应用事件，实现事件驱动开发
    5. ImportSelector / ImportBeanDefinitionRegistrar：根据条件，动态注册 Bean 定义到 Spring 容器，实现配置类模板化。
    6. Spring MVC中@HandlerInterceptor/@ControllerAdvice：用于拦截处理请求并在前中后执行特点代码 / 用于全局处理控制器的异常、数据绑定和校验。
    7. 自定义自动配置类、自定义注解

4.3 Bean 相关高频考点

  1. Bean 的生命周期完整流程：实例化 → 属性赋值 → BeanPostProcessor 前置处理 → 初始化 → BeanPostProcessor 后置处理 → 就绪可用 → 销毁 → 容器关闭
  2. Bean 的生命周期扩展方式：@PostConstruct/@PreDestroy注解、InitializingBean/DisposableBean接口、自定义init-method/destroy-method方法
  3. @Autowired 与 @Resource 的区别：@Autowired是 Spring 原生注解，默认按类型注入；@Resource是 JSR-250 标准注解，默认按名称注入，跨框架通用性更强。
  4. Spring 单例 Bean 是否线程安全：Spring 容器不提供单例 Bean 的线程安全保障，线程安全与否取决于 Bean 本身是否是有状态的；无状态 Bean（仅方法调用，无成员变量修改）线程安全，有状态 Bean（存在可修改的成员变量）非线程安全，需自行处理线程安全问题。

4.4 事务相关高频考点

  1. Spring 事务失效的 8 大核心场景
    1. 事务方法非public修饰
    2. 异常被try-catch捕获吞没，未向外抛出
    3. 抛出非运行时异常（受检异常），且未配置rollbackFor属性（Spring 默认只回滚RuntimeException和Error）
    4. 同类内部方法调用（this.method()），绕过了 Spring 代理对象，无法触发事务切面
    5. 事务传播属性配置错误，导致事务不生效
    6. 多数据源场景下，未正确配置事务管理器
    7. 方法被final/static修饰，无法被 CGLIB 代理，事务切面失效
    8. 未被 Spring 容器管理，类未添加@Component等注解，Spring 无法生成代理对象
  2. Spring 事务的传播行为：7 种传播行为，核心为REQUIRED（默认，当前有事务则加入，无则新建）、REQUIRES_NEW（无论是否有事务，都新建独立事务）、NESTED（嵌套事务）

4.5 设计模式高频考点

Spring 框架中大量使用设计模式，核心高频考点：

  • 工厂模式：BeanFactory、ApplicationContext，负责 Bean 的创建与管理
  • 代理模式：AOP 的 JDK 动态代理、CGLIB 动态代理
  • 单例模式：Spring Bean 默认单例作用域，底层通过单例注册表实现
  • 模板方法模式：JdbcTemplate、HibernateTemplate、RestTemplate，封装固定流程，开放自定义扩展点
  • 观察者模式：Spring 事件驱动模型，ApplicationEvent事件、ApplicationListener监听器、ApplicationEventPublisher发送者。通过定义事件、监听器和发送者实现，观察者注册在 ApplicationContext 中，消息发送由 ApplicationEventMulticaster 完成。
  • 适配器模式：Spring MVC 的HandlerAdapter、AOP 的通知适配器
  • 装饰器模式：DataSource包装、HttpRequest包装

4.6 全家桶高频考点

  1. Spring Boot 自动装配原理：详见本文 3.2.2 章节，核心是@EnableAutoConfiguration注解、AutoConfigurationImportSelector、SPI 机制加载自动配置类、条件注解筛选
  2. Spring MVC 的完整执行流程：详见本文 3.1.2 章节
  3. 过滤器与拦截器的区别：详见本文 3.1.4 章节
  4. Spring Cloud 与 Spring Boot 的版本对应关系：Spring Cloud 基于 Spring Boot 开发，每个 Spring Cloud 版本对应特定的 Spring Boot 版本区间，版本不匹配会导致大量兼容性问题
  5. 微服务中熔断与降级的区别：熔断是应对下游服务故障的被动防御，防止雪崩；降级是应对流量高峰的主动策略，保障核心业务可用
