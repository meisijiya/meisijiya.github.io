第6章 Spring AI Alibaba 全栈开发实战

本章基于课程实操经验，覆盖从Spring AI Alibaba基础整合、核心交互对象、Agent开发、MCP协议、RAG认知到AgentScope框架的全流程

6-1 Spring AI 1.0、1.1、2.0 版本对比

核心概要

课程底层采用与Spring AI版本同步的国内版Spring AI Alibaba，对比三个版本的核心组件差异：1.0提供MCP支持、ChatClient跨模型交互、对话记忆和争议较大的RAG；1.1将MCP内嵌为注解、增强安全性、扩展记忆存储、原生支持Gemini 2.0 Flash；2.0 M1固化MCP、优化存储与向量数据库性能、深化OpenAI和Gemini支持。学习重点为MCP、对话上下文记忆、ChatClient对象，RAG因争议大不作为课程核心。

详细内容

版本核心差异

版本 核心特性 课程定位
1.0 正式版（主力版本） 1. 支持MCP协议 2. 推出ChatClient跨模型交互对象，基础操作固化，后续版本兼容 3. 支持对话上下文记忆 4. 包含RAG（检索增强生成，争议大） 课程基础版本
1.1 正式版 1. MCP内嵌为Spring Boot注解，增强OAuth2安全性，跟随MCP版本迭代 2. 扩展对话记忆存储方案（更多存储类型） 3. 原生支持谷歌Gemini 2.0 Flash 4. 核心重点仍为MCP、ChatClient、对话记忆 进入Agent自主决策时代
2.0 M1（2025年底发布，2026年预计出正式版） 1. 固化MCP组件，要求Java 21+、Spring Boot 4.0+ 2. 优化对话记忆存储（重点支持Redis）、向量数据库（Redis）性能 3. 深化OpenAI SDK原生接入、Gemini支持 4. 继续迭代RAG 未来重点方向

学习建议💡

优先掌握跨版本持续迭代的核心组件：MCP、ChatClient、对话上下文记忆；RAG争议较大，不作为课程重点。

6-2 Spring Boot 整合 Spring AI Alibaba

核心概要

Spring AI Alibaba是面向大模型交互的开发框架，整合必须严格匹配版本，旧版spring-ai-alibaba-starter已停更不适用。整合存在两个关键坑：需引入与内置Spring Boot 3.4.5匹配的Web依赖，且依赖顺序必须靠前，否则触发日志错误导致启动失败。

详细内容

整合核心原则⚠️

严格依据所用版本的官方文档选择依赖，禁止套用旧版教程依赖

依赖说明

  1. 旧版spring-ai-alibaba-starter仅更新到1.0.0-M6.1（2025年3月后停更），不适用于当前1.0.0+版本
  2. 正确整合依赖：对应版本的Spring AI Alibaba整合包已内置spring-boot-starter，内置Spring Boot版本为 3.4.5，无需重复引入

启动两大必踩坑⚠️

  1. 必须引入 Spring Boot Web 3.4.5 依赖（版本与整合包内置Boot版本严格一致）
  2. Spring Boot Web依赖必须放在Spring AI Alibaba整合依赖之前，否则触发日志冲突，启动失败

验证标准

正确配置后Spring Boot启动成功，监听8080端口；仅缺API Key会报大模型相关错误，不影响框架启动。

6-3 和大模型互动的中枢：ChatModel对象

核心概要

ChatModel是与大模型交互的中枢对象，由Spring Boot自动装配初始化，无需手动new。通过构建Controller、组合UserMessage/Prompt/ChatModel可实现完整的大模型调用流程，需对返回结果做非空校验保证稳健性。

详细内容

调用完整流程（Simple Chat接口示例）

  1. 创建RestController，定义GET接口/simpleChat
  2. 对象构建：
    • 用户输入 → UserMessage对象
    • 提示词 → Prompt对象（入参为UserMessage）
    • 注入ChatModel：通过构造方法注入，设为final（自动装配初始化，无需手动创建）
  3. 调用大模型：ChatModel.call(Prompt) → 返回ChatResponse对象
  4. 结果提取：ChatResponse.getResult().getOutput().getText()

稳健性校验⚠️

必须校验ChatResponse、getResult()、getOutput()非空，避免空指针、超时等异常：

配置要求

  • 需配置阿里百炼平台API Key（免费申请）
  • 指定调用模型：仅支持阿里百炼平台模型（如DeepSeek-R1、通义千问等），不支持文心一言等外部模型
  • 配置文件：application.yml中添加spring.ai.alibaba.bailian相关配置

6-4 和大模型互动的窗口：ChatClient对象

核心概要

ChatClient是对ChatModel的封装，支持链式调用，简化交互流程。推荐通过Builder模式创建，避免传统new方式的多属性繁琐、高并发状态不一致、默认值遗漏问题，保证对象不可变。

详细内容

ChatClient与ChatModel的关系

ChatClient底层封装ChatModel，无需手动创建，同样由Spring Boot自动装配初始化，可通过构造方法注入。

Builder模式优势

  1. 避免多属性逐个set的繁琐代码
  2. 创建不可变对象（属性为final），高并发下保证状态一致
  3. 避免默认值遗漏导致的报错
  4. 支持链式调用配置属性

调用示例

实践验证

接口调用成功返回大模型响应，比原生ChatModel流程更简洁。

6-5 ChatClient流式响应获取数据（SSE）

核心概要

传统一次性返回大模型结果的方式在内容较多时等待时间长，流式响应可类打字机效果逐字返回。需基于SSE（Server-Sent Events）协议实现长连接、服务端单向推送，改用ChatClient.stream()方法即可实现。

详细内容

SSE协议基础

  • 基于HTTP长连接，服务端向客户端单向推送流式数据
  • 客户端需在请求头设置Content-Type: text/event-stream声明建立SSE通道

流式调用改造

  1. 替换call()为stream()方法
  2. 返回结果从String变为Flux<String>（响应式流类型）
  3. 接口返回类型改为Flux<String>，配置SSE请求头

实践效果

API调用后返回流式数据，前端可逐字展示结果，降低等待感知。

6-6 Spring AI 1.1 正式进入Agent自主决策时代

核心概要

Spring AI Alibaba 1.1版本将智能体（Agent）提升为核心，替代1.0版本直接通过ChatClient交互的模式。ReAct Agent具备大脑（大模型）、工具调用、记忆、环境感知、规划及自主决策能力，不再是单纯执行者，支持单/多智能体协同完成复杂任务。

详细内容

ReAct Agent核心能力

能力类型 说明
基础能力 大脑（语言大模型，逻辑推理）、工具调用（手/脚，解决实际问题）、记忆（多轮对话一致性）、环境感知（RAG）
1.1新增能力 规划能力（复杂任务自动分解）、自主决策能力（自主选择工具、执行步骤）

6-7 Spring AI 1.1 组建旅游规划的Agent团队

核心概要

1.1版本可快速构建单ReAct Agent，并组合为多Agent团队。以旅游规划场景为例，可构建路线制定、行程规划、费用统筹、主管4类Agent，通过将费用统筹Agent嵌入其他Agent的方式实现协作。

详细内容

旅游规划Agent团队组成

  1. 路线制定Agent：负责城市间交通工具选择（自驾、高铁等）
  2. 行程规划Agent：负责目的地景点、住宿、活动安排
  3. 费用统筹Agent：负责各类费用计算
  4. 主管Agent：负责任务分解、分发，统筹全局

协作逻辑1：工具嵌入

路线制定、行程规划Agent的业务均涉及费用计算，将费用统筹Agent封装为ToolCallback，通过tools()方法嵌入前两个Agent，执行中涉及费用时自动调用。

注意事项⚠️

description属性是Agent的核心标识，大模型通过描述识别Agent的能力边界、分配任务，必须准确填写。

6-8 Spring AI 1.1 Agent团队协同合作

核心概要

通过工作流Agent（Flow Agent）可将多ReAct Agent按嵌入调用、并行执行、路由分发三种逻辑协同，最终实现旅游规划任务。工作流Agent不执行具体业务，仅负责流程编排，通过invoke()方法启动工作流。

详细内容

多Agent协同三大逻辑

  1. 嵌入调用：费用统筹Agent嵌入路线、行程规划Agent（6-7已实现）
  2. 并行执行：路线制定、行程规划Agent业务无交集、无依赖，可并行执行
  3. 路由分发：主管Agent作为LLM路由，将用户需求拆分为交通、行程类任务，分发给对应Agent

工作流Agent类型

工作流Agent 作用 核心配置
ParallelAgent（并行执行） 多个Agent无依赖并行执行 subAgents()传入并行Agent列表
LLMRoutingAgent（LLM路由） 大模型自动拆分任务、分发给候选Agent subAgents()传入候选Agent列表
SequenceAgent（顺序执行） 按指定顺序执行多个工作流Agent subAgents()传入顺序执行的Agent列表

6-12 大模型调用本地自定义的MCP服务

核心概要

MCP客户端连接服务端后，若未将工具清单传递给大模型，会导致大模型无法识别工具、答非所问。需在构建ChatClient时通过defaultToolCallbacks()传入ToolCallbackProvider对象（MCP工具清单），大模型即可自动识别并调用工具。

6-13 RAG只是一个过渡阶段吗？

核心概要

RAG（检索增强生成，又称私人知识库）由检索、增强、生成三大模块构成，最初用于弥补早期大模型数据非实时、意图理解差、幻觉、无法访问私人数据、上下文窗口有限的缺陷。2026年大模型已解决数据实时性、意图理解、上下文窗口问题，但幻觉、私人数据访问仍未突破，RAG地位有所下降，但不会被完全取代。

详细内容

RAG解决的早期大模型缺陷（2022-2024年）

  1. 数据非实时：无法获取最新数据
  2. 用户意图理解差：无法挖掘深层需求
  3. 幻觉：一本正经胡说八道
  4. 无法访问私人数据：企业商业数据等无法上传公共大模型
  5. 上下文窗口有限：无法处理长文本

RAG实现逻辑

  1. 私人数据分块→向量化→存入向量数据库
  2. 检索向量数据库的相关知识，与用户提问组装为上下文，发给大模型
  3. 大模型仅基于提供的数据分析，无需外部检索

2026年现状

  • 大模型已解决：数据实时性、用户意图理解、上下文窗口限制（长上下文支持）
  • 仍未解决：幻觉、私人数据访问
  • 课程定位：RAG争议较大，不作为核心学习内容

6-14 长上下文能取代RAG吗？

核心概要

RAG存在链路长易放大错误、分块机械化导致知识碎片化的缺陷，长上下文解决了部分大模型原生问题，但无法覆盖RAG的核心优势场景：私人数据访问、多元异构数据处理。RAG不会被长上下文取代，Spring AI 2.0仍将其作为核心组件，需持续进化。

详细内容

RAG的固有缺陷

  1. 链路长：数据清洗→分块→向量化→存储→查询→召回→重排，任一节点错误会放大传播
  2. 知识碎片化：分块机械化，易割裂上下文关联，导致检索精度不足

长上下文的局限

  • 已解决：数据实时性、用户意图理解、上下文窗口限制（可处理整本《红楼梦》级别的文本）
  • 未解决：幻觉、私人数据访问（企业数据无法上传公共大模型）

RAG不可替代场景

  1. 私人/企业数据访问：核心优势，公共大模型无法获取私有数据
  2. 多元异构数据处理：音频、结构化/非结构化数据可统一向量化存储查询

结论与建议💡

  • RAG不会被长上下文取代，需持续进化
  • 优先学习有持久性、趋势性的知识（MCP、Agent、ChatClient），避免迭代快的短期内容

第六章核心总结

  1. 版本选择：优先Spring AI Alibaba 1.1+，核心关注MCP、ChatClient、对话记忆、Agent
  2. 基础整合：严格匹配版本，注意Spring Boot Web依赖的顺序和版本（3.4.5）
  3. 交互对象：ChatModel是基础，ChatClient是封装优化，优先用ChatClient+流式响应
  4. Agent开发：1.1版本核心，ReAct Agent具备自主决策规划能力，多Agent通过工作流Agent编排协同
  5. MCP协议：实现大模型调用本地工具，必须将工具清单通过defaultToolCallbacks()传递给大模型
  6. RAG认知：不会被长上下文取代，核心优势是私人数据和异构数据处理
  7. AgentScope：更优的Java多智能体框架，替代原生Spring AI Alibaba 1.1，配置更简便
