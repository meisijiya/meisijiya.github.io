6.1 Spring AI 版本特性对比（1.0/1.1/2.0）

对应课程文稿：6-1 SpringAI 1.0，1.1 和 2.0_文稿

6.1.1 版本关系说明

  • Spring AI Alibaba是Spring AI的国内同步版本，版本号与Spring AI完全对齐，课程多智能体项目基于该框架实现。
  • 三个版本为承接关系，核心持续发展组件为：MCP协议、对话上下文记忆、ChatClient对象，RAG因争议较大不作为课程重点。

6.1.2 各版本核心特性对比

版本 核心特性 课程定位
Spring AI 1.0 1. 首次支持MCP协议 2. 提供跨模型交互对象ChatClient 3. 支持对话上下文记忆 4. 支持RAG（向量检索生成，争议大） 课程主力版本，基础组件接口已固定
Spring AI 1.1 1. MCP内嵌为注解，增强调用安全性 2. 扩展对话记忆存储方案（内存/文本/数据库等） 3. 原生支持谷歌Gemini 2.0 Flash 4. 优化RAG执行效率 引入Agent自主决策核心
Spring AI 2.0 M1 1. MCP固化为固有组件 2. 仅支持Java 21+、Spring Boot 4.0+ 3. 优化存储与向量数据库（Redis）性能 4. 深化OpenAI、Gemini原生支持 明确框架长期发展方向

6.1.3 课程重点说明

✅ 优先学习：MCP、对话上下文记忆、ChatClient对象
❌ 暂不重点：RAG（争议大，定位为过渡技术）

6.4 ChatClient流式响应（SSE协议）

对应课程文稿：6-5 ChatClient流式响应获取数据 (SSE)_文稿

6.4.1 为什么需要流式响应？

传统一次性返回方式在内容较多时等待时间长，流式响应（打字机效果）可提升用户体验，基于SSE（Server-Sent Events）协议实现。

6.4.2 SSE协议核心要点

  1. 基于HTTP长连接，服务端单向向客户端推送数据
  2. 客户端需在请求头中设置Accept: text/event-stream声明建立SSE通道
  3. 连接建立后，服务端持续推送流式数据，直到响应结束

6.5 MCP协议：模型上下文交互

对应课程文稿：6-11 MCP客户端连接MCP服务端_文稿、6-12 大模型调用本地自定义的MCP服务_文稿

6.5.1 MCP核心概念

MCP（Model Context Protocol）是连接大模型与外部数据/工具的协议，三个核心节点：

  • MCP Host：大模型端
  • MCP Client：客户端（基于Spring AI Alibaba实现）
  • MCP Server：服务端（提供工具/数据）

6.5.2 MCP服务端搭建（回顾）

  • 基于Netty启动，默认端口8081
  • 通过@Tool注解定义工具，自动注册到工具清单
  • 课程示例：天气MCP服务端，提供「获取城市温度」「获取城市紫外线值」两个工具

6.6 Agent自主决策开发（Spring AI Alibaba 1.1+）

对应课程文稿：6-6 SpringAi 1.1 正式进入Agent自主决策时代_文稿、6-7 组建旅游规划的Agent团队_文稿、6-8 Agent团队协同合作_文稿

6.6.1 1.1版本核心变化：Agent成为核心

Spring AI Alibaba 1.1将ReAct Agent（具备规划+自主决策能力）作为核心，不再仅通过ChatClient直接交互，而是基于Agent完成复杂任务。

  • 1.0版本的Agent：仅作为执行者，具备基础能力（大模型、工具、记忆、感知）
  • 1.1版本的Agent（ReAct Agent）：新增规划能力、自主决策能力，可自主分解任务、选择工具，支持单/多Agent协同

团队角色定义

Agent名称 职责 能力
费用统筹Agent 计算旅游相关费用（住宿、门票、交通等） 工具：费用计算

协同逻辑实现

  • 并行执行Agent：路线规划与行程规划无业务依赖，可并行执行
  • 路由分发Agent：主管Agent基于LLM动态分发任务
  • 顺序串联工作流：将路由→并行执行串联为完整流程

6.7 AgentScope框架（官方最新资料整合）

补充：AgentScope Java官方文档（2026-04更新，当前最新版本1.0.11，课程笔记中为1.0.7，以最新版为准）

6.7.1 AgentScope概述

  • 是面向智能体的编程框架，Java版基于Spring AI Alibaba 1.1开发，比原生Spring AI Alibaba封装更优，课程后续替代原生Agent开发。
  • 核心特性：
    ✅ ReAct推理-行动范式，支持自主规划+动态工具调用
    ✅ 全程可控：安全中断、优雅取消、人机协作（Hook系统）
    ✅ 生产级能力：长期记忆、RAG、结构化输出、MCP/A2A协议支持
    ✅ 响应式架构：基于Project Reactor，支持非阻塞执行

2. 核心组件

组件 作用 说明

Agent 有状态对象 持有自己的Memory、Toolkit、配置，不可并发调用同一实例

6.8 RAG技术讨论：过渡还是长期方案？

对应课程文稿：6-13 RAG只是一个过渡阶段吗 ？_文稿、6-14 长下文能取代RAG吗 ？_文稿

6.8.1 RAG是什么？

RAG（检索增强生成）是检索+增强+生成三个模块构成的工具，最初用于弥补大模型的5大缺陷：

  1. 数据时效性差（无法获取最新数据）
  2. 用户意图理解不足
  3. 幻觉（一本正经胡说八道）
  4. 无法访问私人数据（企业机密、个人隐私）
  5. 上下文窗口有限

6.8.2 RAG的局限

  1. 链路长易放大错误：数据清洗→分块→向量化→存入向量库→查询→召回→重排，任一节点错误会沿链路放大
  2. 分块机械化导致知识碎片化：固定分块方式可能割裂上下文关联，导致检索精度下降
  3. 部分价值被大模型发展削弱：
    • 数据时效性：当前大模型（豆包、DeepSeek、千问）已支持实时数据查询
    • 意图理解：大模型参数提升后，已能精准挖掘用户深层需求
    • 上下文窗口：长上下文功能已支持百万级Token，可直接传入大量文本分析

6.8.3 长上下文能否取代RAG？

❌ 不能取代，RAG仍有不可替代的场景：

  1. 私人数据场景：企业机密、个人隐私数据无法上传公网大模型，RAG可本地部署向量库实现私域数据访问
  2. 多元异构数据场景：支持音频、视频、结构化/非结构化数据统一向量化存储，大模型无法直接处理多模态异构数据
  3. 官方持续支持：Spring AI 2.0仍将其作为核心组件，优化Redis向量库性能，深化RAG支持

6.8.4 课程定位

RAG当前仍处于快速迭代阶段，课程优先学习有持久化价值的趋势性知识（MCP、Agent、ChatClient等），RAG待技术成熟后补充。

本章学习建议

  1. 版本优先：Spring AI Alibaba 1.0为基础，1.1为核心，2.0看方向
  2. 交互优先：使用ChatClient+Builder模式，优先流式响应
  3. Agent优先：基于AgentScope开发，替代原生Spring AI Alibaba Agent
  4. 避坑优先：依赖版本、顺序、MCP配置、Agent并发约束等实测问题需重点关注
